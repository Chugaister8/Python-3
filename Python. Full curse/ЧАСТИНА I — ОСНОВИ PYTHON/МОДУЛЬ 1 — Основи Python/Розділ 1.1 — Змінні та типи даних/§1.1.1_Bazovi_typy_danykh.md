# § 1.1.1 Базові типи даних

---

## Модуль 0 позаду. Модуль 1 починається тут

Якщо ти дійшов до цього місця і виконав завдання нульового модуля — у тебе вже є те, чого немає у більшості людей, які «починають вчити Python»: налаштоване середовище, встановлені інструменти, перший репозиторій і розуміння навіщо все це потрібно. pyenv слідкує за версіями Python. Віртуальне середовище ізолює залежності. VS Code з розширеннями підказує і форматує. Git зберігає кожен крок. pre-commit не пропускає неохайний код. Це не дрібниці — це фундамент, на якому будуватиметься всі наступні місяці роботи. І він вже стоїть.

Тепер — інший тип роботи. Модуль 0 був про інструменти і середовище, про те як влаштована майстерня. Модуль 1 — про саму мову. Починаємо писати Python не заради `hello.py` і не заради перевірки що все запустилось, а заради реального розуміння: як Python думає про дані, як він зберігає значення, що з ними можна робити і де чатують типові пастки. Це найважливіший модуль у всьому курсі — не тому що найцікавіший, а тому що все що прийде після будується на тому, що тут.

Варто чесно попередити: деякі речі в цьому модулі здадуться банальними. «Ціле число — це ціле число, чого тут пояснювати?» Але Python — мова з власним характером, і цей характер проявляється у деталях, які на перший погляд непомітні. Чому `0.1 + 0.2` не дорівнює `0.3`. Чому функція повернула `None` там де ти очікував рядок. Чому `"42"` і `42` — принципово різні речі хоч і виглядають схоже. Чому `bool("False")` дає `True`. Кожна з цих деталей колись зупинить програму — і людина яка розуміє чому це відбувається, виправить за хвилину. Людина яка не розуміє — витратить годину на пошук «якогось дивного бага».

Є одна порада перед початком, і вона стосується всього модуля: **не просто читай — пиши код**. Кожен приклад — введи в REPL або напиши у файлі і запусти. Поміняй значення. Зламай. Подивись що станеться. Програмування — це не знання яке засвоюється через читання, це навичка яка розвивається через практику. Різниця між тим хто читав і тим хто робив руками — через місяць стає очевидною.

## Перший живий діалог з Python

Відкрий термінал, переконайся що віртуальне середовище активоване, і запусти Python в інтерактивному режимі:

```bash
python
```

Ти потрапиш у REPL — Read-Eval-Print Loop. Назва розшифровується просто: Read — Python читає те що ти ввів, Eval — виконує, Print — виводить результат, Loop — і чекає наступного рядка. Це середовище прямої розмови з інтерпретатором, без файлів і без запуску скриптів. Виглядає ось так:

```
Python 3.12.3 (main, Apr 15 2024, 17:43:11)
Type "help", "copyright", "credits" or "license" for more information.
>>>
```

Символ `>>>` — запрошення до введення. Набираєш будь-який вираз, натискаєш Enter — і Python одразу показує результат. Спробуй:

```python
>>> 2 + 2
4
>>> "Привіт"
'Привіт'
>>> type(42)
<class 'int'>
```

Зверни увагу: ти написав `"Привіт"` у подвійних лапках, а Python відповів `'Привіт'` в одинарних. Це не помилка і не примха. У REPL Python показує так зване repr-представлення — технічне відображення об'єкта, де рядки завжди беруться в одинарні лапки. Коли ж ти використовуєш `print()` — відображення вже людське, без зайвих лапок:

```python
>>> print("Привіт")
Привіт
>>> "Привіт"
'Привіт'
```

Ця дрібна різниця буде ще зустрічатися, і добре що ти її помітив одразу.

REPL — ідеальне місце для експериментів протягом усього курсу. Не впевнений що дасть певний вираз? Введи і побач. Хочеш перевірити як поводиться якийсь тип? Введи і спостерігай. Це в рази швидше ніж писати файл, запускати і дивитись результат. Залиш REPL відкритим поруч з редактором — він стане твоїм постійним супутником.

Вийти з REPL — команда `exit()` або `Ctrl+D` на macOS/Linux, `Ctrl+Z` і Enter на Windows.

## Що таке тип — і навіщо Python так суворий

Уяви що ти наймаєш нового співробітника і кажеш йому: «Візьми ось цей предмет і додай до нього ще один». Якщо предмет — яблуко, він покладе поруч ще одне яблуко. Якщо предмет — число п'ять, він порахує і скаже «шість». Але якщо предмет — яблуко, а «ще один» — це число? Людина здогадається. Запитає. Або зробить щось розумне з контексту.

Python — не людина. Він не здогадується. І якщо ти попросиш його «додати» рядок до числа — він зупиниться і чесно скаже: «Не розумію що ти маєш на увазі». Перевір у REPL:

```python
>>> "5" + 3
TypeError: can only concatenate str (not "int") to str
```

Це не Python примхливий — це він чесний. Рядок `"5"` і число `3` можна об'єднати по-різному: як `"53"` (якщо склеїти тексти) або як `8` (якщо порахувати). Python не вгадує. Він вимагає щоб ти сказав явно що саме хочеш. Це правило пронизує всю мову — і воно рятує від цілого класу помилок, де програма тихо робить щось не те.

Тип даних — це якраз спосіб сказати Python «що це таке». Число `42` — одне. Рядок `"42"` — зовсім інше, хоч і виглядають схоже. Python ставиться до них принципово по-різному, і ця різниця — не деталь, а фундамент.

У Python є п'ять базових типів з якими починається будь-яка програма: рядок `str`, ціле число `int`, число з плаваючою крапкою `float`, логічне значення `bool` і відсутність значення `None`. Це не весь список — є ще списки, словники, кортежі, множини і багато іншого. Але ці п'ять — будівельні цеглини, з яких складається все решта. Зустрічатимуться в кожній програмі без винятку.

## Змінна: ярлик, а не ящик

Перш ніж говорити про типи — потрібно поговорити про змінні. Бо тип без змінної — як слово без речення: існує, але нікуди не прикладеш.

Уяви що ти пишеш програму яка вітає користувача. Ім'я знадобиться кілька разів: у привітанні, у заголовку, в листі підтвердження. Можна писати `"Оксана"` в кожному місці окремо. Але якщо завтра Оксана стане Олексієм — доведеться знайти і поміняти кожне місце. Пропустиш одне — програма виведе неправильно, і ти витратиш час на пошук.

Змінна вирішує цю задачу.

```python
name = "Оксана"
print(name)                     # Оксана
print(f"Привіт, {name}!")       # Привіт, Оксана!
print(f"До побачення, {name}!") # До побачення, Оксана!
```

Тепер ім'я живе в одному місці. Зміниш `"Оксана"` на `"Олексій"` — і всі три рядки оновляться самі.

Знак `=` тут не означає «рівно» у математичному сенсі. Це команда: «прив'яжи ось це значення до ось цього імені». В математиці `x = 5` — твердження. У Python — інструкція.

А тепер дрібна але важлива деталь про те як це влаштовано всередині. Більшість підручників кажуть: «змінна — це ящик куди кладуть значення». Це зрозумілий образ, але неточний. Насправді змінна в Python — це **ярлик**. Коли ти пишеш `name = "Оксана"`, Python створює об'єкт-рядок десь у пам'яті і прикріплює до нього ярлик з іменем `name`. Якщо потім написати `name = "Марія"` — ярлик перекріплюється на новий об'єкт. Старий рядок `"Оксана"` нікуди не зник одразу — він ще якийсь час існує в пам'яті, поки Python не прибере його за непотрібністю.

Навіщо це знати? Тому що ця модель пояснює кілька речей, які інакше здаються магією. Наприклад, чому можна написати `x = x + 1` — це не математична безглуздя, це команда: «обчисли `x + 1`, отримай новий об'єкт, перевісь ярлик `x` на нього». Або чому дві змінні можуть вказувати на один і той самий об'єкт. Ми ще повернемось до цього коли дійдемо до списків — там це стає дуже практичним питанням.

### Як називати змінні

Ім'я змінної може містити латинські літери, цифри і підкреслення, але не може починатись з цифри і не може збігатись з ключовими словами Python:

```python
# Правильно
user_name   = "Іван"
age2        = 30
_private    = "внутрішнє"
total_price = 499.99

# Неправильно
2name = "Іван"    # SyntaxError: починається з цифри
if = True         # SyntaxError: if — ключове слово
class = "клас"    # SyntaxError: class — теж ключове слово
```

Ключових слів небагато — `if`, `else`, `elif`, `for`, `while`, `def`, `class`, `return`, `import`, `from`, `True`, `False`, `None`, `and`, `or`, `not`, `in`, `is` та ще кілька — і всі вони зарезервовані для синтаксису мови.

За угодою у Python змінні пишуть у стилі `snake_case`: всі слова маленькими літерами, розділені підкресленням. `first_name`, `order_total`, `is_logged_in`. Не `firstName` — це JavaScript-стиль, який у Python виглядає чужорідно. Не `FirstName` — це для класів. Ця угода не є технічною вимогою — Python не відмовить якщо напишеш `firstName`. Але вся Python-екосистема її дотримується, і відхилення від неї — як вийти в піджаку на пляж: формально можна, але всі дивляться дивно.

І ще одне — мабуть найважливіше з усього про іменування: ім'я має **пояснювати що всередині**.

```python
# Погано — що це таке взагалі?
d = 7
u = "test@example.com"
f = True

# Добре — все зрозуміло без коментарів
days_until_deadline = 7
user_email          = "test@example.com"
is_admin            = True
```

Хороше ім'я — це безкоштовна документація. Поганий код з хорошими іменами читається краще ніж хороший код з поганими.

## str — рядок

Рядок — це послідовність символів. Будь-який текст: ім'я, адреса, повідомлення, HTML-код, відповідь від API, назва файлу — все це рядки.

Рядок створюється в лапках. Одинарних або подвійних — Python ставиться до них однаково:

```python
name       = "Оксана"
city       = 'Київ'
empty      = ""      # порожній рядок — теж повноцінний рядок
digit_text = "42"    # РЯДОК, не число — тому що в лапках
```

Ось тут варто зупинитись. `"42"` і `42` виглядають схоже, але це принципово різні об'єкти:

```python
>>> type("42")
<class 'str'>
>>> type(42)
<class 'int'>

>>> "42" + "8"
'428'    # конкатенація: просто склеюємо тексти

>>> 42 + 8
50       # арифметика: математичне додавання

>>> "42" + 8
TypeError    # Python не знає що з цим робити
```

Для Python рядок `"42"` — це послідовність двох символів: спочатку `"4"`, потім `"2"`. Він нічого не знає про те що разом вони утворюють число. Тому коли введення приходить від користувача через клавіатуру — а воно завжди приходить як рядок — його спочатку потрібно перетворити. Про перетворення типів — трохи нижче.

Потрійні лапки — `"""` або `'''` — дозволяють рядку займати кілька рядків коду, зберігаючи переноси і відступи всередині:

```python
poem = """Рядок перший,
рядок другий,
рядок третій."""

sql = """
    SELECT name, email
    FROM users
    WHERE is_active = TRUE
"""
```

Потрійні лапки ще використовуються для документаційних рядків — docstrings — спеціальних коментарів, які пояснюють функції і класи. Але про це — коли дійдемо до функцій.

Детально методи рядків, конкатенацію, f-рядки і escape-послідовності розглянемо в § 1.1.2 — це велика і важлива тема, яка заслуговує на власний параграф.

## int — ціле число

`int` — це цілі числа. Будь-яке ціле: додатне, від'ємне або нуль.

```python
age:   int = 25
year:  int = 2025
score: int = -15
big:   int = 10_000_000
```

Підкреслення в числах — лише для читабельності, Python їх ігнорує при обчисленнях. Але для людини що читає код різниця між `10000000` і `10_000_000` очевидна. Ніхто не забороняє писати `10_00_00` — але перший варіант логічніший.

Є одна особливість Python яку варто знати одразу і яка приємно відрізняє його від більшості мов: ціле число може бути **будь-якого розміру**. У Java чи C++ `int` має максимальне значення, і якщо перевищити — число «переповнюється» і стає від'ємним. Це джерело дуже неприємних і важко відловлюваних помилок. Python виділяє під ціле число стільки пам'яті скільки потрібно:

```python
>>> 2 ** 100
1267650600228229401496703205376   # 31 цифра — і жодної проблеми
```

Тепер про ділення — і тут є нюанс, який регулярно дивує тих хто приходить з інших мов. Оператор `/` в Python **завжди** повертає `float`, навіть якщо обидва числа цілі і результат теж ціле:

```python
>>> 10 / 2
5.0        # float! не int
>>> 7 / 7
1.0        # теж float
```

Якщо потрібне ціле — є окремий оператор `//`:

```python
>>> 10 // 2
5          # int
>>> 17 // 5
3          # відкидає дробову частину
```

Саме `//`, а не `/`. Різниця між ними — джерело тихих помилок в обчисленнях, де ти очікуєш ціле і отримуєш `5.0` замість `5`, а потім дивуєшся чому `type()` каже `float`.

Детально арифметику, оператор `%`, пріоритет і модуль `math` розглянемо в § 1.1.3.

## float — число з плаваючою крапкою

`float` — числа з дробовою частиною. Ціна, температура, координата, відсоток, будь-яке вимірювання де важлива точність після коми.

```python
price:       float = 19.99
pi:          float = 3.14159265358979
temperature: float = -7.5
```

Є ще наукова нотація для дуже великих або дуже маленьких чисел — буква `e` означає «помножити на десять у степені»:

```python
avogadro    = 6.022e23    # 6.022 × 10²³
light_speed = 3e8         # 300 000 000.0
tiny        = 9.11e-31    # 9.11 × 10⁻³¹
```

Коли в одному виразі стоять `int` і `float` разом — Python автоматично перетворює `int` на `float`. Логіка проста: `float` «ширший», він може представити будь-яке ціле число, тому перетворення безпечне і відбувається автоматично:

```python
>>> 5 + 2.0
7.0
>>> type(5 + 2.0)
<class 'float'>
```

### Пастка, яка чекає на кожного

Є одна особливість `float`, яка рано чи пізно дивує кожного хто з ним працює. Відкрий REPL і введи:

```python
>>> 0.1 + 0.2
0.30000000000000004
```

Схоже на помилку Python? Ні. Це фундаментальна особливість того як комп'ютери зберігають дробові числа. Вони зберігаються у двійковій системі числення, і деякі десяткові дроби — зокрема `0.1` і `0.2` — не мають точного двійкового представлення. Це як `1/3` у десятковій: можна писати `0.333...` нескінченно, і ніколи не отримати точного результату. Те саме з `0.1` у двійковій. При складанні мікро-похибки накопичуються і дають цей результат.

На практиці для більшості задач це незначно. Але якщо пишеш фінансовий застосунок де кожна копійка важлива — є модуль `decimal`:

```python
from decimal import Decimal

Decimal("0.1") + Decimal("0.2")    # Decimal('0.3') — точно
```

Тут важлива деталь: `Decimal` ініціалізується **рядком** `"0.1"`, а не числом `0.1`. Якщо написати `Decimal(0.1)` — передасться вже неточний `float` і вся точність Decimal нікуди не годиться:

```python
>>> Decimal(0.1)    # неточно!
Decimal('0.1000000000000000055511151231257827021181583404541015625')

>>> Decimal("0.1")  # точно
Decimal('0.1')
```

З тієї самої причини ніколи не порівнюй `float` через `==`:

```python
>>> 0.1 + 0.2 == 0.3
False
```

Про правильне порівняння через `math.isclose()` — в § 1.1.3.

## bool — логічне значення

`bool` — найпростіший тип за кількістю значень: їх рівно два. `True` — істина, `False` — хиба. З великої літери — це обов'язково, `true` або `TRUE` Python не розуміє і впаде з помилкою.

```python
is_admin:   bool = True
has_access: bool = False
```

Здається простим. Але `bool` стоїть за кожним рішенням в програмі — кожне `if`, кожна умова циклу, кожна перевірка спирається на логічні значення. Тому розуміти як він поводиться — не дрібниця.

### Звідки беруться логічні значення

Найчастіше `True` і `False` з'являються не як буквальні значення, а як результат порівнянь:

```python
age = 20

age > 18     # True
age < 18     # False
age == 20    # True  — рівно
age != 20    # False — не рівно
age >= 20    # True  — більше або рівно
age <= 19    # False — менше або рівно
```

І тут — найважливіше попередження цього параграфа. Запам'ятай назавжди:

**`=` — це присвоєння. `==` — це порівняння.**

Одинарний `=` — команда «прив'яжи значення до імені». Подвійний `==` — питання «чи рівні ці два об'єкти?» Плутати їх будеш — всі плутають. Особливо підступна ситуація де обидва варіанти синтаксично правильні, але дають зовсім різний ефект. Python зазвичай кричить про помилку, але не завжди.

Логічні значення можна комбінувати через `and`, `or`, `not`:

```python
age        = 25
is_citizen = True

can_vote = age >= 18 and is_citizen    # True — обидві умови виконані
has_id   = age >= 16 or is_citizen     # True — хоча б одна виконана
is_minor = not (age >= 18)             # False — заперечення
```

`and` повертає `True` тільки якщо обидва операнди `True`. `or` — якщо хоча б один. `not` — перевертає значення.

### bool — це насправді int

Є одна деталь, яку більшість підручників ховають глибоко або взагалі пропускають. У Python `bool` є **підтипом `int`**. `True` — буквально ціле число `1`, `False` — `0`. Це не абстракція — це технічний факт:

```python
>>> True == 1
True
>>> False == 0
True
>>> True + True
2
>>> True + False
1
>>> int(True)
1
>>> int(False)
0
```

На перший погляд — екзотика. Але звідси є практичне наслідування: можна підраховувати True-значення через `sum()`:

```python
answers = [True, False, True, True, False]
sum(answers)    # 3 — кількість True
```

Або — «скільки зі списку користувачів є адмінами»:

```python
sum(u.is_admin for u in users)    # число, а не список
```

### Truthy і Falsy: Python ширший за True/False

Python дозволяє використовувати будь-яке значення там де очікується `bool`. І кожен тип має своє «логічне значення». Загальне правило: **порожнє і нульове — False, все інше — True**:

```python
bool(0)        # False — нуль
bool(0.0)      # False — нульовий float
bool("")       # False — порожній рядок
bool(None)     # False — відсутність значення

bool(1)        # True
bool(-5)       # True — від'ємне теж ненульове
bool("hi")     # True — непорожній рядок
bool("False")  # True! — рядок "False" непорожній, Python не читає вміст
bool("0")      # True! — рядок "0" теж непорожній
```

Останні два — класичні пастки. `bool("False")` дає `True`, бо Python при конверсії в `bool` дивиться лише чи рядок порожній, а не що в ньому написано.

На практиці це дає зручний стиль:

```python
username = get_username()

# Довго
if username != "" and username is not None:
    greet(username)

# Ідіоматичний Python — те саме, коротше
if username:
    greet(username)
```

## None — відсутність значення

`None` — спеціальне значення яке означає «нічого». Не нуль. Не порожній рядок. Не `False`. Саме відсутність будь-якого значення — як порожній конверт, а не конверт з нулем всередині.

```python
result   = None
response = None
```

Де `None` зустрічається найчастіше — три ситуації варті запам'ятати.

**Перша — функція без `return`.** Якщо функція не має явного `return` — вона автоматично повертає `None`. Це одна з найпопулярніших пасток перших тижнів. Виглядає ось так:

```python
def greet(name):
    print(f"Привіт, {name}!")
    # немає return

result = greet("Оксана")
print(result)    # None
```

Функція виконалась і вивела текст — але нічого не **повернула**. Якщо ти спробуєш потім зробити `result.upper()` — отримаєш `AttributeError: 'NoneType' object has no attribute 'upper'`. І витратиш час на пошук де саме пішло не так, доки не згадаєш перевірити `return`.

**Друга — коли щось не знайдено.** Наприклад, метод `dict.get()` повертає `None` якщо ключ відсутній:

```python
user_data = {"name": "Оксана", "age": 28}

phone = user_data.get("phone")    # None — ключа немає
name  = user_data.get("name")     # "Оксана" — є
```

**Третя — початкове значення.** Змінна ще не отримала реального значення, але вже повинна існувати:

```python
current_user = None

# ... якийсь код, що може авторизувати користувача ...

if current_user is None:
    redirect_to_login()
```

### Як перевіряти на None

Перевіряти чи є значення `None` треба через `is`, а не через `==`:

```python
x = None

if x is None:        # правильно
    print("порожньо")

if x is not None:    # правильно
    process(x)

if x == None:        # технічно працює, але не рекомендується
    ...
```

Чому `is`, а не `==`? `None` в Python — singleton: на всю програму існує рівно один об'єкт `None`. Оператор `is` перевіряє **ідентичність** — чи дві змінні вказують на один і той самий об'єкт у пам'яті. Оскільки `None` один, `x is None` — точна і однозначна перевірка. `x == None` теж зазвичай спрацьовує, але деякі бібліотеки перевизначають оператор `==` — і тоді `== None` може дати несподіваний результат. `is None` — завжди надійний.

## type() і isinstance(): що за тип у змінній

Функція `type()` повертає тип будь-якого об'єкта:

```python
print(type("Привіт"))    # <class 'str'>
print(type(42))          # <class 'int'>
print(type(3.14))        # <class 'float'>
print(type(True))        # <class 'bool'>
print(type(None))        # <class 'NoneType'>
```

Корисна для налагодження. Коли не розумієш що за значення прийшло у функцію — `print(type(x))` одразу прояснить ситуацію.

Але для перевірок у логіці програми — `isinstance()`:

```python
x = 42

isinstance(x, int)            # True
isinstance(x, float)          # False
isinstance(x, (int, float))   # True — перевірка на кілька типів одразу
```

Різниця між ними — тонка але важлива. Пам'ятаєш що `bool` є підтипом `int`? Ось де це проявляється:

```python
flag = True

type(flag) == int       # False — тип точно bool, не int
isinstance(flag, int)   # True  — bool є підтипом int, тому True
```

`isinstance()` враховує успадкування типів, `type()` — ні. Для більшості практичних перевірок саме `isinstance()` дає правильну відповідь.

Загальне правило: **`isinstance()` для перевірок у логіці, `type()` для налагодження**.

## Перетворення між типами

Python майже ніколи не перетворює типи автоматично — на відміну від JavaScript де `"5" + 3` дасть `"53"`, тут це одразу `TypeError`. Але перетворити можна явно, викликавши потрібний тип як функцію:

```python
# Рядок → число
int("42")            # 42
float("3.14")        # 3.14
int("3.14")          # ValueError — "3.14" не ціле
int(float("3.14"))   # 3 — спочатку в float, потім відкидаємо дріб

# Число → рядок
str(42)              # "42"
str(True)            # "True"
str(None)            # "None"

# Число → int (відкидає дробову частину, не округлює!)
int(3.9)             # 3  — відкидає, а не округлює
int(-3.9)            # -3 — теж відкидає (до нуля, не до -4)
int(True)            # 1
int(False)           # 0
```

Зверни увагу на `int(3.9)` — це `3`, не `4`. Якщо потрібне округлення — `round()`, не `int()`:

```python
round(3.9)         # 4 — математичне округлення
round(3.14159, 2)  # 3.14 — з двома знаками після крапки
```

І ще одна пастка з конверсією в `bool`:

```python
bool("")       # False — порожній рядок
bool("False")  # True! — рядок "False" непорожній
bool("0")      # True! — рядок "0" теж непорожній
```

Python не читає вміст рядка при конверсії в `bool` — лише перевіряє чи він порожній. Ця деталь кусає людей, які очікують що `bool("False")` дасть `False`.

## Анотації типів

Починаючи з Python 3.5, можна явно вказувати очікуваний тип змінної:

```python
name:  str   = "Оксана"
age:   int   = 28
price: float = 19.99
flag:  bool  = True
```

Python ці анотації **не перевіряє під час виконання** — вони для людини і для статичних аналізаторів. Pylance у VS Code читає їх і підкреслює місця де тип не збігається — ще до запуску:

```python
age: int = 28
age = "двадцять вісім"   # Pylance підкреслить: expected int, got str
```

Анотації — це не обов'язок і не бюрократія. Це спосіб зробити код зрозумілішим для читача — включно з тобою самим через пів року. Ми будемо додавати їх поступово, спочатку там де вони найбільше допомагають.

## Підсумок

Перший параграф з кодом — і одразу п'ять типів, правило `is None`, пастка з `bool("False")`, нюанс `int(3.9)` і різниця між `type()` та `isinstance()`. Це не мало. Але все це — речі які зустрінуться в кожній програмі і які краще знати вже зараз, ніж натикатись на них у найнезручніший момент.

Три речі щоб взяти і не забути:

`=` — присвоєння, `==` — порівняння. Плутати будеш. Це нормально — головне розуміти різницю.

`float` не точний. `0.1 + 0.2 ≠ 0.3`. Для грошей — `Decimal("0.1")`, не `Decimal(0.1)`.

`None` — не нуль і не `False`. Функція без `return` повертає `None`. Перевіряти через `is None`.
