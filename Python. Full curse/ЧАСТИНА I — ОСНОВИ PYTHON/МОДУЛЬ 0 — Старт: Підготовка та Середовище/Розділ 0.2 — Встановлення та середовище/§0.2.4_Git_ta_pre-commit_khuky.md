# § 0.2.4 Git та pre-commit хуки

---

## Проблема, яку вирішує Git

Уяви таку ситуацію. Ти тиждень пишеш програму. Вона вже майже працює, і ти вирішуєш зробити велику зміну — переписати центральну функцію, яка, здається, влаштована неоптимально. Переписуєш. Запускаєш. Щось пішло не так — і тепер не працює те, що до цього працювало бездоганно. Ти намагаєшся повернутись до попередньої версії. Але попередньої версії немає — ти просто редагував той самий файл.

Або інша ситуація. Вас двоє, і ви разом пишете один проєкт. Ти змінюєш один файл, твій напарник одночасно змінює той самий файл. Ви надсилаєте файли один одному через месенджер. Чия версія правильна? Як об'єднати зміни, не загубивши нічого важливого?

Обидва сценарії — добре відомі болі будь-якого розробника, який спробував обійтись без спеціального інструменту. Варіанти «зберігати копії: `project_v1.py`, `project_v2_final.py`, `project_v2_final_СПРАВЖНІЙ.py`» — це народний метод, який погано масштабується і надійно призводить до хаосу.

Git — система контролю версій — вирішує обидві ці проблеми. Він зберігає повну історію змін кожного файлу, дозволяє повернутись до будь-якого попереднього стану, і надає механізм для об'єднання змін від кількох людей без конфліктів — або з чітким позначенням місць, де конфлікти є і їх треба вирішити вручну.

Git — не тільки для командної роботи. Навіть якщо ти пишеш код на самоті, він дає тобі суперсилу: можливість експериментувати без страху. Спробував щось нове — не вийшло — відкатився до попереднього стану одною командою. Це розкіш, від якої дуже важко відмовитись після того як спробував.

## Як Git думає про зміни

Git не зберігає файли цілком при кожній зміні — він зберігає *різниці*. Коли ти кажеш Git «запам'ятай поточний стан», він порівнює що змінилося відносно попереднього збереженого стану і записує саме ці зміни — рядок доданий тут, рядок видалений там. Це робить його дуже ефективним: навіть великий проєкт з тисячами комітів займає розумний обсяг місця.

Кожне таке «запам'ятовування» називається **комітом** (від англ. commit — зафіксувати). Коміт — це точка в історії проєкту. У кожного коміту є автор, дата, і текстовий коментар — повідомлення, яке пояснює що саме і чому змінилось.

Послідовність комітів утворює лінійну або розгалужену **гілку** (branch). За замовчуванням є одна гілка — `main` (або `master` у старіших проєктах). Коли ти хочеш спробувати щось нове, не ризикуючи стабільним кодом, — створюєш нову гілку, робиш зміни там, і коли все добре — **зливаєш** (merge) її назад у `main`. Гілки — це окрема велика тема, але знати що вони існують корисно вже зараз.

## Встановлення Git

На **macOS** Git зазвичай вже є — він встановлюється разом з Xcode Command Line Tools. Перевір:

```bash
git --version
```

Якщо команда не знайдена — встанови Command Line Tools:

```bash
xcode-select --install
```

Або через Homebrew:

```bash
brew install git
```

На **Linux** (Ubuntu/Debian):

```bash
sudo apt update
sudo apt install git
```

На **Windows** завантаж інсталятор з `git-scm.com`. Під час встановлення є кілька екранів з опціями — для більшості питань підходять значення за замовчуванням. Єдине на що варто звернути увагу: на кроці «Choosing the default editor used by Git» рекомендую обрати Visual Studio Code, якщо він вже встановлений. Це означає, що коли Git потребує відкрити текстовий редактор (наприклад, для написання повідомлення коміту), він відкриє VS Code, а не вбудований Vim, що може спантеличити новачка.

Після встановлення перевір:

```bash
git --version
```

## Перше налаштування: ім'я та email

Git записує автора до кожного коміту. Тому перше що потрібно зробити після встановлення — представитись:

```bash
git config --global user.name "Твоє Ім'я"
git config --global user.email "твій@email.com"
```

Прапор `--global` означає що це налаштування застосовується до всіх репозиторіїв на цьому комп'ютері. Якщо для якогось конкретного проєкту потрібні інші дані — можна налаштувати окремо без `--global`, перебуваючи всередині директорії цього проєкту.

Email має збігатися з тим, який ти вкажеш при реєстрації на GitHub або GitLab — це важливо для правильного зв'язку між локальними комітами і твоїм профілем у веб-сервісі.

Переглянути поточні глобальні налаштування:

```bash
git config --global --list
```

## Ініціалізація репозиторію

**Репозиторій** (або просто «репо») — це директорія, за якою Git спостерігає. Ініціалізувати репозиторій означає сказати Git: «починай відстежувати зміни у цій папці».

Перейди в директорію свого проєкту і виконай:

```bash
git init
```

Git створить приховану піддиректорію `.git/` — саме там зберігається вся внутрішня кухня: база комітів, налаштування, тощо. Ти ніколи не редагуєш цю директорію вручну. Якщо хочеш «знищити» репозиторій і зробити директорію звичайною папкою — просто видали `.git/`.

Перевірити стан репозиторію після ініціалізації:

```bash
git status
```

Ця команда — одна з тих, які ти будеш запускати постійно. Вона показує які файли змінені, які вже підготовлені до коміту, а які Git взагалі не відстежує.

## .gitignore: повний розбір

У попередньому параграфі про віртуальні середовища ми вже познайомились з ідеєю `.gitignore` — що такий файл існує і що деякі речі краще не класти до репозиторію. Тут розберемо тему повністю: синтаксис, ієрархію, типові пастки і ситуації, коли звичайного підходу недостатньо.

### Синтаксис шаблонів

`.gitignore` — це звичайний текстовий файл. Кожен рядок у ньому описує шаблон: що саме Git має ігнорувати. Правила такі.

Порожні рядки і рядки що починаються з `#` — це коментарі, Git їх пропускає. Це корисно для структурування файлу і пояснень:

```
# Віртуальні середовища
venv/
.venv/

# Байт-код Python
__pycache__/
*.pyc
```

**Зірочка `*`** замінює будь-яку послідовність символів у межах одного сегменту шляху — тобто не перетинає межу директорій:

```
*.pyc        # будь-який файл з розширенням .pyc у будь-якій директорії
*.log        # будь-який .log файл
temp_*       # будь-який файл або директорія що починається з temp_
```

**Подвійна зірочка `**`** перетинає межі директорій — відповідає будь-якій кількості рівнів вкладеності:

```
**/__pycache__/    # будь-яка директорія __pycache__ на будь-якому рівні
logs/**/*.log      # будь-який .log файл у будь-якій піддиректорії logs/
```

**Слеш `/` в кінці** означає що шаблон стосується директорії, а не файлу. Без слеша Git ігнорує і файл, і директорію з такою назвою; зі слешем — тільки директорію:

```
venv/          # ігнорувати директорію venv
build/         # ігнорувати директорію build
.env           # ігнорувати і файл, і директорію з назвою .env
```

**Слеш `/` на початку** прив'язує шаблон до кореня репозиторію — тобто тільки до конкретного рівня, не рекурсивно:

```
/.env          # ігнорувати .env тільки в корені, не у піддиректоріях
/config.yaml   # тільки config.yaml у корені
```

**Знак питання `?`** замінює рівно один будь-який символ:

```
report_?.txt   # відповідає report_1.txt, report_a.txt, але не report_10.txt
```

**Квадратні дужки `[]`** описують набір символів, як у регулярних виразах:

```
*.[oa]         # файли з розширенням .o або .a
file[0-9].txt  # file0.txt, file1.txt ... file9.txt
```

**Знак оклику `!`** на початку рядка — це **негативний шаблон**: виключення з виключень. Якщо файл потрапляє під якийсь шаблон ігнорування, але наступний шаблон з `!` до нього підходить — файл перестає ігноруватись:

```
*.log          # ігнорувати всі .log файли
!important.log # але не important.log — його Git відстежуватиме
```

Важлива деталь: негативний шаблон не може скасувати ігнорування файлу всередині проігнорованої директорії. Тобто якщо `logs/` ігнорується цілком — `!logs/important.log` не допоможе. Спочатку треба прибрати ігнорування директорії, і тоді вже керувати файлами всередині.

### Повний .gitignore для Python-проєкту

З розумінням синтаксису читати реальний файл набагато легше:

```
# ── Віртуальні середовища ────────────────────────────────────
venv/
.venv/
env/
ENV/

# ── Байт-код і кеш Python ────────────────────────────────────
__pycache__/
*.py[cod]
*$py.class
*.so

# ── Дистрибуція та збірка ────────────────────────────────────
dist/
build/
*.egg-info/
*.egg
MANIFEST

# ── Тестування і покриття ────────────────────────────────────
.coverage
.coverage.*
htmlcov/
.pytest_cache/
.tox/

# ── Змінні середовища і секрети ──────────────────────────────
.env
.env.*
!.env.example

# ── Редактори і IDE ──────────────────────────────────────────
.vscode/
.idea/
*.swp
*.swo
*~

# ── Операційні системи ──────────────────────────────────────
.DS_Store
.DS_Store?
Thumbs.db
Desktop.ini

# ── Логи і тимчасові файли ───────────────────────────────────
*.log
*.tmp
*.temp
```

Зверни увагу на рядок `!.env.example` — це той самий негативний шаблон. Ми ігноруємо всі `.env`-файли, але робимо виняток для `.env.example`. Це стандартна практика: `.env` містить реальні секрети і не йде в репозиторій, а `.env.example` — це шаблон з усіма потрібними ключами, але без значень, щоб інші розробники знали які змінні потрібно заповнити:

```
# .env.example — цей файл в репозиторії
DATABASE_URL=
SECRET_KEY=
API_KEY=
```

```
# .env — цей файл НЕ в репозиторії
DATABASE_URL=postgresql://user:password@localhost/mydb
SECRET_KEY=super-secret-key-12345
API_KEY=sk-реальний-ключ
```

### Ієрархія: кілька .gitignore в одному проєкті

`.gitignore` може жити не тільки в корені репозиторію. Git читає `.gitignore`-файли рекурсивно — якщо в піддиректорії є власний `.gitignore`, він застосовується до файлів саме цієї піддиректорії і нижче.

Це корисно у великих проєктах, де різні модулі мають різні специфічні файли для ігнорування. Наприклад, у директорії `docs/` може бути власний `.gitignore`, який ігнорує згенеровані HTML-файли документації, а в корені — загальний для всього проєкту.

```
my-project/
├── .gitignore          ← загальний для всього проєкту
├── src/
│   └── .gitignore      ← специфічний для src/
└── docs/
    └── .gitignore      ← специфічний для docs/
```

### Глобальний .gitignore: файли для всіх проєктів

Є файли, які ти ніколи не хочеш бачити в жодному репозиторії на своїй машині — незалежно від проєкту. Наприклад, `.DS_Store` на macOS або `.idea/` від PyCharm. Додавати їх в `.gitignore` кожного проєкту — зайня робота.

Для цього є глобальний `.gitignore` — один файл для всієї машини. Налаштовується через конфігурацію Git:

```bash
git config --global core.excludesfile ~/.gitignore_global
```

Потім створи файл `~/.gitignore_global` і додай туди все машино-специфічне:

```
# macOS
.DS_Store
.DS_Store?
.Spotlight-V100
.Trashes

# Windows
Thumbs.db
Desktop.ini
ehthumbs.db

# JetBrains IDE (PyCharm, IntelliJ)
.idea/

# VS Code (якщо не хочеш ділитись налаштуваннями)
.vscode/

# Vim
*.swp
*.swo
*~
```

Тепер ці файли ігноруватимуться автоматично у кожному репозиторії, і тобі не потрібно думати про них при старті нового проєкту.

### Якщо файл вже потрапив у репозиторій

Ось класична пастка: ти забув додати щось до `.gitignore` і зробив коміт. Тепер Git уже відстежує цей файл, і просто додавання його в `.gitignore` нічого не змінить — він продовжуватиме відстежуватись.

Причина проста: `.gitignore` керує тільки **невідстежуваними** файлами. Якщо файл вже в репозиторії — він вже відстежується, і `.gitignore` на нього не впливає.

Рішення — прибрати файл з індексу Git (зупинити відстеження), не видаляючи його з диска:

```bash
git rm --cached назва_файлу
```

Для директорії — рекурсивно:

```bash
git rm --cached -r назва_директорії/
```

Після цього зроби коміт — Git зафіксує що файл більше не відстежується, але фізично він залишиться у тебе на диску. І тепер `.gitignore` вже спрацює.

> ⚠️ **Увага:** Якщо файл із секретами вже потрапив у публічний репозиторій — просто видалити його з `.gitignore` і прибрати з відстеження недостатньо. Файл залишається в історії комітів і будь-хто може його переглянути. У такому випадку треба вважати секрет скомпрометованим — змінити пароль, перегенерувати ключ — і звернутись до документації GitHub або GitLab про повне видалення файлу з історії репозиторію.

### Перевірити чому файл ігнорується

Інколи файл зникає з `git status` і незрозуміло якому саме шаблону він відповідає. Git може пояснити:

```bash
git check-ignore -v назва_файлу
```

Виводить шлях до файлу `.gitignore` і номер рядка із шаблоном, який спрацював. Дуже корисно для налагодження складних конфігурацій.

Перевірити чи відстежується конкретний файл взагалі:

```bash
git ls-files --error-unmatch назва_файлу
```

Якщо файл відстежується — команда нічого не виводить. Якщо ні — повертає помилку. Трохи незвично, але саме так це влаштовано.

> 💡 **Порада:** Сайт `gitignore.io` (або `toptal.com/developers/gitignore`) генерує готові `.gitignore`-файли для будь-якої комбінації мов, середовищ і операційних систем. Введи «Python», «VS Code», «macOS» — отримаєш вичерпний файл, який можна взяти за основу і підправити під свої потреби.

## Перший коміт: цикл роботи з Git

Робота з Git будується навколо простого циклу з трьох кроків: змінив файли → підготував до коміту → зафіксував коміт. Це повторюється знову і знову.

**Крок 1: перевірити стан**

```bash
git status
```

Git покаже які файли змінені (червоним) і які вже підготовлені до коміту (зеленим). Новий репозиторій після `git init` показуватиме всі файли як «untracked» — ще не відстежувані.

**Крок 2: підготувати файли (staging)**

```bash
git add назва_файлу.py
```

Або додати всі змінені файли одразу:

```bash
git add .
```

Крапка означає «поточна директорія і все в ній». `git add` переміщує зміни у так звану **staging area** — це проміжна зона, де збираються зміни перед тим як потрапити в коміт. Можна додавати файли поступово, перевіряти що потрапило, і лише тоді робити коміт. Це дозволяє формувати логічні, цілісні коміти навіть якщо одночасно змінювалось багато різних речей.

**Крок 3: зробити коміт**

```bash
git commit -m "feat: init project"
```

Прапор `-m` дозволяє написати повідомлення коміту прямо в командному рядку. Якщо опустити `-m` — Git відкриє текстовий редактор для написання повідомлення.

Що саме написати у повідомленні — питання, яке з часом набуває важливості. Зараз достатньо знати одне просте правило: повідомлення коміту пояснює *що* змінилось і *чому*, а не *як* (це видно з коду). «Виправлено помилку» — погане повідомлення. «Виправлено: помилка ділення на нуль при порожньому списку вхідних даних» — хороше.

## Conventional Commits: стандарт повідомлень

У реальних проєктах і командах часто використовують угоду про формат повідомлень, яка називається **Conventional Commits**. Вона проста: повідомлення починається з типу змін, двокрапки і короткого опису.

Основні типи:

`feat:` — нова функціональність
`fix:` — виправлення помилки
`docs:` — зміни у документації
`refactor:` — рефакторинг коду без зміни поведінки
`test:` — додавання або виправлення тестів
`chore:` — технічні зміни, що не стосуються логіки (оновлення залежностей, налаштування)
`style:` — зміни форматування, відступів, без змін у логіці

Приклади:

```
feat: додати функцію пошуку за ключовим словом
fix: виправити помилку при обробці порожнього рядка
docs: оновити README з інструкцією з встановлення
refactor: перенести логіку валідації у окремий модуль
chore: оновити залежності до актуальних версій
```

Ця угода корисна вже на самому початку навчання з двох причин. По-перше, вона змушує думати *що* ти зробив перш ніж написати повідомлення — це дисциплінує. По-друге, коли через рік відкриєш старий проєкт і побачиш список комітів, ти одразу розумітимеш де що змінювалось, не читаючи кожен коміт повністю.

## Корисні команди Git: перший набір

Це команди, які ти будеш використовувати щодня. Не намагайся запам'ятати всі одразу — вони запам'ятаються самі через практику.

Переглянути історію комітів:

```bash
git log
```

Скорочена однорядкова версія — зручніша для огляду:

```bash
git log --oneline
```

Переглянути що змінилось у файлах відносно останнього коміту:

```bash
git diff
```

Якщо файли вже в staging area (після `git add`) — використовуй:

```bash
git diff --staged
```

Відмінити зміни у файлі і повернути його до стану останнього коміту:

```bash
git restore назва_файлу.py
```

> ⚠️ **Увага:** `git restore` незворотно відкидає незакомічені зміни у файлі. Якщо ти щось написав і не зберіг у коміт — після цієї команди воно зникне назавжди. Перед відновленням переконайся що справді хочеш це зробити.

Прибрати файл зі staging area (якщо додав `git add`, але ще не комітив і передумав):

```bash
git restore --staged назва_файлу.py
```

Переглянути стислу різницю між двома комітами:

```bash
git diff abc1234 def5678
```

де `abc1234` і `def5678` — перші кілька символів хешів комітів, які видно у `git log --oneline`.

## GitHub: Git ≠ GitHub

Тут варто зупинитись і розвіяти поширену плутанину.

**Git** — це програма, інструмент контролю версій, яка встановлюється на твій комп'ютер. Вона працює локально, без інтернету.

**GitHub** — це веб-сервіс, де можна зберігати репозиторії онлайн: ділитись кодом, співпрацювати з іншими, показувати своє портфоліо. Є й аналоги — GitLab, Bitbucket — з тими самими функціями.

GitHub використовує Git як основу, але це різні речі. Git можна використовувати без GitHub. GitHub без Git — не існує.

Щоб відправити локальний репозиторій на GitHub: зареєструйся, створи новий репозиторій, і GitHub покаже команди для «підключення» локального репо до онлайн-сховища:

```bash
git remote add origin https://github.com/твій-нікнейм/назва-репо.git
git branch -M main
git push -u origin main
```

`git remote add origin` — каже Git де знаходиться «віддалений» (remote) репозиторій. `origin` — стандартна назва за угодою. `git push` — відправляє коміти на сервер. Після першого разу з прапором `-u` достатньо просто `git push`.

Завантажити зміни з сервера на локальний комп'ютер:

```bash
git pull
```

## pre-commit: автоматичні перевірки перед кожним комітом

Тепер — до pre-commit хуків. Це механізм, який дозволяє автоматично запускати певні перевірки щоразу коли ти намагаєшся зробити коміт. Якщо хоч одна перевірка не пройшла — коміт не відбувається і тобі показується що саме не так.

Навіщо це потрібно? Уяви: ти пишеш код, забуваєш відформатувати файл, робиш коміт. В репозиторій потрапляє неохайно відформатований код. Або забуваєш видалити невикористаний імпорт. Або випадково залишаєш `print()`, який клав для налагодження. Або, що найгірше, стаєш причиною помилки у всієї команди, бо закомітив файл з синтаксичною помилкою.

pre-commit — це «воротар» між тобою і репозиторієм. Перш ніж зміни потраплять у коміт, вони проходять через перевірки. Black переформатував? Коміт відбудеться з відформатованим кодом. Ruff знайшов невикористаний імпорт? Коміт зупиниться, Ruff автоматично видалить імпорт, ти побачиш що змінилось і зробиш коміт ще раз.

Це може здаватись зайвим ускладненням на початку. Насправді це найкраще, що можна зробити для якості свого коду з першого дня — виробити звичку, при якій неохайний код просто не може потрапити в репозиторій технічно.

---

## Встановлення і налаштування pre-commit

Спочатку встанови бібліотеку (в активованому віртуальному середовищі):

```bash
pip install pre-commit
```

Потім у корені проєкту створи файл `.pre-commit-config.yaml`. Це конфігурація — список перевірок і звідки їх взяти:

```yaml
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-added-large-files

  - repo: https://github.com/psf/black
    rev: 24.4.2
    hooks:
      - id: black

  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.4.4
    hooks:
      - id: ruff
        args: [--fix]

  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.10.0
    hooks:
      - id: mypy
        additional_dependencies: []
```

Розберемо кожен блок.

**`pre-commit-hooks`** — набір простих загальних перевірок від команди pre-commit:
- `trailing-whitespace` — видаляє зайві пробіли в кінці рядків
- `end-of-file-fixer` — забезпечує що файл завершується рядком переносу
- `check-yaml` — перевіряє синтаксис YAML-файлів
- `check-added-large-files` — попереджає якщо намагаєшся закомітити великий файл (наприклад, випадково доданий датасет або бінарний файл)

**`black`** — форматує Python-код. Якщо форматування відрізняється від стандарту Black — він переформатує файл, і коміт зупиниться з повідомленням «1 file reformatted». Запускаєш `git add` знову і комітиш ще раз — тепер код вже відформатований.

**`ruff`** з аргументом `--fix` — запускає лінтер і автоматично виправляє все, що можна виправити автоматично. Те, що не можна — показує як попередження і зупиняє коміт, щоб ти вирішив сам.

**`mypy`** — перевіряє типи. На початку навчання mypy може бути надлишковим — він суворий і видає попередження там, де ти ще не думаєш про типи. Для першого проєкту можна закоментувати цей блок і повернутись до нього пізніше, коли дійдеш до теми типізації.

> ⚠️ **Увага:** Поле `rev` у кожному блоці — це конкретна версія хука. Версії вказані на момент написання цього матеріалу і можуть застаріти. Щоб оновити всі хуки до актуальних версій одразу — виконай:
> ```bash
> pre-commit autoupdate
> ```
> Це оновить `rev` у `.pre-commit-config.yaml` до останніх стабільних версій.

## Активація хуків

Після того як `.pre-commit-config.yaml` створений, треба встановити хуки в репозиторій — сказати Git «запускай ці перевірки перед кожним комітом»:

```bash
pre-commit install
```

Git зберігає хуки у директорії `.git/hooks/`. pre-commit додає туди скрипт `pre-commit`, який запускається автоматично перед кожним `git commit`.

Перший запуск хуків займе деякий час — pre-commit скачає і встановить потрібні залежності для кожного хука в ізольоване середовище. Наступні запуски будуть миттєвими — все вже завантажено.

Запустити всі хуки вручну на всіх файлах, не роблячи коміт — корисно для першої перевірки або після зміни конфігурації:

```bash
pre-commit run --all-files
```

Тимчасово пропустити хуки (іноді потрібно, але не зловживай):

```bash
git commit --no-verify -m "wip: тимчасовий коміт"
```

> 💡 **Порада:** `.pre-commit-config.yaml` додається до репозиторію разом з кодом (`git add .pre-commit-config.yaml`). Тоді кожен, хто клонує проєкт, після запуску `pre-commit install` отримає ті самі перевірки. Це забезпечує єдиний стандарт якості для всієї команди або для тебе самого на різних машинах.

## Типовий день з Git: як це виглядає на практиці

Щоб усе вище перетворилось на живу картину — ось як виглядає звичайна робота з Git протягом одного сеансу.

Зайшов у проєкт, перевірив стан:

```bash
git status
```

Написав код. Щось працює. Хочу зафіксувати цей момент:

```bash
git add .
git commit -m "feat: додати функцію валідації email"
```

pre-commit запускається автоматично. Ruff знайшов невикористаний імпорт — видалив його. Коміт зупинився. Виконую знову:

```bash
git add .
git commit -m "feat: додати функцію валідації email"
```

Тепер все чисто. Коміт пройшов.

Продовжую писати. Зіпсував щось. Хочу подивитись що змінилось відносно останнього коміту:

```bash
git diff
```

Вирішую відкинути зміни в одному файлі:

```bash
git restore utils.py
```

Відправляю на GitHub:

```bash
git push
```

Від початку до кінця — кілька хвилин зусиль, зате вся робота зафіксована, відтворювана і захищена.

## Підсумок і завдання

Git — не опція і не зручність для «потім». Це основна частина робочого процесу будь-якого розробника, і починати з нього з першого проєкту — правильне рішення. Звичка комітити часто і з осмисленими повідомленнями виробляється саме на початку, і потім залишається назавжди.

pre-commit хуки — це твій автоматичний контролер якості, який не дає неохайному коду потрапити в репозиторій. Black форматує, Ruff знаходить помилки, базові хуки прибирають технічний сміття. Один раз налаштував — і більше не думаєш про це.

## 📋 Завдання 0 — Налаштування середовища

Перше практичне завдання курсу об'єднує все з цього розділу: Python, venv, VS Code і Git. Виконай кожен пункт послідовно.

Встанови Python 3.12 або новіший через pyenv, VS Code з усіма вісьмома розширеннями і Git з початковим налаштуванням імені та email.

Створи робочу директорію для всього курсу і ініціалізуй її:

```bash
mkdir python-course
cd python-course
git init
pyenv local 3.12.3
python -m venv venv
source venv/bin/activate    # Linux / macOS
# або venv\Scripts\activate на Windows
```

Створи `.gitignore` з правильними виключеннями для Python-проєкту.

Встанови pre-commit і налаштуй `.pre-commit-config.yaml` з Black і Ruff (mypy поки можна не додавати):

```bash
pip install pre-commit
pre-commit install
```

Напиши перший файл `hello.py` — програму, яка виводить твоє ім'я і поточний рік:

```python
from datetime import date

name = "Твоє Ім'я"
year = date.today().year
```

```bash
python hello.py
```

Переконайся що все виводиться правильно. Потім зроби перший коміт:

```bash
git add .
git commit -m "feat: init project"
```

pre-commit запуститься автоматично — перевірить і відформатує файл. Якщо коміт зупинився через виправлення — виконай `git add .` ще раз і повтори коміт.
