# § 0.2.1 Python та pyenv: встановлення і управління версіями

---

## Перший практичний крок

До цього моменту ми говорили про програмування як ідею — про алгоритми, про мислення, про те, де Python зустрічається у реальному житті. Тепер пора зробити перший практичний крок: підготувати середовище, в якому ти будеш писати код.

Звучить нудно. Насправді — ні. Те, як влаштоване середовище розробника, дуже багато говорить про природу самого програмування: про те, що реальні проєкти мають залежності, що версії мають значення, що одна й та сама машина може одночасно слугувати для кількох зовсім різних задач. Розуміти, чому середовище влаштоване саме так, — означає розуміти трохи більше про те, як насправді влаштована розробка програмного забезпечення.

Тому цей параграф — не просто «скопіюй і встав команду». Він пояснює, що відбувається і чому саме так.

---

## Версії Python: чому це взагалі питання

Якщо ти зараз відкриєш сайт python.org, ти побачиш кнопку завантаження з номером версії — скажімо, Python 3.12.3. Але якщо заглибитись трохи далі, виявиться, що там також є Python 3.11, 3.10, 3.9 — і навіть старіші. Навіщо стільки версій, якщо можна просто взяти найновішу?

Відповідь — в реальності розробки.

Уяви, що ти приходиш на роботу і тобі дають проєкт, який розпочали два роки тому. Він написаний на Python 3.9, використовує певний набір бібліотек, які на той момент підтримували саме цю версію. За два роки вийшло кілька нових версій Python, деякі бібліотеки оновились, деякі змінили інтерфейс, деякі були замінені іншими. Якщо просто встановити Python 3.12 і запустити старий проєкт — він може не запрацювати: щось застаріло, щось змінило поведінку, щось конфліктує.

Або інша ситуація: ти одночасно ведеш два власні проєкти. Один використовує Python 3.11, бо запускається на корпоративному сервері, де версія зафіксована адміністратором. Другий — твій новий проєкт, де ти хочеш використовувати функції, які з'явились у 3.12. На одній машині тобі потрібні обидві версії, і вони не повинні заважати одна одній.

Це не надумана проблема — це повсякденна реальність розробника. Саме для її вирішення і існує pyenv.

---

## Що таке pyenv і що він вирішує

pyenv — це інструмент для управління версіями Python на одній машині. Він дозволяє встановити скільки завгодно різних версій Python, перемикатися між ними однією командою і навіть прив'язати конкретну версію до конкретної директорії з проєктом — так, що заходячи в одну папку ти автоматично опиняєшся в одній версії Python, а в іншу папку — в іншій.

Без pyenv стандартна ситуація на комп'ютері виглядає так: є одна «системна» версія Python, яка встановлена разом з операційною системою або завантажена вручну з сайту. Вона одна, вона глобальна, і будь-який проєкт на цій машині використовує саме її. Змінити версію — означає перевстановити Python для всієї системи. Мати дві версії одночасно без конфліктів — складно і ненадійно без спеціального інструменту.

З pyenv картина інша: є набір ізольованих Python-версій, кожна з яких живе у власній директорії і не знає про існування інших. pyenv підміняє виклик команди `python` так, щоб вона завжди вказувала на потрібну версію залежно від контексту. Для тебе це виглядає просто: написав `python` у терміналі — отримав потрібну версію. Решту бере на себе pyenv.

## Встановлення pyenv на macOS і Linux

На macOS і Linux pyenv встановлюється через термінал і відчуває себе там цілком природно — він і розроблений для Unix-подібних систем.

Найпростіший спосіб — через офіційний скрипт встановлення. Відкрий термінал і виконай:

```bash
curl https://pyenv.run | bash
```

Ця команда завантажує і запускає скрипт, який клонує репозиторій pyenv і розміщує його у домашній директорії користувача — зазвичай у `~/.pyenv`. Скрипт також встановлює кілька пов'язаних плагінів: `pyenv-update` для оновлення самого pyenv і `pyenv-virtualenv` для роботи з віртуальними середовищами, про які ми поговоримо окремо.

Після виконання скрипт скаже тобі, що потрібно додати кілька рядків до конфігурації оболонки. Це важливий крок, без якого pyenv не буде «бачити» командний рядок. Для більшості систем з Bash — це файл `~/.bashrc` або `~/.bash_profile`, для Zsh — `~/.zshrc`. Додай туди наступне:

```bash
export PYENV_ROOT="$HOME/.pyenv"
export PATH="$PYENV_ROOT/bin:$PATH"
eval "$(pyenv init -)"
eval "$(pyenv virtualenv-init -)"
```

Після збереження файлу треба або перезапустити термінал, або виконати команду `source ~/.zshrc` (або відповідний файл для твоєї оболонки), щоб зміни набрали чинності.

На macOS є ще один шлях — через Homebrew, пакетний менеджер для macOS. Якщо Homebrew вже встановлений:

```bash
brew update
brew install pyenv
```

Homebrew сам подбає про розміщення pyenv і скаже, що додати до конфігурації.

Перевірити, чи все спрацювало, можна однією командою:

```bash
pyenv --version
```

Якщо у відповідь з'явився номер версії — pyenv встановлений і готовий до роботи.

## Встановлення pyenv на Windows

З Windows ситуація трохи інша, бо pyenv створювався для Unix-систем і офіційно Windows не підтримує.

Є два основних варіанти.

Перший і рекомендований — WSL, тобто Windows Subsystem for Linux. Це функція Windows 10 і 11, яка дозволяє запустити повноцінний Linux-термінал прямо всередині Windows без віртуальної машини. Якщо ти плануєш серйозно займатись розробкою, WSL — це інвестиція, яка окупиться дуже швидко: більшість серверів у світі працюють на Linux, більшість інструментів розробки першочергово розробляються для Linux, і мати Linux-середовище під рукою означає менше проблем із сумісністю.

Встановити WSL можна через PowerShell, запущений від імені адміністратора:

```powershell
wsl --install
```

Ця команда встановить WSL і Ubuntu за замовчуванням. Після перезавантаження у тебе буде повноцінний Ubuntu-термінал, де можна встановити pyenv так само, як описано вище для Linux.

Другий варіант — `pyenv-win`, неофіційний порт pyenv для Windows. Він встановлюється через PowerShell:

```powershell
Invoke-WebRequest -UseBasicParsing -Uri "https://raw.githubusercontent.com/pyenv-win/pyenv-win/master/pyenv-win/install-pyenv-win.ps1" -OutFile "./install-pyenv-win.ps1"; &"./install-pyenv-win.ps1"
```

`pyenv-win` покриває більшість базових потреб і підходить, якщо з якоїсь причини WSL не варіант. Але він менш функціональний, ніж оригінальний pyenv, і деякі команди працюють дещо інакше.

## Встановлення конкретної версії Python

Після того як pyenv налаштований, можна встановити Python. pyenv завантажує і компілює Python із вихідного коду — тобто кожна версія будується спеціально для твоєї системи. Це означає, що потрібні деякі системні залежності — компілятор і бібліотеки. Якщо вони відсутні, pyenv попередить про це.

На Ubuntu і Debian-подібних системах встановлення залежностей:

```bash
sudo apt update
sudo apt install -y make build-essential libssl-dev zlib1g-dev \
  libbz2-dev libreadline-dev libsqlite3-dev wget curl llvm \
  libncursesw5-dev xz-utils tk-dev libxml2-dev libxmlsec1-dev \
  libffi-dev liblzma-dev
```

На macOS, якщо встановлений Homebrew, більшість залежностей вже є або встановлюються разом з Xcode Command Line Tools:

```bash
xcode-select --install
```

Після цього — власне встановлення Python. Подивитись, які версії доступні:

```bash
pyenv install --list
```

Список буде довгий — там є офіційні версії CPython (саме їх ти і будеш використовувати), а також альтернативні реалізації: PyPy, Jython, Anaconda. Нас зараз цікавлять рядки виду `3.12.3` або `3.11.9` — звичайний CPython.

Встановити конкретну версію:

```bash
pyenv install 3.12.3
```

pyenv завантажить вихідний код, скомпілює його і розмістить у `~/.pyenv/versions/3.12.3/`. Це займе кілька хвилин — компіляція не миттєва.

Перевірити, які версії вже встановлені:

```bash
pyenv versions
```

Зірочкою позначена та, що активна зараз.

## Глобальна, локальна і активна версія

pyenv розрізняє три рівні налаштування версії, і розуміти різницю між ними важливо.

**Глобальна версія** — та, яка використовується скрізь, якщо більш конкретне налаштування не вказане. Встановлюється командою:

```bash
pyenv global 3.12.3
```

Після цього у будь-якому місці на твоєму комп'ютері команда `python` запустить Python 3.12.3 — якщо тільки інше налаштування не перекриє це.

**Локальна версія** — версія для конкретної директорії. Коли ти заходиш у певну папку, pyenv автоматично перемикається на потрібну версію. Щоб налаштувати це, зайди в директорію проєкту і виконай:

```bash
cd my-project
pyenv local 3.11.9
```

pyenv створить файл `.python-version` у цій директорії з текстом `3.11.9`. Щоразу, коли ти або будь-який інструмент зайде в цю директорію, буде використовуватись саме ця версія. Файл `.python-version` варто додавати до системи контролю версій (git) — тоді всі, хто клонує проєкт, автоматично знатимуть, яка версія Python потрібна.

**Активна версія** — тимчасова, тільки для поточного сеансу термінала. Встановлюється командою `pyenv shell 3.10.14` і скидається при закритті термінала. Корисна, коли треба швидко перевірити щось у конкретній версії, не змінюючи ніяких файлів.

Пріоритет у pyenv такий: активна (shell) перекриває локальну, яка перекриває глобальну. Якщо жодна не вказана — pyenv не знайде версію і поверне помилку, тому глобальну версію варто встановити одразу після встановлення pyenv.

## Що таке системний Python і чому його не варто чіпати

На macOS і більшості Linux-дистрибутивів вже є Python — він встановлений разом з операційною системою. На macOS це `python3`, який ти можеш знайти через команду `which python3`. На Ubuntu — аналогічно.

Цей Python існує не для тебе — він існує для системи. Операційна система і деякі системні утиліти написані з його використанням і розраховують, що він стабільний, незмінний і завжди доступний. Якщо ти почнеш встановлювати у нього власні бібліотеки, оновлювати його або, що найгірше, видалити його — можеш зламати щось в операційній системі.

Звідси просте правило: системний Python — недоторканний. Для власної розробки завжди використовуй версію, встановлену через pyenv. Вона повністю ізольована від системної, живе у власній директорії і ніяк не впливає на систему.

pyenv при правильному налаштуванні перехоплює виклики `python` і `python3` і спрямовує їх на потрібну версію. Системний Python при цьому залишається на місці, просто ти ніколи не будеш до нього звертатись безпосередньо.

## Перевірка: чи все правильно налаштовано

Після встановлення pyenv і Python корисно пройтись по кількох перевіркових командах, щоб переконатися, що все в порядку.

```bash
pyenv versions
```
Повинні бачити встановлені версії, зірочкою позначена активна.

```bash
python --version
```
Повинен з'явитись номер версії, яку ти встановив як глобальну. Якщо з'являється щось інше або команда не знайдена — скоріш за все, рядки конфігурації не додані або термінал не перезапущений.

```bash
which python
```
Повинна з'явитись шлях виду `/home/yourname/.pyenv/shims/python` — це підтверджує, що pyenv перехопив команду і використовує власну версію, а не системну.

```bash
python -c "import sys; print(sys.version)"
```
Розгорнута інформація про версію — хороший спосіб переконатись у деталях.

## Оновлення Python і pyenv

Python оновлюється регулярно. Нові мінорні версії — 3.12.1, 3.12.2, 3.12.3 — виходять кілька разів на рік і зазвичай містять виправлення помилок і покращення безпеки. Нові мажорні версії — 3.13, 3.14 — виходять раз на рік, восени, і можуть містити нові можливості мови.

Щоб встановити нову версію Python через pyenv, спочатку оновлюємо сам pyenv (щоб він знав про нові версії):

```bash
pyenv update
```

Потім встановлюємо потрібну версію стандартною командою. Стару версію можна залишити — вона нікуди не зникне і буде доступна для проєктів, які на неї покладаються. Видалити версію, яка більше не потрібна:

```bash
pyenv uninstall 3.10.14
```

## Маленький відступ: навіщо взагалі знати як це працює

Можна було б написати цей параграф простіше: «виконай ці три команди і рухайся далі». Деякі курси так і роблять. І для перших кількох тижнів це, мабуть, спрацювало б — ти мав би працюючий Python і міг би писати код.

Але рано чи пізно відбувається ось що: щось йде не так. Python не знаходиться. Версія не та. Бібліотека встановилася, але код її не бачить. Або все налаштоване правильно, але IDE використовує не той Python, що термінал. Якщо ти розумієш, як влаштований pyenv — де він живе, що таке shims, чому існує `.python-version` файл, яка різниця між глобальною і локальною версією — ти розберешся з проблемою за п'ять хвилин. Якщо ти просто скопіював команди, не розуміючи що вони роблять — доведеться гугли-ти навмання і гадати, що пішло не так.

Це загальний принцип, не специфічний для pyenv: інструменти, які ти розумієш, слугують тобі. Інструменти, яких ти не розумієш, час від часу підводять тебе у найневідповідніший момент.

## Корисні команди pyenv — коротко

Для зручності зберемо основні команди в одному місці, щоб можна було повернутись, коли знадобиться.

Переглянути всі доступні для встановлення версії Python:

```bash
pyenv install --list
```

Встановити конкретну версію:

```bash
pyenv install 3.12.3
```

Переглянути встановлені версії:

```bash
pyenv versions
```

Встановити глобальну версію за замовчуванням:

```bash
pyenv global 3.12.3
```

Встановити версію для конкретної директорії:

```bash
pyenv local 3.11.9
```

Встановити версію тільки для поточного сеансу термінала:

```bash
pyenv shell 3.10.14
```

Видалити версію:

```bash
pyenv uninstall 3.11.9
```

Оновити pyenv і список доступних версій:

```bash
pyenv update
```

Переглянути, яка версія зараз активна і чому:

```bash
pyenv version
```

## Що далі

Після того як Python встановлений через pyenv, наступний кроком буде налаштування віртуальних середовищ. Це ще один рівень ізоляції — але вже не для версій Python, а для бібліотек: кожен проєкт матиме власний ізольований набір пакетів і не буде конфліктувати з іншими. Про це — в наступному параграфі.

Поки що запам'ятай головне: pyenv розв'язує проблему кількох версій Python на одній машині. Він не є частиною самого Python — це зовнішній інструмент, але настільки зручний і поширений, що в реальній розробці він давно став стандартом де-факто. І тепер ти знаєш не лише як його налаштувати, а й чому саме так воно влаштоване.
