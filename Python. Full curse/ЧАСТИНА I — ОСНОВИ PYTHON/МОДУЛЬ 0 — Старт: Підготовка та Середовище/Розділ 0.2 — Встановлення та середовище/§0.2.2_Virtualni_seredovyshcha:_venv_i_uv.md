# § 0.2.2 Віртуальні середовища: venv і uv

---

## Найважливіша звичка Python-розробника

Якщо з усього цього параграфа ти запам'ятаєш лише одне речення — нехай це буде таке: **кожен проєкт повинен мати власне ізольоване середовище**.

Не «бажано», не «якщо не ліньки», не «коли проєкт стає великим». Кожен. Завжди. З першого дня. Навіть якщо проєкт — це три рядки коду, які ти пишеш заради експерименту.

Це звучить як надмірна педантичність. Насправді це захист від цілого класу проблем, які в іншому випадку наздоженуть тебе в найнесподіваніший момент. Щоб зрозуміти чому — потрібно спочатку зрозуміти, яку саме проблему вирішує віртуальне середовище.

---

## Проблема, яку не видно, поки не станеться

Python — це не тільки сама мова. Разом з мовою існує величезна екосистема бібліотек: готових модулів, які можна встановити і використати у своїй програмі. Бібліотека для роботи з таблицями, бібліотека для HTTP-запитів, бібліотека для побудови графіків — замість того, щоб писати все з нуля, ти встановлюєш готовий компонент і користуєшся ним.

Встановлення бібліотек відбувається через менеджер пакетів `pip`. Ти пишеш `pip install requests` — і бібліотека `requests` з'являється в системі, готова до використання. Зручно. Поки не виникає ситуація, коли на одній машині живуть два проєкти, яким потрібні різні версії однієї і тієї самої бібліотеки.

Уяви: є старий проєкт, який використовує бібліотеку `Django` версії 3.2 — так склалося, що він написаний давно і оновити його поки немає ресурсів. І є новий проєкт, для якого ти хочеш взяти Django 5.0, де з'явились зручні нові функції. Якщо обидва проєкти користуються однією глобальною інсталяцією бібліотек — вони не можуть мати різні версії Django одночасно. Встановиш нову — зламаєш старий проєкт. Залишиш стару — не зможеш використати нові можливості.

Або інша ситуація: ти встановлюєш бібліотеку для одного маленького скрипта, вона тягне за собою кілька залежностей, ті залежності конфліктують з тим, що вже встановлено — і якась інша програма, яка досі працювала бездоганно, раптом починає поводитись дивно. Це реальний сценарій, і він стається не лише з новачками.

Коренева причина в тому, що без ізоляції всі бібліотеки всіх проєктів живуть в одному глобальному просторі. Вони не знають одна про одну, не домовляються, і будь-який конфлікт версій стає проблемою всієї системи.

## Що таке віртуальне середовище

Віртуальне середовище — це ізольована копія Python з власним простором для бібліотек. Коли ти його активуєш, команди `python` і `pip` вказують не на глобальну установку, а на цю конкретну ізольовану копію. Все, що ти встановлюєш через `pip`, потрапляє тільки сюди — і жодним чином не стосується інших проєктів або системного Python.

Фізично це просто директорія — зазвичай вона називається `venv` або `.venv` і лежить у корені твого проєкту. Всередині — невелика ієрархія папок: там є інтерпретатор Python, там є `pip`, там є місце куди йдуть встановлені пакети. Нічого магічного — просто структура директорій плюс кілька скриптів, які перемикають змінні середовища при активації.

Видалити віртуальне середовище — означає просто видалити цю директорію. Ніяких слідів у системі, ніяких реєстрів, ніяких залишків. Це теж перевага: якщо щось пішло не так або хочеш почати з чистого аркуша — видалив папку і створив заново.

## Створення і активація: venv

Модуль `venv` входить до стандартної бібліотеки Python — тобто нічого додатково встановлювати не потрібно, він є завжди.

Перейди в директорію проєкту і виконай:

```bash
python -m venv venv
```

Тут `python -m venv` — це запуск модуля `venv`, а останнє `venv` — назва директорії, яку він створить. Назва може бути будь-якою, але `venv` і `.venv` — найпоширеніші конвенції. Деякі розробники надають перевагу `.venv` з крапкою на початку: на Unix-системах крапка в назві робить директорію прихованою, що трохи прибирає візуальний шум у файловому менеджері.

Після створення середовище треба **активувати**. Це важливий крок, який початківці часто пропускають, а потім дивуються чому бібліотеки, які вони «встановили», не знаходяться.

На macOS і Linux:

```bash
source venv/bin/activate
```

На Windows (Command Prompt):

```cmd
venv\Scripts\activate.bat
```

На Windows (PowerShell):

```powershell
venv\Scripts\Activate.ps1
```

> ⚠️ **Увага (Windows, PowerShell):** Якщо при спробі активувати середовище PowerShell повідомляє про заборону виконання скриптів, виконай таку команду від імені адміністратора: `Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser`. Це дозволить запускати локально підписані скрипти, не відкриваючи систему для небезпечних.

Після активації ти помітиш, що рядок запрошення у терміналі змінився — спереду з'явилась назва середовища в дужках:

```
(venv) user@machine:~/my-project$
```

Це візуальний індикатор: ти всередині. Тепер `python` і `pip` вказують на це середовище, а не на глобальне.

Вийти з середовища:

```bash
deactivate
```

Дужки зникнуть, і все повернеться до попереднього стану.

## pip: встановлення, оновлення, видалення

`pip` — це менеджер пакетів Python. З активованим середовищем він встановлює бібліотеки саме туди, куди потрібно.

Встановити бібліотеку:

```bash
pip install requests
```

Встановити конкретну версію:

```bash
pip install requests==2.31.0
```

Встановити версію не нижче певної:

```bash
pip install "requests>=2.28"
```

Оновити бібліотеку до останньої версії:

```bash
pip install --upgrade requests
```

Видалити:

```bash
pip uninstall requests
```

Переглянути всі встановлені бібліотеки у поточному середовищі:

```bash
pip list
```

Знайти конкретну бібліотеку і перевірити її версію:

```bash
pip show requests
```

## requirements.txt: фіксація залежностей

Якщо ти пишеш проєкт самостійно, а потім хочеш передати його колезі або задеплоїти на сервер — потрібен спосіб передати інформацію про те, які бібліотеки і якої версії потрібні для запуску. Саме для цього існує файл `requirements.txt`.

Згенерувати його зі списку всього, що встановлено у поточному середовищі:

```bash
pip freeze > requirements.txt
```

Вміст файлу буде виглядати приблизно так:

```
certifi==2024.2.2
charset-normalizer==3.3.2
idna==3.6
requests==2.31.0
urllib3==2.2.1
```

Кожен рядок — назва бібліотеки і точна версія. Коли хтось інший клонує твій репозиторій і хоче запустити проєкт, він створює власне середовище і одною командою встановлює все необхідне:

```bash
pip install -r requirements.txt
```

`-r` означає «читай залежності з файлу».

Є важлива деталь, про яку варто знати одразу: `pip freeze` включає абсолютно все — не тільки бібліотеки, які ти встановлював свідомо, а й усі їхні залежності, тобто ті бібліотеки, які вони самі потягли за собою. Часто це корисно — забезпечує повну відтворюваність. Але інколи потрібен файл тільки з «прямими» залежностями, щоб він залишався читабельним і підтримуваним. Для цього існують більш просунуті інструменти на кшталт `pip-tools` або того ж `uv`, про який мова піде далі. На початку навчання звичайного `pip freeze` цілком достатньо.

> 💡 **Порада:** Файл `requirements.txt` додається до git-репозиторію разом з кодом. Сама директорія `venv/` — ні: вона важка, відтворювана з `requirements.txt` і містить бінарні файли, специфічні для операційної системи. У `.gitignore` завжди має бути рядок `venv/` або `.venv/`.

## .gitignore: що не потрібно в репозиторії

Якщо ти ще не знайомий з git — не хвилюйся, про нього буде окремий параграф. Але один практичний момент варто засвоїти вже зараз, бо він стосується безпосередньо роботи з середовищами.

Деякі файли і директорії у проєкті не повинні потрапляти до системи контролю версій. Для Python-проєктів мінімальний `.gitignore` виглядає так:

```
venv/
.venv/
__pycache__/
*.pyc
*.pyo
.env
.DS_Store
```

Директорія `venv/` — тому що вона велика, специфічна для системи і повністю відтворюється з `requirements.txt`. `__pycache__/` і файли `.pyc` — це скомпільований байт-код Python, який генерується автоматично і не потребує зберігання. `.env` — файл із секретними змінними середовища: паролями, ключами API, рядками підключення до бази даних. Цей файл ніколи, за жодних обставин, не повинен потрапляти до репозиторію — особливо публічного. Про це теж буде окрема розмова.

## Сучасна альтернатива: uv

Стандартний `venv` + `pip` — надійний і перевірений підхід, який працює скрізь де є Python. Але в останні роки з'явився інструмент, який робить ту саму роботу значно швидше і зручніше. Він називається `uv` і розроблений компанією Astral — тією самою, що зробила лінтер Ruff.

Головна відмінність, яку відчуваєш одразу: швидкість. `uv` написаний мовою Rust і встановлює пакети у десятки разів швидше за звичайний `pip`. Для великих проєктів з багатьма залежностями різниця може бути між кількома хвилинами і кількома секундами.

Але крім швидкості є ще кілька приємних речей. `uv` може сам завантажити і встановити потрібну версію Python без pyenv — в простих сценаріях він повністю бере на себе роботу з версіями. Він має зручніший інтерфейс для управління залежностями з файлом блокування (`uv.lock`), який фіксує точні версії і забезпечує відтворюваність. І він сумісний з існуючими `requirements.txt` — тобто можна використовувати його в наявних проєктах без переналаштування.

Встановлення `uv`:

```bash
# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh

# або через pip (якщо вже є Python)
pip install uv
```

На Windows:

```powershell
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
```

Основні команди `uv`, які замінюють стандартний набір:

Створити нове середовище (аналог `python -m venv venv`):

```bash
uv venv
```

За замовчуванням `uv` створить директорію `.venv`. Активація — тим самим способом, що і для звичайного venv:

```bash
source .venv/bin/activate      # Linux / macOS
.venv\Scripts\activate         # Windows
```

Встановити пакет (аналог `pip install`):

```bash
uv pip install requests
```

Встановити з файлу (аналог `pip install -r requirements.txt`):

```bash
uv pip install -r requirements.txt
```

Зафіксувати залежності (аналог `pip freeze`):

```bash
uv pip freeze > requirements.txt
```

Як бачиш, інтерфейс майже ідентичний `pip` — тільки додається `uv` на початку. Якщо ти вже знаєш `pip`, перехід займає кілька хвилин.

> 💡 **Порада:** Для цього курсу можна використовувати і `venv` + `pip`, і `uv` — обидва підходи рівноцінні з точки зору результату. `venv` + `pip` є у стандартній бібліотеці, завжди під рукою і не вимагає нічого додаткового. `uv` — швидший і зручніший, особливо відчутно на великих проєктах. Рекомендація: почни з `venv` + `pip`, щоб розуміти як це влаштовано, а потім за бажанням перейди на `uv`.

## Середовище і IDE: важливий нюанс

Є одна деталь, яка часто збиває з пантелику початківців, і про яку варто сказати окремо.

Коли ти активуєш середовище в терміналі — це не означає, що твій редактор коду або IDE автоматично теж його «бачить». VS Code, PyCharm та інші середовища мають власне налаштування інтерпретатора Python, яке не залежить від того, що активовано в терміналі.

У VS Code це налаштовується через вибір інтерпретатора: відкрий командну палітру (`Ctrl+Shift+P` або `Cmd+Shift+P`), введи `Python: Select Interpreter` і обери Python із твого `venv` або `.venv`. Шлях буде виглядати приблизно як `./venv/bin/python` або `./.venv/bin/python`. Після цього VS Code буде використовувати саме цей інтерпретатор для підказок, перевірки типів, запуску коду і налагодження.

Якщо цього не зробити — редактор може підкреслювати імпорти як «невідомі», навіть якщо бібліотека встановлена у середовищі. Він просто не знає де шукати.

Хороша новина: VS Code досить розумний, щоб автоматично запропонувати вибрати виявлене середовище, коли ти відкриваєш проєкт де вже є папка `venv` або `.venv`. Зазвичай достатньо просто натиснути «Yes» на його пропозицію.

## Типовий робочий процес: від початку до кінця

Щоб усе вище склалось у цілісну картину — ось як виглядає стандартна послідовність дій при початку нового проєкту.

Спочатку створити директорію проєкту і зайти в неї:

```bash
mkdir my-project
cd my-project
```

Ініціалізувати git-репозиторій (докладніше — у параграфі про Git):

```bash
git init
```

Створити `.gitignore` з необхідними виключеннями:

```bash
echo "venv/\n.venv/\n__pycache__/\n*.pyc\n.env" > .gitignore
```

Якщо використовуєш pyenv — зафіксувати версію Python для цього проєкту:

```bash
pyenv local 3.12.3
```

Створити віртуальне середовище:

```bash
python -m venv venv
```

Або через uv:

```bash
uv venv
```

Активувати:

```bash
source venv/bin/activate     # Linux / macOS
venv\Scripts\activate        # Windows
```

Переконатись що середовище активне (у рядку запрошення з'явились дужки) і встановити перші потрібні бібліотеки:

```bash
pip install requests
```

Зафіксувати залежності:

```bash
pip freeze > requirements.txt
```

Ця послідовність займає менше хвилини і з часом стає настільки автоматичною, що ти виконуєш її не замислюючись — як умиваєш руки перед їжею.

## Що відбувається «під капотом» при активації

Розуміти деталі не обов'язково для роботи, але корисно для відчуття впевненості: ти знаєш що відбувається, а не просто сліпо довіряєшся команді.

Коли ти виконуєш `source venv/bin/activate` — насправді запускається невеликий shell-скрипт, який робить три речі. Перше: зберігає поточне значення змінної `PATH` у тимчасову змінну, щоб можна було відновити його при `deactivate`. Друге: додає шлях до директорії `venv/bin/` на початок `PATH`. Це змушує систему знаходити саме тут команди `python` і `pip` — раніше ніж шукати їх у глобальних директоріях. Третє: змінює рядок запрошення термінала, додаючи назву середовища в дужках — суто для твоєї зручності.

Коли виконуєш `deactivate` — скрипт відновлює `PATH` з тимчасової змінної. Все повертається як було.

Це означає, що якщо ти відкриєш новий термінал або нову вкладку — середовище там не буде активоване, навіть якщо воно активоване в іншому. Активація не є глобальною — вона живе тільки в поточному сеансі терміналу. Це і перевага (ізольованість), і момент на який варто звертати увагу.

## Підсумок

Ізоляція бібліотек через віртуальне середовище — не бюрократичний ритуал, а реальний захист від реальних проблем. Кожен проєкт живе у власному просторі, зі своїм набором пакетів своїх версій. Жодних конфліктів, жодних несподіваних поломок через сторонній проєкт.

`venv` — стандарт, вбудований у Python, надійний і простий. `uv` — сучасний, швидший, з додатковими зручностями. Обидва вирішують одну задачу і добре сумісні між собою.

`requirements.txt` — спосіб зафіксувати залежності і передати їх іншим. Він завжди в репозиторії, `venv/` — ніколи.

І ще раз головне, бо воно того варте: **кожен проєкт — окреме середовище**. Завжди.
