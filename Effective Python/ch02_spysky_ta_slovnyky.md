# Розділ 2. Списки та словники

Багато програм пишеться для автоматизації повторюваних завдань, які краще вирішуються машинами, ніж людьми. У Python найпоширенішим способом організації такої роботи є використання послідовності значень, збережених у типі `list`. Списки надзвичайно універсальні й можуть застосовуватися для вирішення різноманітних задач.

Природним доповненням до списків є тип `dict`, що зберігає ключі для пошуку, зіставлені з відповідними значеннями (що часто називається асоціативним масивом або хеш-таблицею). Словники забезпечують константний час виконання (амортизований) для операцій присвоєння та доступу, що робить їх ідеальними для обліку динамічної інформації.

Python має спеціальний синтаксис та вбудовані модулі, що підвищують читабельність і розширюють можливості списків та словників за межі того, чого можна очікувати від простих типів масивів, векторів і хеш-таблиць в інших мовах.

## Порада 11. Знайте, як нарізати послідовності

Python містить синтаксис для нарізання послідовностей на частини. Зрізи дозволяють отримати доступ до підмножини елементів послідовності з мінімальними зусиллями. Найпростіше зрізи застосовуються до вбудованих типів `list`, `str` та `bytes`. Зрізи можна розширити до будь-якого Python-класу, що реалізує спеціальні методи `__getitem__` та `__setitem__` (см. Порада 43: «Успадковуйте від collections.abc для власних типів-контейнерів»).

Базова форма синтаксису зрізів — `somelist[start:end]`, де `start` включається, а `end` — ні:

```python
a = ['a', 'b', 'c', 'd', 'e', 'f', 'g', 'h']
print('Middle two: ', a[3:5])
print('All but ends:', a[1:7])
>>>
Middle two:  ['d', 'e']
All but ends: ['b', 'c', 'd', 'e', 'f', 'g']
```

При нарізанні від початку списку слід опускати нульовий індекс, щоб зменшити візуальний шум:

```python
assert a[:5] == a[0:5]
```

При нарізанні до кінця списку слід опускати кінцевий індекс, оскільки він надлишковий:

```python
assert a[5:] == a[5:len(a)]
```

Використання від'ємних чисел для зрізів зручне для зміщень відносно кінця списку. Всі ці форми зрізів будуть зрозумілі новому читачеві вашого коду:

```python
a[:]    # ['a', 'b', 'c', 'd', 'e', 'f', 'g', 'h']
a[:5]   # ['a', 'b', 'c', 'd', 'e']
a[:-1]  # ['a', 'b', 'c', 'd', 'e', 'f', 'g']
a[4:]   #           ['e', 'f', 'g', 'h']
a[-3:]  #                ['f', 'g', 'h']
a[2:5]  #           ['c', 'd', 'e']
a[2:-1] #           ['c', 'd', 'e', 'f', 'g']
a[-3:-1] #               ['f', 'g']
```

Тут немає жодних несподіванок, і я заохочую використовувати ці варіанти.

Зрізи коректно обробляють початкові та кінцеві індекси, що виходять за межі списку, беззвучно пропускаючи відсутні елементи. Ця поведінка полегшує встановлення максимальної довжини для розгляду вхідної послідовності:

```python
first_twenty_items = a[:20]
last_twenty_items = a[-20:]
```

Натомість безпосереднє звернення за тим самим індексом спричинює виняток:

```python
a[20]
>>>
Traceback ...
IndexError: list index out of range
```

> **Примітка.** Слід бути уважними: індексування списку за заперечною змінною є одним із небагатьох випадків, коли зрізи можуть давати несподівані результати. Наприклад, вираз `somelist[-n:]` буде коректно працювати, коли `n` більше одиниці (наприклад, `somelist[-3:]`). Однак коли `n` дорівнює нулю, вираз `somelist[-0:]` еквівалентний `somelist[:]` і повертає копію оригінального списку.

Результат нарізання списку — це цілком новий список. Посилання на об'єкти з оригінального списку зберігаються. Зміна результату нарізання не впливає на оригінальний список:

```python
b = a[3:]
print('Before: ', b)
b[1] = 99
print('After:  ', b)
print('No change:', a)
>>>
Before:  ['d', 'e', 'f', 'g', 'h']
After:   ['d', 99, 'f', 'g', 'h']
No change: ['a', 'b', 'c', 'd', 'e', 'f', 'g', 'h']
```

При використанні у присвоєннях зрізи замінюють вказаний діапазон в оригінальному списку. На відміну від присвоєнь із розпакуванням (наприклад, `a, b = c[:2]`; см. Порада 6: «Надавайте перевагу розпакуванню з множинним присвоєнням над індексуванням»), довжини присвоєнь зрізів не повинні збігатися. Значення до і після призначеного зрізу зберігатимуться. Тут список скорочується, оскільки замінний список коротший за вказаний зріз:

```python
print('Before ', a)
a[2:7] = [99, 22, 14]
print('After  ', a)
>>>
Before  ['a', 'b', 'c', 'd', 'e', 'f', 'g', 'h']
After   ['a', 'b', 99, 22, 14, 'h']
```

А тут список розширюється, оскільки призначений список довший за вказаний зріз:

```python
print('Before ', a)
a[2:3] = [47, 11]
print('After  ', a)
>>>
Before  ['a', 'b', 99, 22, 14, 'h']
After   ['a', 'b', 47, 11, 22, 14, 'h']
```

Якщо опустити обидва індекси — початковий і кінцевий — при нарізанні, отримаємо копію оригінального списку:

```python
b = a[:]
assert b == a and b is not a
```

Якщо виконати присвоєння зрізу без початкового та кінцевого індексів, весь вміст списку замінюється копією посилань (без виділення нового списку):

```python
b = a
print('Before a', a)
print('Before b', b)
a[:] = [101, 102, 103]
assert a is b             # Все ще той самий об'єкт списку
print('After a ', a)      # Тепер містить інший вміст
print('After b ', b)      # Той самий список — той самий вміст, що й a
>>>
Before a ['a', 'b', 47, 11, 22, 14, 'h']
Before b ['a', 'b', 47, 11, 22, 14, 'h']
After a  [101, 102, 103]
After b  [101, 102, 103]
```

**Що слід запам'ятати**

- Уникайте зайвої багатослівності при нарізанні: не вказуйте `0` як початковий індекс і довжину послідовності як кінцевий.
- Зрізи є поблажливими до початкових або кінцевих індексів, що виходять за межі, що дозволяє легко виражати зрізи на передніх або задніх межах послідовності (наприклад, `a[:20]` або `a[-20:]`).
- Присвоєння зрізу списку замінює цей діапазон в оригінальній послідовності на те, на що посилається права частина, навіть якщо довжини відрізняються.

## Порада 12. Уникайте кроку та зрізу в одному виразі

На додаток до базового нарізання (см. Порада 11: «Знайте, як нарізати послідовності»), Python має спеціальний синтаксис для кроку зрізу у формі `somelist[start:end:stride]`. Це дозволяє брати кожен n-й елемент при нарізанні послідовності. Наприклад, крок спрощує групування за парними та непарними індексами у списку:

```python
x = ['red', 'orange', 'yellow', 'green', 'blue', 'purple']
odds = x[::2]
evens = x[1::2]
print(odds)
print(evens)
>>>
['red', 'yellow', 'blue']
['orange', 'green', 'purple']
```

Проблема в тому, що синтаксис кроку часто призводить до несподіваної поведінки, яка може вносити помилки. Наприклад, поширений Python-прийом для обернення байтового рядка — нарізання рядка з кроком `-1`:

```python
x = b'mongoose'
y = x[::-1]
print(y)
>>>
b'esoognom'
```

Це також коректно працює для рядків Unicode (см. Порада 3: «Знайте відмінності між bytes і str»):

```python
x = 'ᇓৌ'
y = x[::-1]
print(y)
>>>
ৌᇓ
```

Але це зламається, якщо дані Unicode закодовані як байтовий рядок UTF-8:

```python
w = 'ᇓৌ'
x = w.encode('utf-8')
y = x[::-1]
z = y.decode('utf-8')
>>>
Traceback ...
UnicodeDecodeError: 'utf-8' codec can't decode byte 0xb8 in
position 0: invalid start byte
```

Чи корисні від'ємні кроки, крім `-1`? Розглянемо такі приклади:

```python
x = ['a', 'b', 'c', 'd', 'e', 'f', 'g', 'h']
x[::2]   # ['a', 'c', 'e', 'g']
x[::-2]  # ['h', 'f', 'd', 'b']
```

Тут `::2` означає «вибрати кожен другий елемент, починаючи з початку». Складніше: `::-2` означає «вибрати кожен другий елемент, починаючи з кінця й рухаючись назад».

Що, на вашу думку, означає `2::2`? А `-2::-2` порівняно з `-2:2:-2` чи `2:2:-2`?

```python
x[2::2]    # ['c', 'e', 'g']
x[-2::-2]  # ['g', 'e', 'c', 'a']
x[-2:2:-2] # ['g', 'e']
x[2:2:-2]  # []
```

Суть у тому, що частина синтаксису зрізів, що відповідає за крок, може бути надзвичайно заплутаною. Три числа у квадратних дужках — вже достатньо важко читати через їхню щільність. До того ж не очевидно, коли починають діяти початковий та кінцевий індекси відносно значення кроку, особливо коли крок від'ємний.

Щоб уникнути проблем, рекомендується не використовувати крок разом із початковим і кінцевим індексами. Якщо крок необхідний, надавайте перевагу додатньому значенню та опускайте початковий і кінцевий індекси. Якщо крок потрібно використовувати разом із початковим або кінцевим індексами, розгляньте використання одного присвоєння для кроку та іншого для зрізу:

```python
y = x[::2]   # ['a', 'c', 'e', 'g']
z = y[1:-1]  # ['c', 'e']
```

Крок із подальшим зрізом створює додаткову поверхневу копію даних. Перша операція повинна намагатися зменшити розмір результуючого зрізу якомога більше. Якщо ваша програма не може дозволити собі час або пам'ять, необхідні для двох кроків, розгляньте використання методу `islice` з вбудованого модуля `itertools` (см. Порада 36: «Розгляньте itertools для роботи з ітераторами та генераторами»), який простіший для читання та не допускає від'ємних значень для початку, кінця або кроку.

**Що слід запам'ятати**

- Вказівка `start`, `end` та `stride` у зрізі може бути надзвичайно заплутаною.
- Надавайте перевагу додатнім значенням кроку в зрізах без початкового або кінцевого індексів. Уникайте від'ємних значень кроку там, де це можливо.
- Уникайте використання `start`, `end` та `stride` разом в одному зрізі. Якщо потрібні всі три параметри, розгляньте два окремих присвоєння (одне для кроку, інше для зрізу) або використання `islice` з вбудованого модуля `itertools`.

## Порада 13. Надавайте перевагу узагальнюючому розпакуванню над зрізами

Одним з обмежень базового розпакування (см. Порада 6: «Надавайте перевагу розпакуванню з множинним присвоєнням над індексуванням») є те, що потрібно заздалегідь знати довжину послідовностей, які розпаковуються. Наприклад, є список вікових показників автомобілів, що здаються на комісію в автосалон. При спробі взяти перші два елементи списку за допомогою базового розпакування під час виконання виникає виняток:

```python
car_ages = [0, 9, 4, 8, 7, 20, 19, 1, 6, 15]
car_ages_descending = sorted(car_ages, reverse=True)
oldest, second_oldest = car_ages_descending
>>>
Traceback ...
ValueError: too many values to unpack (expected 2)
```

Початківці Python часто вдаються до індексування та зрізів (см. Порада 11: «Знайте, як нарізати послідовності») у такій ситуації. Наприклад, тут витягуємо найстаріший, другий за старістю та інші вікові показники зі списку щонайменше з двома елементами:

```python
oldest = car_ages_descending[0]
second_oldest = car_ages_descending[1]
others = car_ages_descending[2:]
print(oldest, second_oldest, others)
>>>
20 19 [15, 9, 8, 7, 6, 4, 1, 0]
```

Це працює, але все це індексування і нарізання є візуально шумним. На практиці такий спосіб розподілу елементів послідовності на підмножини також є схильним до помилок, оскільки значно підвищується ймовірність помилок на одиницю; наприклад, можна змінити межі в одному рядку і забути оновити інші.

Для кращого вирішення цієї ситуації Python також підтримує узагальнююче розпакування через зірочковий вираз. Цей синтаксис дозволяє одній частині присвоєння з розпакуванням отримати всі значення, що не відповідають жодній іншій частині шаблону розпакування. Тут зірочковий вираз використовується для досягнення того самого результату без індексування або нарізання:

```python
oldest, second_oldest, *others = car_ages_descending
print(oldest, second_oldest, others)
>>>
20 19 [15, 9, 8, 7, 6, 4, 1, 0]
```

Цей код коротший, легший для читання і більше не має схильної до помилок крихкості граничних індексів, які потрібно тримати синхронізованими між рядками.

Зірочковий вираз може з'являтися в будь-якій позиції, тому можна отримати переваги узагальнюючого розпакування щоразу, коли потрібно витягти один зріз:

```python
oldest, *others, youngest = car_ages_descending
print(oldest, youngest, others)
*others, second_youngest, youngest = car_ages_descending
print(youngest, second_youngest, others)
>>>
20 0 [19, 15, 9, 8, 7, 6, 4, 1]
0 1 [20, 19, 15, 9, 8, 7, 6, 4]
```

Однак для розпакування присвоєнь, що містять зірочковий вираз, потрібна хоча б одна обов'язкова частина, інакше виникне `SyntaxError`. Не можна використовувати узагальнюючий вираз самостійно:

```python
*others = car_ages_descending
>>>
Traceback ...
SyntaxError: starred assignment target must be in a list or tuple
```

Також не можна використовувати кілька узагальнюючих виразів в одному рівні шаблону розпакування:

```python
first, *middle, *second_middle, last = [1, 2, 3, 4]
>>>
Traceback ...
SyntaxError: two starred expressions in assignment
```

Але можна використовувати кілька зірочкових виразів в одному операторі присвоєння з розпакуванням, якщо вони є узагальнювачами для різних частин багаторівневої структури, що розпаковується. Не рекомендується виконувати наведене нижче (см. Порада 19: «Ніколи не розпаковуйте більше трьох змінних при поверненні кількох значень з функцій»), але розуміння цього допоможе сформувати інтуїцію щодо того, як зірочкові вирази можуть використовуватися в присвоєннях із розпакуванням:

```python
car_inventory = {
    'Downtown': ('Silver Shadow', 'Pinto', 'DMC'),
    'Airport': ('Skyline', 'Viper', 'Gremlin', 'Nova'),
}
((loc1, (best1, *rest1)),
 (loc2, (best2, *rest2))) = car_inventory.items()
print(f'Best at {loc1} is {best1}, {len(rest1)} others')
print(f'Best at {loc2} is {best2}, {len(rest2)} others')
>>>
Best at Downtown is Silver Shadow, 2 others
Best at Airport is Skyline, 3 others
```

Зірочкові вирази у всіх випадках перетворюються на екземпляри `list`. Якщо в послідовності, що розпаковується, не залишилося елементів, узагальнювальна частина буде порожнім списком. Це особливо корисно при обробці послідовності, про яку заздалегідь відомо, що вона містить щонайменше N елементів:

```python
short_list = [1, 2]
first, second, *rest = short_list
print(first, second, rest)
>>>
1 2 []
```

Можна також розпаковувати довільні ітератори за допомогою синтаксису розпакування. Для базового оператора множинного присвоєння це не має великого значення. Наприклад, тут розпаковуємо значення від ітерування по діапазону довжиною 2. Це не здається корисним, оскільки простіше просто присвоїти статичний список, що відповідає шаблону розпакування (наприклад, `[1, 2]`):

```python
it = iter(range(1, 3))
first, second = it
print(f'{first} and {second}')
>>>
1 and 2
```

Але з додаванням зірочкових виразів цінність розпакування ітераторів стає очевидною. Наприклад, є генератор, що видає рядки CSV-файлу з усіма замовленнями автомобілів від дилерства за цей тиждень:

```python
def generate_csv():
    yield ('Date', 'Make', 'Model', 'Year', 'Price')
    ...
```

Обробка результатів цього генератора за допомогою індексів і зрізів можлива, але вимагає кількох рядків і є візуально шумною:

```python
all_csv_rows = list(generate_csv())
header = all_csv_rows[0]
rows = all_csv_rows[1:]
print('CSV Header:', header)
print('Row count: ', len(rows))
>>>
CSV Header: ('Date', 'Make', 'Model', 'Year', 'Price')
Row count:  200
```

Розпакування зі зірочковим виразом спрощує обробку першого рядка — заголовка — окремо від решти вмісту ітератора. Це значно зрозуміліше:

```python
it = generate_csv()
header, *rows = it
print('CSV Header:', header)
print('Row count: ', len(rows))
>>>
CSV Header: ('Date', 'Make', 'Model', 'Year', 'Price')
Row count:  200
```

Однак слід пам'ятати, що оскільки зірочковий вираз завжди перетворюється на список, розпакування ітератора також ризикує вичерпати всю пам'ять комп'ютера та спричинити аварійне завершення програми. Тому узагальнююче розпакування для ітераторів слід використовувати лише тоді, коли є вагомі підстави вважати, що всі результуючі дані вмістяться в пам'яті (см. Порада 31: «Будьте захищеними при ітеруванні по аргументах»).

**Що слід запам'ятати**

- Присвоєння з розпакуванням може використовувати зірочковий вираз для захоплення всіх значень, що не були присвоєні іншим частинам шаблону розпакування, у список.
- Зірочкові вирази можуть з'являтися в будь-якій позиції і завжди перетворюватимуться на список, що містить нуль або більше отриманих значень.
- При розподілі списку на непересічні частини узагальнююче розпакування є значно менш схильним до помилок, ніж нарізання та індексування.

## Порада 14. Сортуйте за складними критеріями за допомогою параметра key

Вбудований тип `list` надає метод `sort` для впорядкування елементів екземпляра списку за різними критеріями. За замовчуванням `sort` впорядковує вміст списку в природному зростаючому порядку елементів. Наприклад, тут сортується список цілих чисел від найменшого до найбільшого:

```python
numbers = [93, 86, 11, 68, 70]
numbers.sort()
print(numbers)
>>>
[11, 68, 70, 86, 93]
```

Метод `sort` працює для майже всіх вбудованих типів (рядки, числа з рухомою комою тощо), що мають природний порядок. Що робить `sort` з об'єктами? Наприклад, тут визначається клас — включаючи метод `__repr__`, щоб екземпляри можна було виводити; см. Порада 75: «Використовуйте рядки repr для налагодження» — для представлення різних інструментів, які можуть знадобитися на будівельному майданчику:

```python
class Tool:
    def __init__(self, name, weight):
        self.name = name
        self.weight = weight
    def __repr__(self):
        return f'Tool({self.name!r}, {self.weight})'

tools = [
    Tool('level', 3.5),
    Tool('hammer', 1.25),
    Tool('screwdriver', 0.5),
    Tool('chisel', 0.25),
]
```

Сортування об'єктів цього типу не працює, оскільки метод `sort` намагається викликати спеціальні методи порівняння, не визначені в класі:

```python
tools.sort()
>>>
Traceback ...
TypeError: '<' not supported between instances of 'Tool' and 'Tool'
```

Якщо клас повинен мати природний порядок, як цілі числа, можна визначити необхідні спеціальні методи (см. Порада 73: «Знайте, як використовувати heapq для черг з пріоритетом»), щоб `sort` працював без додаткових параметрів. Але частіше об'єкти можуть потребувати підтримки кількох порядків, що робить визначення природного порядку недоцільним.

Часто є атрибут об'єкта, який потрібно використати для сортування. Для підтримки такого сценарію метод `sort` приймає параметр `key`, що є функцією. Функція `key` отримує один аргумент — елемент зі списку, що сортується. Значення, яке повертає функція `key`, має бути порівнюваним (тобто мати природний порядок) і використовується замість елемента для цілей сортування.

Тут ключове слово `lambda` застосовується для визначення функції параметра `key`, що дозволяє сортувати список об'єктів `Tool` за іменем в алфавітному порядку:

```python
print('Unsorted:', repr(tools))
tools.sort(key=lambda x: x.name)
print('\nSorted:  ', tools)
>>>
Unsorted: [Tool('level', 3.5), Tool('hammer', 1.25),
           Tool('screwdriver', 0.5), Tool('chisel', 0.25)]
Sorted:   [Tool('chisel', 0.25), Tool('hammer', 1.25),
           Tool('level', 3.5), Tool('screwdriver', 0.5)]
```

Так само легко можна визначити іншу lambda-функцію для сортування за вагою та передати її як параметр `key` методу `sort`:

```python
tools.sort(key=lambda x: x.weight)
print('By weight:', tools)
>>>
By weight: [Tool('chisel', 0.25), Tool('screwdriver', 0.5),
            Tool('hammer', 1.25), Tool('level', 3.5)]
```

У lambda-функції, переданій як параметр `key`, можна звертатися до атрибутів елементів, як це зроблено тут, індексувати елементи (для послідовностей, кортежів та словників) або використовувати будь-який інший коректний вираз.

Для базових типів, як-от рядки, можна навіть використовувати функцію `key` для виконання перетворень значень перед сортуванням. Наприклад, тут метод `lower` застосовується до кожного елемента списку назв місць, щоб забезпечити їх алфавітний порядок без урахування регістру (оскільки в природному лексикографічному порядку рядків великі літери стоять перед малими):

```python
places = ['home', 'work', 'New York', 'Paris']
places.sort()
print('Case sensitive: ', places)
places.sort(key=lambda x: x.lower())
print('Case insensitive:', places)
>>>
Case sensitive:   ['New York', 'Paris', 'home', 'work']
Case insensitive: ['home', 'New York', 'Paris', 'work']
```

Іноді може знадобитися кілька критеріїв сортування. Наприклад, є список електроінструментів, який потрібно відсортувати спочатку за вагою, а потім за іменем. Як цього досягти?

```python
power_tools = [
    Tool('drill', 4),
    Tool('circular saw', 5),
    Tool('jackhammer', 40),
    Tool('sander', 4),
]
```

Найпростіше рішення в Python — використання типу `tuple`. Кортежі — незмінні послідовності довільних Python-значень. Кортежі за замовчуванням є порівнюваними та мають природний порядок, тобто реалізують усі необхідні спеціальні методи, наприклад `__lt__`, яких вимагає метод `sort`. Кортежі реалізують ці компаратори, ітеруючи по кожній позиції кортежу та порівнюючи відповідні значення по одному індексу за раз. Тут показано, як це працює, коли один інструмент важчий за інший:

```python
saw = (5, 'circular saw')
jackhammer = (40, 'jackhammer')
assert not (jackhammer < saw)  # Відповідає очікуванням
```

Якщо перші позиції порівнюваних кортежів рівні — у цьому випадку вага — порівняння кортежу переходить до другої позиції тощо:

```python
drill = (4, 'drill')
sander = (4, 'sander')
assert drill[0] == sander[0]  # Однакова вага
assert drill[1] < sander[1]   # Алфавітно менше
assert drill < sander          # Отже, drill іде першим
```

Можна скористатися цією поведінкою порівняння кортежів для сортування списку електроінструментів спочатку за вагою, а потім за іменем. Тут визначається функція `key`, що повертає кортеж, який містить два атрибути для сортування в порядку пріоритетності:

```python
power_tools.sort(key=lambda x: (x.weight, x.name))
print(power_tools)
>>>
[Tool('drill', 4), Tool('sander', 4),
 Tool('circular saw', 5), Tool('jackhammer', 40)]
```

Одне обмеження при поверненні кортежу функцією `key` полягає в тому, що напрямок сортування для всіх критеріїв повинен бути однаковим (або всі за зростанням, або всі за спаданням). Якщо передати параметр `reverse` методу `sort`, він однаково вплине на обидва критерії в кортежі (зверніть увагу, що тепер `'sander'` стоїть перед `'drill'`, а не після):

```python
power_tools.sort(key=lambda x: (x.weight, x.name),
                 reverse=True)  # Всі критерії — за спаданням
print(power_tools)
>>>
[Tool('jackhammer', 40), Tool('circular saw', 5),
 Tool('sander', 4), Tool('drill', 4)]
```

Для числових значень можна змішувати напрямки сортування за допомогою унарного оператора мінус у функції `key`. Це заперечує одне зі значень у поверненому кортежі, фактично обертаючи порядок його сортування, залишаючи інші незмінними. Тут цей підхід використовується для сортування за вагою за спаданням, а потім за іменем за зростанням (зверніть увагу, що тепер `'sander'` стоїть після `'drill'`, а не перед):

```python
power_tools.sort(key=lambda x: (-x.weight, x.name))
print(power_tools)
>>>
[Tool('jackhammer', 40), Tool('circular saw', 5),
 Tool('drill', 4), Tool('sander', 4)]
```

На жаль, унарне заперечення можливе не для всіх типів. Тут спроба досягти того самого результату за допомогою аргументу `reverse` для сортування за вагою за спаданням і заперечення імені для зростаючого порядку:

```python
power_tools.sort(key=lambda x: (x.weight, -x.name),
                 reverse=True)
>>>
Traceback ...
TypeError: bad operand type for unary -: 'str'
```

Для подібних ситуацій Python забезпечує стабільний алгоритм сортування. Метод `sort` типу `list` збереже порядок вхідного списку, коли функція `key` повертає значення, що рівні між собою. Це означає, що можна викликати `sort` кілька разів для одного й того самого списку, комбінуючи різні критерії. Тут отримується той самий порядок сортування — вага за спаданням і ім'я за зростанням — але за допомогою двох окремих викликів `sort`:

```python
power_tools.sort(key=lambda x: x.name)           # Ім'я за зростанням
power_tools.sort(key=lambda x: x.weight,         # Вага за спаданням
                 reverse=True)
print(power_tools)
>>>
[Tool('jackhammer', 40), Tool('circular saw', 5),
 Tool('drill', 4), Tool('sander', 4)]
```

Щоб зрозуміти, чому це працює, зверніть увагу, як перший виклик `sort` розміщує імена в алфавітному порядку:

```python
power_tools.sort(key=lambda x: x.name)
print(power_tools)
>>>
[Tool('circular saw', 5), Tool('drill', 4),
 Tool('jackhammer', 40), Tool('sander', 4)]
```

Коли виконується другий виклик `sort` за вагою за спаданням, він бачить, що і `'sander'`, і `'drill'` мають вагу 4. Це змушує метод `sort` помістити обидва елементи в кінцевий список результатів у тому ж порядку, в якому вони з'являлися в оригінальному списку, зберігаючи таким чином їхній відносний порядок за іменем за зростанням:

```python
power_tools.sort(key=lambda x: x.weight, reverse=True)
print(power_tools)
>>>
[Tool('jackhammer', 40), Tool('circular saw', 5),
 Tool('drill', 4), Tool('sander', 4)]
```

Цей самий підхід можна використовувати для комбінування будь-якої кількості різних типів критеріїв сортування в будь-яких напрямках. Потрібно лише переконатися, що сортування виконуються у зворотній послідовності відносно того, що має містити кінцевий список. У цьому прикладі потрібен порядок сортування за вагою за спаданням і потім за іменем за зростанням, тому спочатку виконувалося сортування за іменем, а потім за вагою.

Тим не менш, підхід із поверненням кортежу функцією `key` та використанням унарного заперечення для змішування порядків сортування є простішим для читання та вимагає меншого обсягу коду. Рекомендується використовувати кілька викликів `sort` лише тоді, коли це абсолютно необхідно.

**Що слід запам'ятати**

- Метод `sort` типу `list` може використовуватися для перевпорядкування вмісту списку в природному порядку вбудованих типів, як-от рядки, цілі числа, кортежі тощо.
- Метод `sort` не працює для об'єктів, якщо вони не визначають природний порядок за допомогою спеціальних методів, що є нетиповим.
- Параметр `key` методу `sort` може використовуватися для надання допоміжної функції, що повертає значення для сортування замість кожного елемента зі списку.
- Повернення кортежу з функції `key` дозволяє комбінувати кілька критеріїв сортування. Унарний оператор мінус може використовуватися для обернення окремих порядків сортування для типів, що це допускають.
- Для типів, які не можна заперечити, можна комбінувати кілька критеріїв сортування, викликаючи метод `sort` кілька разів із різними функціями `key` та значеннями `reverse`, у порядку від виклику з найнижчим рангом до виклику з найвищим.

## Порада 15. Будьте обережні, покладаючись на порядок вставки dict

У Python 3.5 і раніше ітерування по `dict` повертало ключі у довільному порядку. Порядок ітерації не відповідав порядку вставки елементів. Наприклад, тут створюється словник, що відображає назви тварин на відповідні назви їхніх дитинчат, після чого виводиться на екран (см. Порада 75: «Використовуйте рядки repr для налагодження» — про те, як це працює):

```python
# Python 3.5
baby_names = {
    'cat': 'kitten',
    'dog': 'puppy',
}
print(baby_names)
>>>
{'dog': 'puppy', 'cat': 'kitten'}
```

При створенні словника ключі були в порядку `'cat'`, `'dog'`, але при виведенні ключі виявилися у зворотному порядку `'dog'`, `'cat'`. Така поведінка є несподіваною, ускладнює відтворення тестових випадків, підвищує складність налагодження та є особливо заплутаною для початківців Python.

Це відбувалося через те, що тип `dict` раніше реалізовував алгоритм хеш-таблиці з комбінацією вбудованої функції `hash` та випадкового початкового числа, що призначалося при запуску інтерпретатора Python. Разом ці механізми призводили до того, що порядок словника не відповідав порядку вставки та випадково змінювався між запусками програми.

Починаючи з Python 3.6, і офіційно закріплено в специфікації Python версії 3.7, словники зберігають порядок вставки. Тепер цей код завжди виводитиме словник у тому вигляді, в якому він спочатку був створений програмістом:

```python
baby_names = {
    'cat': 'kitten',
    'dog': 'puppy',
}
print(baby_names)
>>>
{'cat': 'kitten', 'dog': 'puppy'}
```

У Python 3.5 і раніше всі методи `dict`, що покладаються на порядок ітерації, включаючи `keys`, `values`, `items` та `popitem`, аналогічно демонстрували цю випадкову поведінку:

```python
# Python 3.5
print(list(baby_names.keys()))
print(list(baby_names.values()))
print(list(baby_names.items()))
print(baby_names.popitem())  # Вибирає елемент випадково
>>>
['dog', 'cat']
['puppy', 'kitten']
[('dog', 'puppy'), ('cat', 'kitten')]
('dog', 'puppy')
```

Тепер ці методи забезпечують послідовний порядок вставки, на який можна покладатися при написанні програм:

```python
print(list(baby_names.keys()))
print(list(baby_names.values()))
print(list(baby_names.items()))
print(baby_names.popitem())  # Останній вставлений елемент
>>>
['cat', 'dog']
['kitten', 'puppy']
[('cat', 'kitten'), ('dog', 'puppy')]
('dog', 'puppy')
```

Ця зміна має численні наслідки для інших можливостей Python, що залежать від типу `dict` та його конкретної реалізації. Іменовані аргументи функцій — включаючи узагальнювальний параметр `**kwargs`; см. Порада 23: «Надавайте необов'язкову поведінку за допомогою іменованих аргументів» — раніше надходили у, здавалося б, випадковому порядку, що ускладнювало налагодження викликів функцій:

```python
# Python 3.5
def my_func(**kwargs):
    for key, value in kwargs.items():
        print('%s = %s' % (key, value))
my_func(goose='gosling', kangaroo='joey')
>>>
kangaroo = joey
goose = gosling
```

Тепер порядок іменованих аргументів завжди зберігається відповідно до того, як програміст спочатку викликав функцію:

```python
def my_func(**kwargs):
    for key, value in kwargs.items():
        print(f'{key} = {value}')
my_func(goose='gosling', kangaroo='joey')
>>>
goose = gosling
kangaroo = joey
```

Класи також використовують тип `dict` для своїх словників екземплярів. У попередніх версіях Python поля об'єктів демонстрували цю хаотичну поведінку:

```python
# Python 3.5
class MyClass:
    def __init__(self):
        self.alligator = 'hatchling'
        self.elephant = 'calf'
a = MyClass()
for key, value in a.__dict__.items():
    print('%s = %s' % (key, value))
>>>
elephant = calf
alligator = hatchling
```

Тепер можна вважати, що порядок присвоєння цих полів екземпляра відображатиметься в `__dict__`:

```python
class MyClass:
    def __init__(self):
        self.alligator = 'hatchling'
        self.elephant = 'calf'
a = MyClass()
for key, value in a.__dict__.items():
    print(f'{key} = {value}')
>>>
alligator = hatchling
elephant = calf
```

Спосіб, яким словники зберігають порядок вставки, тепер є частиною специфікації мови Python. Для наведених вище можливостей мови можна покладатися на цю поведінку й навіть робити її частиною API, що ви проєктуєте для своїх класів і функцій.

> **Примітка.** Вже тривалий час у вбудованому модулі `collections` є клас `OrderedDict`, що зберігає порядок вставки. Хоча поведінка цього класу аналогічна поведінці стандартного типу `dict` (починаючи з Python 3.7), характеристики продуктивності `OrderedDict` суттєво відрізняються. Якщо потрібно обробляти велику кількість вставок ключів і викликів `popitem` (наприклад, для реалізації кешу за принципом «останній використаний» (LRU)), `OrderedDict` може краще підходити, ніж стандартний тип `dict` у Python (см. Порада 70: «Профілюйте перед оптимізацією» про те, як переконатися, що це необхідно).

Однак не слід завжди вважати, що поведінка збереження порядку вставки буде присутня при роботі зі словниками. Python спрощує для програмістів визначення власних типів-контейнерів, що емулюють стандартні протоколи, відповідні типам `list`, `dict` та іншим (см. Порада 43: «Успадковуйте від collections.abc для власних типів-контейнерів»). Python не є статично типізованою мовою, тому більшість коду покладається на duck typing — де поведінка об'єкта є його фактичним типом — замість жорстких ієрархій класів. Це може призводити до несподіваних пасток.

Наприклад, є програма для відображення результатів конкурсу на найгарніше дитинча тварини. Починаємо зі словника, що містить загальну кількість голосів за кожного:

```python
votes = {
    'otter': 1281,
    'polar bear': 587,
    'fox': 863,
}
```

Визначаємо функцію для обробки цих даних голосування та збереження рейтингу кожної назви тварини у наданий порожній словник. У цьому випадку словник може бути моделлю даних для елемента інтерфейсу:

```python
def populate_ranks(votes, ranks):
    names = list(votes.keys())
    names.sort(key=votes.get, reverse=True)
    for i, name in enumerate(names, 1):
        ranks[name] = i
```

Також потрібна функція, яка повідомить переможця конкурсу. Вона припускає, що `populate_ranks` заповнить вміст словника `ranks` у зростаючому порядку, тобто перший ключ має бути переможцем:

```python
def get_winner(ranks):
    return next(iter(ranks))
```

Тут підтверджуємо, що ці функції працюють так, як задумано, і дають очікуваний результат:

```python
ranks = {}
populate_ranks(votes, ranks)
print(ranks)
winner = get_winner(ranks)
print(winner)
>>>
{'otter': 1, 'fox': 2, 'polar bear': 3}
otter
```

Тепер уявімо, що вимоги програми змінилися. Елемент інтерфейсу, що відображає результати, повинен показуватися в алфавітному порядку, а не в порядку рейтингу. Для цього можна використати вбудований модуль `collections.abc` для визначення нового словникоподібного класу, що ітерує свій вміст в алфавітному порядку:

```python
from collections.abc import MutableMapping

class SortedDict(MutableMapping):
    def __init__(self):
        self.data = {}
    def __getitem__(self, key):
        return self.data[key]
    def __setitem__(self, key, value):
        self.data[key] = value
    def __delitem__(self, key):
        del self.data[key]
    def __iter__(self):
        keys = list(self.data.keys())
        keys.sort()
        for key in keys:
            yield key
    def __len__(self):
        return len(self.data)
```

Екземпляр `SortedDict` можна використовувати замість стандартного `dict` із попередніми функціями, і помилок не виникне, оскільки цей клас відповідає протоколу стандартного словника. Однак результат буде хибним:

```python
sorted_ranks = SortedDict()
populate_ranks(votes, sorted_ranks)
print(sorted_ranks.data)
winner = get_winner(sorted_ranks)
print(winner)
>>>
{'otter': 1, 'fox': 2, 'polar bear': 3}
fox
```

Проблема в тому, що реалізація `get_winner` припускає, що ітерація по словнику відбувається в порядку вставки, щоб відповідати `populate_ranks`. Цей код використовує `SortedDict` замість `dict`, тому це припущення більше не є правдивим. Таким чином, значення, повернуте для переможця, — `'fox'`, що є першим в алфавітному порядку.

Є три способи пом'якшити цю проблему. По-перше, можна перереалізувати функцію `get_winner` так, щоб вона більше не припускала наявність конкретного порядку ітерації словника `ranks`. Це найконсервативніше та найнадійніше рішення:

```python
def get_winner(ranks):
    for name, rank in ranks.items():
        if rank == 1:
            return name

winner = get_winner(sorted_ranks)
print(winner)
>>>
otter
```

Другий підхід — додати явну перевірку на початку функції, щоб переконатися, що тип `ranks` відповідає очікуванням, і підняти виняток, якщо ні. Це рішення, ймовірно, має кращу продуктивність під час виконання, ніж консервативний підхід:

```python
def get_winner(ranks):
    if not isinstance(ranks, dict):
        raise TypeError('must provide a dict instance')
    return next(iter(ranks))

get_winner(sorted_ranks)
>>>
Traceback ...
TypeError: must provide a dict instance
```

Третій варіант — використання анотацій типів для примусового забезпечення того, що значення, передане в `get_winner`, є екземпляром `dict`, а не `MutableMapping` зі словникоподібною поведінкою (см. Порада 90: «Розгляньте статичний аналіз за допомогою typing для усунення помилок»). Тут запускаємо інструмент `mypy` у суворому режимі для анотованої версії наведеного вище коду:

```python
from typing import Dict, MutableMapping

def populate_ranks(votes: Dict[str, int],
                   ranks: Dict[str, int]) -> None:
    names = list(votes.keys())
    names.sort(key=votes.get, reverse=True)
    for i, name in enumerate(names, 1):
        ranks[name] = i

def get_winner(ranks: Dict[str, int]) -> str:
    return next(iter(ranks))

class SortedDict(MutableMapping[str, int]):
    ...
```

```
$ python3 -m mypy --strict example.py
.../example.py:48: error: Argument 2 to "populate_ranks" has
incompatible type "SortedDict"; expected "Dict[str, int]"
.../example.py:50: error: Argument 1 to "get_winner" has
incompatible type "SortedDict"; expected "Dict[str, int]"
```

Це коректно виявляє невідповідність між типами `dict` та `MutableMapping` і позначає некоректне використання як помилку. Це рішення забезпечує найкращий баланс між статичною безпекою типів та продуктивністю під час виконання.

**Що слід запам'ятати**

- Починаючи з Python 3.7, можна покладатися на те, що ітерування вмісту екземпляра `dict` відбуватиметься в тому самому порядку, в якому ключі були спочатку додані.
- Python спрощує визначення об'єктів, що поводяться як словники, але не є екземплярами `dict`. Для таких типів не можна припускати збереження порядку вставки.
- Є три способи бути обережними щодо словникоподібних класів: писати код, що не покладається на порядок вставки; явно перевіряти тип `dict` під час виконання; або вимагати значень типу `dict` за допомогою анотацій типів і статичного аналізу.

## Порада 16. Надавайте перевагу get над in та KeyError для обробки відсутніх ключів словника

Три фундаментальні операції для взаємодії зі словниками — це доступ, присвоєння та видалення ключів і пов'язаних із ними значень. Вміст словників є динамічним, тому цілком можливо — і навіть вірогідно, — що при спробі звернення або видалення ключа він ще не буде присутній у словнику.

Наприклад, потрібно визначити улюблений вид хліба людей, щоб скласти меню для сендвіч-бару. Тут визначається словник лічильників із поточною кількістю голосів за кожен вид:

```python
counters = {
    'pumpernickel': 2,
    'sourdough': 1,
}
```

Щоб збільшити лічильник для нового голосу, потрібно перевірити наявність ключа, вставити ключ зі значенням лічильника за замовчуванням рівним нулю, якщо він відсутній, а потім збільшити значення лічильника. Це вимагає двох звернень до ключа та одного присвоєння. Тут завдання виконується за допомогою оператора `if` з виразом `in`, що повертає `True`, якщо ключ присутній:

```python
key = 'wheat'
if key in counters:
    count = counters[key]
else:
    count = 0
counters[key] = count + 1
```

Іншим способом досягти тієї самої поведінки є використання того, що словники піднімають виняток `KeyError`, коли намагаєшся отримати значення для неіснуючого ключа. Цей підхід є ефективнішим, оскільки вимагає лише одного доступу та одного присвоєння:

```python
try:
    count = counters[key]
except KeyError:
    count = 0
counters[key] = count + 1
```

Цей потік отримання ключа, що існує, або повернення значення за замовчуванням є настільки поширеним, що вбудований тип `dict` надає метод `get` для виконання цього завдання. Другий параметр `get` — це значення за замовчуванням, що повертається у разі відсутності ключа — першого параметра. Це також вимагає лише одного доступу та одного присвоєння, але значно коротше за приклад із `KeyError`:

```python
count = counters.get(key, 0)
counters[key] = count + 1
```

Можна скорочувати вирази з `in` та `KeyError` різними способами, але всі ці альтернативи страждають від необхідності дублювання коду для присвоєнь, що знижує читабельність і заслуговує на уникнення:

```python
if key not in counters:
    counters[key] = 0
counters[key] += 1

if key in counters:
    counters[key] += 1
else:
    counters[key] = 1

try:
    counters[key] += 1
except KeyError:
    counters[key] = 1
```

Таким чином, для словника з простими типами використання методу `get` є найкоротшим і найзрозумілішим варіантом.

> **Примітка.** Якщо ви ведете словники лічильників, як цей, варто розглянути клас `Counter` з вбудованого модуля `collections`, який надає більшість необхідних функцій.

Що робити, якщо значеннями словника є складніший тип, наприклад список? Скажімо, замість простого підрахунку голосів також потрібно знати, хто голосував за кожен вид хліба. Тут це реалізується шляхом асоціювання списку імен із кожним ключем:

```python
votes = {
    'baguette': ['Bob', 'Alice'],
    'ciabatta': ['Coco', 'Deb'],
}
key = 'brioche'
who = 'Elmer'
if key in votes:
    names = votes[key]
else:
    votes[key] = names = []
names.append(who)
print(votes)
>>>
{'baguette': ['Bob', 'Alice'],
 'ciabatta': ['Coco', 'Deb'],
 'brioche': ['Elmer']}
```

Використання виразу `in` вимагає двох звернень, якщо ключ присутній, або одного звернення та одного присвоєння, якщо ключ відсутній. Цей приклад відрізняється від прикладу з лічильниками вище тим, що значення для кожного ключа можна сліпо присвоїти значенням за замовчуванням — порожній список, якщо ключ ще не існує. Оператор потрійного присвоєння (`votes[key] = names = []`) заповнює ключ в одному рядку замість двох. Після того, як значення за замовчуванням вставлено в словник, більше не потрібно присвоювати його знову, оскільки список змінюється за посиланням у подальшому виклику `append`.

Також можна покладатися на виняток `KeyError` при використанні списків як значень словника. Цей підхід вимагає одного звернення до ключа, якщо він присутній, або одного звернення та одного присвоєння, якщо відсутній, що робить його ефективнішим за умову `in`:

```python
try:
    names = votes[key]
except KeyError:
    votes[key] = names = []
names.append(who)
```

Аналогічно можна використовувати метод `get` для отримання значення-списку, коли ключ присутній, або одного отримання та одного присвоєння, якщо ключ відсутній:

```python
names = votes.get(key)
if names is None:
    votes[key] = names = []
names.append(who)
```

Підхід із використанням `get` для отримання значень-списків можна додатково скоротити на один рядок, якщо використати вираз присвоєння (введений у Python 3.8; см. Порада 10: «Запобігайте повторенню за допомогою виразів присвоєння») в операторі `if`, що підвищує читабельність:

```python
if (names := votes.get(key)) is None:
    votes[key] = names = []
names.append(who)
```

Тип `dict` також надає метод `setdefault` для ще більшого скорочення цього шаблону. `setdefault` намагається отримати значення ключа зі словника. Якщо ключ відсутній, метод присвоює цьому ключу надане значення за замовчуванням. Потім метод повертає значення для цього ключа: або первісно наявне, або щойно вставлене за замовчуванням. Тут `setdefault` використовується для реалізації тієї ж логіки, що і в прикладі з `get` вище:

```python
names = votes.setdefault(key, [])
names.append(who)
```

Це працює як очікується і коротше, ніж використання `get` із виразом присвоєння. Однак читабельність цього підходу не є ідеальною. Назва методу `setdefault` не одразу розкриває свою мету. Чому «set», якщо насправді він отримує значення? Чому не назвати його `get_or_set`? Це лінгвістична суперечка, але суть у тому, що якщо ви є новим читачем коду і не зовсім знайомі з Python, вам може бути важко зрозуміти, що цей код намагається зробити, оскільки `setdefault` не є самопояснювальним.

Є також одна важлива пастка: значення за замовчуванням, передане в `setdefault`, присвоюється безпосередньо до словника, коли ключ відсутній, а не копіюється. Тут демонструється ефект цього, коли значення є списком:

```python
data = {}
key = 'foo'
value = []
data.setdefault(key, value)
print('Before:', data)
value.append('hello')
print('After: ', data)
>>>
Before: {'foo': []}
After:  {'foo': ['hello']}
```

Це означає, що потрібно завжди конструювати нове значення за замовчуванням для кожного ключа, до якого звертаємося через `setdefault`. Це призводить до значних накладних витрат у цьому прикладі, оскільки для кожного виклику доводиться виділяти екземпляр списку. При повторному використанні об'єкта як значення за замовчуванням — що може спробуватися заради ефективності або читабельності — можна внести дивну поведінку та помилки (см. Порада 24: «Використовуйте None та рядки документації для визначення динамічних аргументів за замовчуванням»).

Повертаючись до попереднього прикладу зі словниками лічильників замість списків голосуючих: чому б також не використати метод `setdefault` у цьому випадку? Тут переімплементуємо той самий приклад за допомогою цього підходу:

```python
count = counters.setdefault(key, 0)
counters[key] = count + 1
```

Проблема тут у тому, що виклик `setdefault` є зайвим. Завжди потрібно присвоювати ключ у словнику новому значенню після збільшення лічильника, тому додаткове присвоєння, яке робить `setdefault`, є непотрібним. Попередній підхід із використанням `get` для оновлення лічильників вимагає лише одного звернення та одного присвоєння, тоді як `setdefault` вимагає одного звернення та двох присвоєнь.

Є лише кілька обставин, коли `setdefault` є найкоротшим способом обробки відсутніх ключів словника: наприклад, коли значення за замовчуванням дешево конструювати, є змінними, і немає потенціалу для виникнення винятків (наприклад, екземпляри `list`). У цих дуже специфічних випадках може здатися доцільним прийняти заплутану назву методу `setdefault` замість того, щоб писати більше символів і рядків для використання `get`. Однак у таких ситуаціях часто слід використовувати `defaultdict` (см. Порада 17: «Надавайте перевагу defaultdict над setdefault для обробки відсутніх елементів у внутрішньому стані»).

**Що слід запам'ятати**

- Є чотири поширені способи виявлення та обробки відсутніх ключів у словниках: вирази `in`, винятки `KeyError`, метод `get` та метод `setdefault`.
- Метод `get` найкраще підходить для словників, що містять базові типи на кшталт лічильників, і є кращим варіантом разом із виразами присвоєння, коли створення значень словника має високу вартість або може викликати винятки.
- Якщо метод `setdefault` типу `dict` здається найкращим рішенням для вашої задачі, слід розглянути використання `defaultdict`.

## Порада 17. Надавайте перевагу defaultdict над setdefault для обробки відсутніх елементів у внутрішньому стані

При роботі зі словником, який ви не створювали, існують різні способи обробки відсутніх ключів (см. Порада 16: «Надавайте перевагу get над in та KeyError для обробки відсутніх ключів словника»). Хоча використання методу `get` є кращим підходом, ніж вирази `in` та винятки `KeyError`, для деяких випадків `setdefault` виглядає як найкоротший варіант.

Наприклад, потрібно відстежувати міста, відвідані в різних країнах світу. Тут це реалізується за допомогою словника, що відображає назви країн на екземпляри `set` із відповідними назвами міст:

```python
visits = {
    'Mexico': {'Tulum', 'Puerto Vallarta'},
    'Japan': {'Hakone'},
}
```

Можна використовувати метод `setdefault` для додавання нових міст до множин незалежно від того, чи є вже назва країни в словнику. Цей підхід значно коротший за досягнення тієї самої поведінки за допомогою методу `get` та виразу присвоєння (доступного починаючи з Python 3.8):

```python
visits.setdefault('France', set()).add('Arles')     # Коротко

if (japan := visits.get('Japan')) is None:          # Довго
    visits['Japan'] = japan = set()
japan.add('Kyoto')

print(visits)
>>>
{'Mexico': {'Tulum', 'Puerto Vallarta'},
 'Japan': {'Kyoto', 'Hakone'},
 'France': {'Arles'}}
```

А як щодо ситуації, коли ви контролюєте створення словника, до якого звертаєтесь? Зазвичай це відбувається, коли ви використовуєте екземпляр словника для відстеження внутрішнього стану класу. Тут попередній приклад загортається в клас із допоміжними методами для доступу до динамічного внутрішнього стану, що зберігається в словнику:

```python
class Visits:
    def __init__(self):
        self.data = {}
    def add(self, country, city):
        city_set = self.data.setdefault(country, set())
        city_set.add(city)
```

Цей новий клас приховує складність правильного виклику `setdefault` і надає більш зручний інтерфейс для програміста:

```python
visits = Visits()
visits.add('Russia', 'Yekaterinburg')
visits.add('Tanzania', 'Zanzibar')
print(visits.data)
>>>
{'Russia': {'Yekaterinburg'}, 'Tanzania': {'Zanzibar'}}
```

Однак реалізація методу `Visits.add` все ще не є ідеальною. Метод `setdefault` все ще має заплутану назву, що ускладнює новому читачеві коду негайне розуміння того, що відбувається. Крім того, реалізація є неефективною, оскільки конструює новий екземпляр `set` при кожному виклику, незалежно від того, чи вже присутня вказана країна у словнику `data`.

На щастя, клас `defaultdict` з вбудованого модуля `collections` спрощує цей поширений випадок використання, автоматично зберігаючи значення за замовчуванням при відсутності ключа. Все, що потрібно зробити, — надати функцію, яка повертатиме значення за замовчуванням щоразу, коли ключ відсутній (приклад Поради 38: «Приймайте функції замість класів для простих інтерфейсів»). Тут клас `Visits` переписується з використанням `defaultdict`:

```python
from collections import defaultdict

class Visits:
    def __init__(self):
        self.data = defaultdict(set)
    def add(self, country, city):
        self.data[country].add(city)

visits = Visits()
visits.add('England', 'Bath')
visits.add('England', 'London')
print(visits.data)
>>>
defaultdict(<class 'set'>, {'England': {'London', 'Bath'}})
```

Тепер реалізація `add` є короткою і простою. Код може вважати, що звернення до будь-якого ключа у словнику `data` завжди повертатиме наявний екземпляр `set`. Зайвих екземплярів `set` виділятися не буде, що може бути витратним при великій кількості викликів методу `add`.

Використання `defaultdict` значно краще, ніж `setdefault` для такого типу ситуацій (см. Порада 37: «Компонуйте класи замість глибокого вкладення вбудованих типів»). Є ще випадки, де `defaultdict` не вирішуватиме ваших проблем, але в Python є ще більше інструментів для подолання цих обмежень (см. Порада 18: «Знайте, як створювати значення за замовчуванням, що залежать від ключа, за допомогою `__missing__`», Порада 43: «Успадковуйте від collections.abc для власних типів-контейнерів» та вбудований клас `collections.Counter`).

**Що слід запам'ятати**

- Якщо ви створюєте словник для управління довільним набором потенційних ключів, слід надавати перевагу використанню екземпляра `defaultdict` з вбудованого модуля `collections`, якщо він підходить для вашої задачі.
- Якщо словник із довільними ключами передається вам і ви не контролюєте його створення, слід надавати перевагу методу `get` для доступу до його елементів. Однак варто розглянути використання методу `setdefault` у небагатьох ситуаціях, де він призводить до коротшого коду.

## Порада 18. Знайте, як створювати значення за замовчуванням, що залежать від ключа, за допомогою \_\_missing\_\_

Метод `setdefault` вбудованого типу `dict` призводить до коротшого коду при обробці відсутніх ключів у деяких конкретних обставинах (см. Порада 16: «Надавайте перевагу get над in та KeyError для обробки відсутніх ключів словника»). Для багатьох із цих ситуацій кращим інструментом є тип `defaultdict` з вбудованого модуля `collections` (см. Порада 17: «Надавайте перевагу defaultdict над setdefault для обробки відсутніх елементів у внутрішньому стані»). Однак є випадки, коли ні `setdefault`, ні `defaultdict` не підходять.

Наприклад, є програма для управління фотографіями профілів соціальної мережі у файловій системі. Потрібен словник для відображення шляхів до зображень профілів на відкриті файлові дескриптори, щоб можна було читати та записувати ці зображення за потреби. Тут це реалізується за допомогою звичайного екземпляра `dict` та перевірки наявності ключів методом `get` та виразом присвоєння (введений у Python 3.8; см. Порада 10: «Запобігайте повторенню за допомогою виразів присвоєння»):

```python
pictures = {}
path = 'profile_1234.png'
if (handle := pictures.get(path)) is None:
    try:
        handle = open(path, 'a+b')
    except OSError:
        print(f'Failed to open path {path}')
        raise
    else:
        pictures[path] = handle
handle.seek(0)
image_data = handle.read()
```

Якщо файловий дескриптор вже існує в словнику, цей код виконує лише одне звернення до словника. У разі відсутності файлового дескриптора словник звертається один раз через `get`, а потім йому присвоюється значення в блоці `else` конструкції `try/except`. (Цей підхід також працює з `finally`; см. Порада 65: «Скористайтеся перевагами кожного блоку в try/except/else/finally».) Виклик методу `read` чітко відділений від коду, що викликає `open` та обробляє винятки.

Хоча можна використовувати вираз `in` або підходи з `KeyError` для реалізації тієї самої логіки, ці варіанти вимагають більше звернень до словника та рівнів вкладеності. З огляду на те, що ці інші варіанти працюють, можна також припустити, що метод `setdefault` теж спрацює:

```python
try:
    handle = pictures.setdefault(path, open(path, 'a+b'))
except OSError:
    print(f'Failed to open path {path}')
    raise
else:
    handle.seek(0)
    image_data = handle.read()
```

Цей код має багато проблем. Вбудована функція `open` для створення файлового дескриптора завжди викликається, навіть якщо шлях вже присутній у словнику. Це призводить до додаткового файлового дескриптора, що може конфліктувати з наявними відкритими дескрипторами в тій самій програмі. Винятки можуть виникати при виклику `open` і потребувати обробки, але неможливо відрізнити їх від винятків, що можуть виникати при виклику `setdefault` в тому ж рядку (що можливо для інших словникоподібних реалізацій; см. Порада 43: «Успадковуйте від collections.abc для власних типів-контейнерів»).

При управлінні внутрішнім станом можна також припустити, що `defaultdict` можна використовувати для відстеження цих фотографій профілів. Тут спроба реалізувати ту саму логіку, що і раніше, але тепер за допомогою допоміжної функції та класу `defaultdict`:

```python
from collections import defaultdict

def open_picture(profile_path):
    try:
        return open(profile_path, 'a+b')
    except OSError:
        print(f'Failed to open path {profile_path}')
        raise

pictures = defaultdict(open_picture)
handle = pictures[path]
handle.seek(0)
image_data = handle.read()
>>>
Traceback ...
TypeError: open_picture() missing 1 required positional
argument: 'profile_path'
```

Проблема в тому, що `defaultdict` очікує, що функція, передана в його конструктор, не вимагає жодних аргументів. Це означає, що допоміжна функція, яку викликає `defaultdict`, не знає, до якого конкретного ключа відбувається звернення, що унеможливлює виклик `open`. У цій ситуації і `setdefault`, і `defaultdict` не відповідають потребам.

На щастя, ця ситуація є досить поширеною, і Python має інше вбудоване рішення. Можна успадкуватися від типу `dict` та реалізувати спеціальний метод `__missing__` для додавання власної логіки обробки відсутніх ключів. Тут це реалізується шляхом визначення нового класу, що використовує той самий допоміжний метод `open_picture`, визначений вище:

```python
class Pictures(dict):
    def __missing__(self, key):
        value = open_picture(key)
        self[key] = value
        return value

pictures = Pictures()
handle = pictures[path]
handle.seek(0)
image_data = handle.read()
```

Коли звернення до словника `pictures[path]` виявляє, що ключ `path` відсутній у словнику, викликається метод `__missing__`. Цей метод повинен створити нове значення за замовчуванням для ключа, вставити його в словник та повернути викликаючому коду. Подальші звернення за тим самим шляхом не викликатимуть `__missing__`, оскільки відповідний елемент вже присутній (аналогічно до поведінки `__getattr__`; см. Порада 47: «Використовуйте `__getattr__`, `__getattribute__` та `__setattr__` для «ледачих» атрибутів»).

**Що слід запам'ятати**

- Метод `setdefault` типу `dict` погано підходить, коли створення значення за замовчуванням має високу обчислювальну вартість або може викликати винятки.
- Функція, передана в `defaultdict`, не повинна вимагати жодних аргументів, що унеможливлює залежність значення за замовчуванням від ключа, до якого відбувається звернення.
- Можна визначити власний підклас `dict` із методом `__missing__` для конструювання значень за замовчуванням, яким необхідно знати, до якого ключа відбувалося звернення.
