# Розділ 12. Об'єктно-орієнтоване проектування

## Цілі

- Розуміти процес об'єктно-орієнтованого проектування.
- Вміти читати і розуміти об'єктно-орієнтовані програми.
- Розуміти концепції інкапсуляції, поліморфізму і успадкування стосовно ООП-проектування.
- Вміти проектувати програми середньої складності за допомогою об'єктно-орієнтованого проектування.


## 12.1 Процес ООП

Тепер, знаючи деякі техніки структурування даних, настав час розправити крила і справді застосувати ці інструменти. Більшість сучасних комп'ютерних застосунків проектуються з використанням підходу, орієнтованого на дані. Цей так званий процес **об'єктно-орієнтованого проектування** (ООП) є потужним доповненням до низхідного проектування для розробки надійних і рентабельних програмних систем.

Суть проектування полягає в описі системи у вигляді магічних «чорних ящиків» та їхніх інтерфейсів. Кожен компонент надає набір послуг через свій інтерфейс. Інші компоненти є користувачами, або клієнтами, цих послуг.

Клієнту потрібно лише розуміти інтерфейс послуги; деталі реалізації не важливі. Насправді внутрішні деталі можуть кардинально змінитися, не зачіпаючи клієнта. Цей розподіл відповідальності і робить проектування складних систем можливим.

- При **низхідному проектуванні** роль магічних чорних ящиків відіграють функції.
- При **ООП** чорними ящиками є об'єкти. Магія об'єктів лежить у визначеннях класів.

Якщо ми можемо розбити велику задачу на набір взаємодіючих класів, ми різко зменшуємо складність, яку потрібно розуміти в будь-якій частині програми. Кожен клас стоїть сам по собі. Об'єктно-орієнтоване проектування — це процес знаходження і визначення корисного набору класів для конкретної задачі. Як і будь-яке проектування, це поєднання мистецтва і науки.

Ось деякі інтуїтивні орієнтири для ООП:

1. **Шукайте кандидатів на об'єкти.** Уважно розгляньте умову задачі. Об'єкти зазвичай описуються іменниками. Підкресліть всі іменники в умові і розгляньте їх по черзі.

2. **Визначте змінні примірника.** Коли ви знайшли потенційні об'єкти, подумайте про те, яку інформацію кожен об'єкт буде потребувати для своєї роботи.

3. **Думайте про інтерфейси.** Коли ви визначили потенційний об'єкт/клас і деякі асоційовані дані, подумайте, які операції будуть потрібні. Розгляньте дієслова в умові.

4. **Уточнюйте нетривіальні методи.** Деякі методи можна реалізувати кількома рядками. Інші вимагають значних зусиль. Використовуйте низхідне проектування і поетапне уточнення.

5. **Проектуйте ітеративно.** Ви будете переходити між проектуванням нових класів і додаванням методів до існуючих. Беріться за те, що здається найбільш нагальним.

6. **Спробуйте альтернативи.** Не бійтеся відкинути підхід, що не спрацьовує. Хороше проектування передбачає чимало спроб і помилок. Фред Брукс, легендарний програмний інженер, сформулював правило: «Плануйте викинути один варіант» — нерідко ви не знаєте, як слід будувати систему, поки не побудуєте її неправильно.

7. **Залишайте простоту.** На кожному кроці намагайтеся знайти найпростіший підхід.

Наступні розділи проведуть вас через кілька прикладів, що ілюструють аспекти ООП.


## 12.2 Приклад: симуляція ракетболу

Для нашого першого прикладу повернемося до симуляції ракетболу з розділу 9. Цього разу також враховуватимемо «всухо». Якщо один гравець досягає 7, а інший ще не набрав жодного очка — гра закінчується. Симуляція повинна відстежувати як перемоги кожного гравця, так і кількість «всухо».


### 12.2.1 Кандидати на об'єкти і методи

Нам потрібно симулювати серію ігор і записувати статистику. Це вказує на два базових завдання: симуляцію гри і ведення статистики.

**Клас RBallGame** — об'єкт для однієї гри. Конструктор приймає два параметри — ймовірності. Методи: `play()` — симулює гру.

**Клас SimStats** — збирає статистику по серії ігор. Методи: конструктор (ініціалізація лічильників), `update(game)` (оновлення після кожної гри), `printReport()` (вивід звіту).

Маючи ці два класи, ми можемо написати функцію `main`:

```python
def main():
    printIntro()
    probA, probB, n = getInputs()
    # Зіграти ігри
    stats = SimStats()
    for i in range(n):
        theGame = RBallGame(probA, probB)  # створити нову гру
        theGame.play()                     # зіграти
        stats.update(theGame)              # отримати результат
    # Вивести результати
    stats.printReport()
```

### 12.2.5 Повна програма

```python
# objrball.py -- Симуляція гри в ракетбол.
#                Ілюструє проектування з об'єктами.

from random import random


class Player:
    # Player відстежує ймовірність подачі і рахунок

    def __init__(self, prob):
        self.prob = prob
        self.score = 0

    def winsServe(self):
        return random() < self.prob

    def incScore(self):
        self.score = self.score + 1

    def getScore(self):
        return self.score


class RBallGame:
    # RBallGame представляє гру в процесі. Гра має двох гравців
    # і відстежує, хто зараз подає.

    def __init__(self, probA, probB):
        self.playerA = Player(probA)
        self.playerB = Player(probB)
        self.server = self.playerA  # гравець A завжди подає першим

    def play(self):
        while not self.isOver():
            if self.server.winsServe():
                self.server.incScore()
            else:
                self.changeServer()

    def isOver(self):
        a, b = self.getScores()
        return a == 15 or b == 15 or \
               (a == 7 and b == 0) or (b == 7 and a == 0)

    def changeServer(self):
        if self.server == self.playerA:
            self.server = self.playerB
        else:
            self.server = self.playerA

    def getScores(self):
        return self.playerA.getScore(), self.playerB.getScore()


class SimStats:
    # SimStats обробляє накопичення статистики по кількох
    #     (завершених) іграх.

    def __init__(self):
        self.winsA = 0
        self.winsB = 0
        self.shutsA = 0
        self.shutsB = 0

    def update(self, aGame):
        a, b = aGame.getScores()
        if a > b:
            self.winsA = self.winsA + 1
            if b == 0:
                self.shutsA = self.shutsA + 1
        else:
            self.winsB = self.winsB + 1
            if a == 0:
                self.shutsB = self.shutsB + 1

    def printReport(self):
        n = self.winsA + self.winsB
        print("Summary of", n, "games:\n")
        print("         wins (% total)  shutouts (% wins)")
        print("--------------------------------------------")
        self.printLine("A", self.winsA, self.shutsA, n)
        self.printLine("B", self.winsB, self.shutsB, n)

    def printLine(self, label, wins, shuts, n):
        template = "Player {0}: {1:5}   ({2:5.1%}) {3:11}   ({4})"
        if wins == 0:
            shutStr = "-----"
        else:
            shutStr = "{0:4.1%}".format(float(shuts) / wins)
        print(template.format(label, wins, float(wins) / n, shuts, shutStr))


def printIntro():
    print("Ця програма симулює ігри в ракетбол між двома")
    print('гравцями, що називаються "A" і "B". Здатність кожного гравця')
    print("позначається ймовірністю (числом від 0 до 1) того, що")
    print("гравець виграє очко при подачі. Гравець A завжди")
    print("подає першим.\n")


def getInputs():
    a = float(input("Яка ймовірність перемоги гравця A на подачі? "))
    b = float(input("Яка ймовірність перемоги гравця B на подачі? "))
    n = int(input("Скільки ігор симулювати? "))
    return a, b, n


def main():

    probA, probB, n = getInputs()

    stats = SimStats()
    for i in range(n):
        theGame = RBallGame(probA, probB)
        theGame.play()
        stats.update(theGame)

    stats.printReport()


main()
input("\nНатисніть <Enter>, щоб вийти")
```


## 12.3 Приклад: покер на кістках

Завершимо розділ, розглянувши графічний застосунок із використанням деяких віджетів, розроблених у попередніх розділах.


### 12.3.1 Специфікація програми

Ціль — написати програму-відеопокер на кістках. Правила:

- Гравець починає з $100.
- Кожен раунд коштує $10.
- Гравець спочатку кидає всі п'ять кісток.
- Гравець має дві можливості покращити руку, перекидаючи деякі або всі кістки.
- Наприкінці руки гравець отримує виплату:

| рука | виплата |
|------|---------|
| Дві пари | $5 |
| Трійка | $8 |
| Фул-хаус (пара + трійка) | $12 |
| Каре | $15 |
| Стрейт (1–5 або 2–6) | $20 |
| П'ятірка | $30 |

Програма повинна мати графічний інтерфейс.


### 12.3.2 Визначення кандидатів на об'єкти

Гра використовує п'ять кісток — це колекція. **Клас Dice** інкапсулює цю колекцію. Очевидні операції: конструктор, `rollAll`, `roll`, `values`, `score`.

Також можна розглядати всю програму як об'єкт — **PokerApp**.

Третій компонент — **інтерфейс користувача**. Гарний підхід — відокремити інтерфейс від основної логіки гри. Це називається підходом **модель-представлення** (model-view). Наша програма реалізує модель (покер на кістках), а інтерфейс — представлення поточного стану моделі.

Використовуватимемо об'єкт інтерфейсу — **PokerInterface**.


### 12.3.3 Реалізація моделі

**Клас Dice:**

```python
class Dice:
    def __init__(self):
        self.dice = [0] * 5
        self.rollAll()

    def roll(self, which):
        for pos in which:
            self.dice[pos] = randrange(1, 7)

    def rollAll(self):
        self.roll(range(5))

    def values(self):
        return self.dice[:]  # повертаємо копію!

    def score(self):
        # Створити список лічильників
        counts = [0] * 7
        for value in self.dice:
            counts[value] = counts[value] + 1

        # Оцінити руку
        if 5 in counts:
            return "Five of a Kind", 30
        elif 4 in counts:
            return "Four of a Kind", 15
        elif (3 in counts) and (2 in counts):
            return "Full House", 12
        elif 3 in counts:
            return "Three of a Kind", 8
        elif not (2 in counts) and (counts[1] == 0 or counts[6] == 0):
            return "Straight", 20
        elif counts.count(2) == 2:
            return "Two Pairs", 5
        else:
            return "Garbage", 0
```

Зверніть увагу на метод `values`: він повертає копію списку кісток (`self.dice[:]`). Це **захисне програмування** — якщо клієнт змінить отриманий список, це не зачепить оригінал всередині об'єкта.

Для перевірки на стрейт: оскільки ми вже перевірили 5, 4 і 3 однакових, відсутність пари гарантує п'ять різних значень. Якщо немає 6 — то значення 1–5, якщо немає 1 — то 2–6.

**Клас PokerApp** з текстовим інтерфейсом для тестування:

```python
class PokerApp:
    def __init__(self, interface):
        self.dice = Dice()
        self.money = 100
        self.interface = interface

    def run(self):
        while self.money >= 10 and self.interface.wantToPlay():
            self.playRound()
        self.interface.close()

    def playRound(self):
        self.money = self.money - 10
        self.interface.setMoney(self.money)
        self.doRolls()
        result, score = self.dice.score()
        self.interface.showResult(result, score)
        self.money = self.money + score
        self.interface.setMoney(self.money)

    def doRolls(self):
        self.dice.rollAll()
        roll = 1
        self.interface.setDice(self.dice.values())
        toRoll = self.interface.chooseDice()
        while roll < 3 and toRoll != []:
            self.dice.roll(toRoll)
            roll = roll + 1
            self.interface.setDice(self.dice.values())
            if roll < 3:
                toRoll = self.interface.chooseDice()
```

### 12.3.5 Розробка GUI

Тепер побудуємо графічний інтерфейс. Основні рішення щодо взаємодії:

- Дисплей рахунку та кістки відображаються постійно.
- Кнопки «Roll Dice», «Score» і «Quit».

**Допоміжний метод `choose`** активує потрібні кнопки і чекає кліку:

```python
def choose(self, choices):
    buttons = self.buttons

    # активувати вибрані кнопки, деактивувати інші
    for b in buttons:
        if b.getLabel() in choices:
            b.activate()
        else:
            b.deactivate()

    # чекати кліку на активну кнопку
    while True:
        p = self.win.getMouse()
        for b in buttons:
            if b.clicked(p):
                return b.getLabel()  # вихід із методу
```

**Метод `chooseDice`** — основний метод введення в GUI:

```python
def chooseDice(self):
    choices = []  # Ще жодна кістка не вибрана
    while True:
        b = self.choose(["Die 1", "Die 2", "Die 3", "Die 4", "Die 5",
                         "Roll Dice", "Score"])

        if b[0] == "D":              # Клацнута кнопка кістки
            i = int(b[4]) - 1       # Перетворити мітку на індекс
            if i in choices:         # Зараз вибрана — зняти вибір
                choices.remove(i)
                self.dice[i].setColor("black")
            else:                    # Зараз невибрана — вибрати
                choices.append(i)
                self.dice[i].setColor("gray")
        else:                        # Клацнуто Roll або Score
            for d in self.dice:      # Відновити вигляд всіх кісток
                d.setColor("black")
            if b == "Score":         # Score — ігнорувати вибір
                return []
            elif choices != []:      # Roll не приймати без вибору
                return choices
```


## 12.4 Концепції ООП

Два приклади — симуляція ракетболу і відеопокер — ілюструють потужність і принади об'єктно-орієнтованого підходу. Більшість спеціалістів з ООП виділяють три властивості, що разом роблять розробку справді об'єктно-орієнтованою: **інкапсуляція**, **поліморфізм** і **успадкування**.


### 12.4.1 Інкапсуляція

Об'єкти щось знають і щось можуть робити. Вони поєднують дані і операції. Цей процес упаковки даних разом із набором операцій, що можна виконувати над ними, і називається **інкапсуляцією**.

Інкапсуляція надає критично важливу послугу розподілу відповідальності між «що» і «як». Фактична реалізація об'єкта не залежить від способу його використання. Реалізація може змінюватися, але поки інтерфейс зберігається, інші компоненти, що залежать від об'єкта, не зламаються.

Ще одна перевага — підтримка **повторного використання коду**. Класи DieView і Button — хороші приклади багаторазово використовуваних компонентів.

Інкапсуляція — мабуть, головна перевага використання об'єктів. Але сама по собі вона робить систему лише **об'єктно-базованою**. Щоб бути справді об'єктно-орієнтованою, система також повинна мати поліморфізм і успадкування.


### 12.4.2 Поліморфізм

Буквально слово «поліморфізм» означає «багато форм». В ООП-літературі це означає, що те, що об'єкт робить у відповідь на повідомлення (виклик методу), залежить від типу або класу об'єкта.

Наша програма-покер проілюструвала один аспект поліморфізму. PokerApp використовувався як із TextInterface, так і з GraphicsInterface. Були дві різних форми інтерфейсу, і PokerApp чудово функціонував із будь-яким. Коли PokerApp викликав метод `showDice`, TextInterface показував кістки одним способом, а GraphicsInterface — іншим.

Примітно, що один рядок у програмі може виконувати зовсім різні методи в різні моменти часу. Наприклад, малювання всіх об'єктів у списку:

```python
for obj in objects:
    obj.draw(win)
```

Коли `obj` — коло, виконується метод `draw` класу Circle. Коли `obj` — прямокутник, виконується метод `draw` класу Rectangle. Поліморфізм дає об'єктно-орієнтованим системам гнучкість для того, щоб кожен об'єкт виконував дію саме так, як це потрібно для цього об'єкта.


### 12.4.3 Успадкування

Третя важлива властивість, **успадкування** — та, яку ми ще не використовували широко. Ідея полягає в тому, що новий клас може бути визначений для запозичення поведінки з іншого класу. Новий клас (той, що запозичує) називається **підкласом** (subclass), а існуючий клас (той, у якого запозичують) — **суперкласом** (superclass).

Наприклад, якщо ми будуємо систему для обліку співробітників, може бути клас Employee із загальною інформацією. Підкласами могли б бути SalariedEmployee і HourlyEmployee — вони успадковують такі методи, як `homeAddress`, але кожен має свою реалізацію `monthlyPay`.

Успадкування дає дві переваги: уникнення дублювання операцій і можливість базувати нові класи на існуючих (сприяє повторному використанню коду).

Ми могли б використати успадкування для нашої програми-покеру. Замість модифікації вихідного класу DieView можна залишити його незмінним і створити новий підклас ColorDieView:

```python
class ColorDieView(DieView):

    def setValue(self, value):
        self.value = value
        DieView.setValue(self, value)

    def setColor(self, color):
        self.foreground = color
        self.setValue(self.value)
```

Перший рядок визначення говорить, що ми визначаємо новий клас ColorDieView, що є підкласом DieView.

Метод `setValue` у ColorDieView **перевизначає** (overrides) визначення setValue, надане в DieView. Він спочатку зберігає значення, а потім покладається на метод setValue суперкласу для фактичного малювання крапок. Зверніть увагу, як виконується виклик методу суперкласу:

```python
DieView.setValue(self, value)
```

Звичайний `self.setValue(value)` посилався б на метод ColorDieView, оскільки `self` є примірником ColorDieView. Щоб викликати оригінальний метод суперкласу, потрібно явно вказати ім'я класу, а фактичний об'єкт, до якого застосовується метод, передається як перший параметр.


## 12.5 Підсумок розділу

- Об'єктно-орієнтоване проектування (ООП) — це процес розробки набору класів для розв'язання задачі. Воно схоже на низхідне проектування тим, що метою є розробка набору чорних ящиків і пов'язаних інтерфейсів. Там, де низхідне проектування шукає функції, ООП шукає об'єкти.

- Існує багато різних способів робити ООП. Найкращий спосіб навчитися — практикуватися. Інтуїтивні орієнтири: 1) шукайте кандидатів на об'єкти; 2) визначте змінні примірника; 3) думайте про інтерфейси; 4) уточнюйте нетривіальні методи; 5) проектуйте ітеративно; 6) пробуйте альтернативи; 7) залишайте простоту.

- При розробці програм із складними інтерфейсами корисно відокремлювати програму на компоненти **моделі** і **представлення**. Одна перевага цього підходу — програма може мати кілька різних вглядів (наприклад, текстовий і GUI-інтерфейси).

- Три фундаментальні принципи ООП:
  - **Інкапсуляція** — відокремлення деталей реалізації об'єкта від способу його використання. Це дозволяє модульне проектування складних програм.
  - **Поліморфізм** — різні класи можуть реалізовувати методи з тим самим підписом. Це робить програми гнучкішими, дозволяючи одному рядку коду викликати різні методи в різних ситуаціях.
  - **Успадкування** — новий клас може бути похідним від існуючого. Це підтримує спільне використання методів між класами і повторне використання коду.


## 12.6 Вправи

### Питання для повторення

**Правда чи хиба**

1. Об'єктно-орієнтоване проектування — це процес знаходження і визначення корисного набору функцій для вирішення задачі.
2. Кандидатів на об'єкти можна знаходити, дивлячись на дієслова в описі задачі.
3. Як правило, процес проектування включає значну кількість спроб і помилок.
4. GUI-системи нерідко будуються з архітектурою модель-представлення.
5. Приховування деталей об'єкта у визначенні класу називається інстанціюванням.
6. Поліморфізм буквально означає «багато змін».
7. Суперклас успадковує поведінку від своїх підкласів.
8. GUI-системи, як правило, простіше писати, ніж текстові інтерфейси.
