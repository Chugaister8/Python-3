## 13.1 Пошук

Почнемо з розгляду дуже поширеної і добре вивченої задачі програмування: пошуку. Пошук — це процес знаходження конкретного значення в колекції. Наприклад, програма, що веде список членів клубу, може потребувати пошуку інформації про конкретного члена.


### 13.1.1 Проста задача пошуку

Ось специфікація простої функції пошуку:

```python
def search(x, nums):
    # nums — список чисел, x — число
    # Повертає позицію в списку, де зустрічається x,
    # або -1, якщо x немає в списку.
```

Кілька інтерактивних прикладів:

```python
>>> search(4, [3, 1, 4, 2, 5])
2
>>> search(7, [3, 1, 4, 2, 5])
-1
```

Як зазначалося, Python надає кілька вбудованих методів для пошуку. Ми могли б реалізувати `search` за допомогою `index`, перехоплюючи виняток:

```python
def search(x, nums):
    try:
        return nums.index(x)
    except:
        return -1
```

Але це обходить справжнє питання. Як насправді Python здійснює пошук у списку? Який алгоритм?


### 13.1.2 Стратегія 1: лінійний пошук

Спробуємо розробити алгоритм пошуку, використовуючи простий підхід «будь комп'ютером». Уявіть, що вам дали сторінку чисел у довільному порядку і запитали, чи є серед них число 13. Найімовірніше, ви просто проглядали б список, порівнюючи кожне значення з 13. Коли бачите 13 — зупиняєтеся. Якщо дійшли до кінця без знаходження — кажете, що 13 немає.

Ця стратегія називається **лінійним пошуком**. Він безпосередньо перекладається в простий код:

```python
def search(x, nums):
    for i in range(len(nums)):
        if nums[i] == x:   # елемент знайдено — повертаємо індекс
            return i
    return -1              # цикл завершено — елемента немає
```

Цей алгоритм добре підходить для невеликих списків і для невпорядкованих даних. Оператори Python `in` і `index` обидва реалізують алгоритми лінійного пошуку.

Якщо дані дуже великі, можна їх впорядкувати і тоді шукати значно ефективніше.


### 13.1.3 Стратегія 2: бінарний пошук

Якщо список впорядкований, існує значно краща стратегія пошуку, яку ви, мабуть, вже знаєте. Чи доводилося вам грати в гру «відгадай число»? Я загадую число від 1 до 100, і ви намагаєтеся вгадати його. Після кожної відповіді я кажу, чи більше, чи менше моє число. Яка ваша стратегія?

Практично будь-який дорослий спочатку вгадає 50. Якщо число більше — діапазон 50–100. Наступна логічна відповідь — 75. Щоразу ми вгадуємо середину з тих, що залишилися, щоб звузити можливий діапазон. Ця стратегія називається **бінарним пошуком**.

```python
def search(x, nums):
    low = 0
    high = len(nums) - 1

    while low <= high:            # ще є діапазон для пошуку
        mid = (low + high) // 2   # позиція середнього елемента
        item = nums[mid]
        if x == item:             # знайдено — повертаємо індекс
            return mid
        elif x < item:            # x у нижній половині
            high = mid - 1        # переміщуємо верхню межу вниз
        else:                     # x у верхній половині
            low = mid + 1         # переміщуємо нижню межу вгору
    return -1                     # діапазону не залишилося, x відсутній
```


### 13.1.4 Порівняння алгоритмів

Який з них кращий? Лінійний пошук простіший, але бінарний ефективніший. Комп'ютерні вчені аналізують ефективність алгоритмів, розглядаючи кількість кроків відносно розміру введення.

**Лінійний пошук:** для списку з n елементів може знадобитися до n порівнянь. Кількість кроків лінійно залежить від n. Це алгоритм лінійного часу.

**Бінарний пошук:** для списку з 16 елементів:

| розмір списку | ділень навпіл |
|---|---|
| 1 | 0 |
| 2 | 1 |
| 4 | 2 |
| 8 | 3 |
| 16 | 4 |

Якщо цикл виконується i разів, він знаходить значення в списку розміром 2ⁱ. Щоб знайти, скільки елементів переглядається для списку розміром n, вирішуємо: n = 2ⁱ для i. Тобто i = log₂ n. Бінарний пошук — це алгоритм **логарифмічного часу**.

Наскільки бінарний пошук ефективніший? Телефонна книга Нью-Йорка з 12 мільйонами записів. Щоб знайти ім'я бінарним пошуком, потрібно log₂(12 000 000) ≈ 24 здогади. Для лінійного пошуку — в середньому 6 мільйонів. Бінарний пошук — феноменально хороший алгоритм!

Чому Python не використовує бінарний пошук для своїх вбудованих операцій? Бінарний пошук менш загальний: щоб він працював, список має бути впорядкованим. Якщо потрібно використовувати бінарний пошук для невпорядкованого списку, спочатку його треба відсортувати.


## 13.2 Рекурсивне розв'язання задач

Основна ідея бінарного пошуку — послідовне ділення задачі навпіл. Це іноді називають підходом «**розділяй і владарюй**» (divide-and-conquer), і він нерідко призводить до дуже ефективних алгоритмів.

Один цікавий аспект алгоритмів «розділяй і владарюй» — вихідна задача ділиться на підзадачі, що є меншими версіями вихідної. Наприклад, бінарний пошук у половині списку — це просто бінарний пошук меншого розміру.


### 13.2.1 Рекурсивні визначення

Опис чогось, що посилається на себе, називається **рекурсивним визначенням**. Класичний рекурсивний приклад у математиці — факторіал.

У розділі 3 ми визначили факторіал як:

n! = n(n−1)(n−2)···(1)

Але є інше, рекурсивне визначення:

```
n! = { 1           якщо n = 0
       n * (n−1)!  інакше
```

Це визначення не є круговим. Щоразу при застосуванні ми запитуємо факторіал меншого числа. Зрештою дійдемо до 0, що не потребує іншого застосування визначення. Це **базовий випадок** рекурсії.

Всі правильні рекурсивні визначення мають такі ключові характеристики:

1. Є один або більше базових випадків, для яких рекурсія не потрібна.
2. Всі ланцюжки рекурсії зрештою досягають одного з базових випадків.


### 13.2.2 Рекурсивні функції

Рекурсивне визначення факторіала безпосередньо перекладається в код:

```python
def fact(n):
    if n == 0:
        return 1
    else:
        return n * fact(n - 1)
```

Функція спочатку перевіряє базовий випадок n == 0 і, якщо він виконується, повертає 1. В іншому разі вона повертає результат множення n на факторіал n−1. Останнє обчислюється рекурсивним викликом `fact(n−1)`.

```python
>>> from recfact import fact
>>> fact(4)
24
>>> fact(10)
3628800
```

Це природно випливає з семантики функцій: кожен виклик функції запускає її заново, маючи власну копію локальних значень і параметрів.


### 13.2.3 Приклад: обернення рядка

Рядок можна розглядати як рекурсивний об'єкт: перший символ і «решта». Якщо перевернути «решту» і потім додати перший символ у кінець, отримаємо обернений рядок.

Спочатку без базового випадку (помилковий варіант):

```python
def reverse(s):
    return reverse(s[1:]) + s[0]
```

Це призводить до нескінченної рекурсії! Правильна версія з базовим випадком:

```python
def reverse(s):
    if s == "":
        return s
    else:
        return reverse(s[1:]) + s[0]
```

```python
>>> reverse("Hello")
'olleH'
```


### 13.2.4 Приклад: анаграми

Анаграма — слово, утворене перестановкою літер. Функція для генерації всіх анаграм рядка:

```python
def anagrams(s):
    if s == "":
        return [s]
    else:
        ans = []
        for w in anagrams(s[1:]):
            for pos in range(len(w) + 1):
                ans.append(w[:pos] + s[0] + w[pos:])
        return ans
```

```python
>>> anagrams("abc")
['abc', 'bac', 'bca', 'acb', 'cab', 'cba']
```

Кількість анаграм слова дорівнює факторіалу його довжини.


### 13.2.5 Приклад: швидке піднесення до степеня

Наївний спосіб обчислити aⁿ — перемножити a само на себе n разів. Алгоритм «розділяй і владарюй» дає кращий варіант. Знаючи, що 2⁸ = 2⁴ × 2⁴, можна обчислити 2⁴ один раз і помножити результат на себе.

Загальне рекурсивне відношення:

```
aⁿ = a^(n//2) * a^(n//2)           якщо n парне
aⁿ = a^(n//2) * a^(n//2) * a       якщо n непарне
```

Базовий випадок: a⁰ = 1.

```python
def recPower(a, n):
    # Зводить a до цілого степеня n
    if n == 0:
        return 1
    else:
        factor = recPower(a, n // 2)
        if n % 2 == 0:    # n парне
            return factor * factor
        else:             # n непарне
            return factor * factor * a
```

Використання проміжної змінної `factor` гарантує, що a^(n//2) обчислюється лише один раз.


### 13.2.6 Приклад: бінарний пошук

Тепер реалізуємо бінарний пошук рекурсивно:

```python
def recBinSearch(x, nums, low, high):
    if low > high:           # Ніде більше шукати — повертаємо -1
        return -1
    mid = (low + high) // 2
    item = nums[mid]
    if item == x:            # Знайдено — повертаємо індекс
        return mid
    elif x < item:           # Шукаємо у нижній половині
        return recBinSearch(x, nums, low, mid - 1)
    else:                    # Шукаємо у верхній половині
        return recBinSearch(x, nums, mid + 1, high)


def search(x, nums):
    return recBinSearch(x, nums, 0, len(nums) - 1)
```

Рекурсивна версія чіткіше показує структуру «розділяй і владарюй» бінарного пошуку, хоча цикловий варіант, мабуть, трохи швидший.


### 13.2.7 Рекурсія проти ітерації

Рекурсивні функції є узагальненням циклів. Все, що можна зробити циклом, можна зробити і рекурсивною функцією, і навпаки. Але деякі речі, що просто виражаються рекурсивно, досить складно виразити циклом.

Порівняння:

- **Факторіал і бінарний пошук:** цикловий і рекурсивний варіанти виконують практично однакові обчислення, циклові — трохи швидші.
- **Швидке піднесення до степеня:** рекурсивний варіант реалізує логарифмічний алгоритм, а циклова версія є лінійною — рекурсивний алгоритм явно кращий.

Однак є і погані рекурсивні алгоритми. Класичний приклад — рекурсивні числа Фібоначчі:

```python
def fib(n):
    if n < 3:
        return 1
    else:
        return fib(n - 1) + fib(n - 2)
```

Цей алгоритм виконує безліч дублюючих обчислень. `fib(4)` обчислюється двічі, `fib(3)` — тричі, `fib(2)` — п'ять разів. Циклова версія є лінійною і може легко обчислювати `fib(50000)`, а рекурсивна корисна лише до приблизно 30.

Висновок: **рекурсія — ще один інструмент** у вашому арсеналі. Іноді вона дає більш елегантне або ефективне розв'язання; іноді цикл краще; а іноді рекурсивна версія жахливо неефективна.


## 13.3 Алгоритми сортування

Задача сортування — взяти список і переставити його так, щоб значення розміщувалися в порядку зростання.


### 13.3.1 Наївне сортування: сортування вибором

Уявіть колоду карток із числами в довільному порядку. Один простий метод — знайти найменше значення і помістити його на початок, потім знайти найменше з тих, що залишилися, і помістити наступним і т.д.

Цей алгоритм називається **сортуванням вибором**. При перестановці найменшого елемента на початок потрібно зберегти поточний елемент на тому місці. Використовуємо одночасне присвоювання:

```python
def selSort(nums):
    # Сортує nums у порядку зростання
    n = len(nums)
    # Для кожної позиції (крім останньої)
    for bottom in range(n - 1):
        # Знайти найменший елемент у nums[bottom]...nums[n-1]
        mp = bottom                    # спочатку bottom — найменший
        for i in range(bottom + 1, n): # переглядаємо кожну позицію
            if nums[i] < nums[mp]:     # цей менший
                mp = i                 # запам'ятовуємо його індекс

        # Поміняти найменший елемент на bottom
        nums[bottom], nums[mp] = nums[mp], nums[bottom]
```

Зверніть увагу: замість зберігання мінімального значення `mp` запам'ятовує лише **позицію** мінімуму.

Сортування вибором легко написати і добре підходить для невеликих списків, але воно не є дуже ефективним.


### 13.3.2 «Розділяй і владарюй»: сортування злиттям

Підхід «розділяй і владарюй» дає значно ефективніший алгоритм. Уявіть, що ми вдвох сортуємо колоду карток. Ми могли б розділити її навпіл і кожен відсортував би свою половину. Потім нам просто потрібно **злити** дві відсортовані стопки.

Процес злиття простий: у кожної стопки найменше значення зверху. Яке з двох верхніх значень менше — те й іде першим у результат. Продовжуємо, поки одна зі стопок не закінчиться.

```python
def merge(lst1, lst2, lst3):
    # Зливає відсортовані lst1 і lst2 у lst3
    i1, i2, i3 = 0, 0, 0   # всі починаються спочатку
    n1, n2 = len(lst1), len(lst2)

    # Цикл, поки і lst1, і lst2 мають елементи
    while i1 < n1 and i2 < n2:
        if lst1[i1] < lst2[i2]:   # верхній з lst1 менший
            lst3[i3] = lst1[i1]   # копіюємо на поточну позицію lst3
            i1 = i1 + 1
        else:                      # верхній з lst2 менший
            lst3[i3] = lst2[i2]
            i2 = i2 + 1
        i3 = i3 + 1               # елемент додано до lst3, оновлюємо позицію

    # Тут lst1 або lst2 вичерпана. Один із циклів нижче завершить злиття.
    while i1 < n1:
        lst3[i3] = lst1[i1]
        i1 = i1 + 1
        i3 = i3 + 1
    while i2 < n2:
        lst3[i3] = lst2[i2]
        i2 = i2 + 1
        i3 = i3 + 1
```

Тепер сортування злиттям рекурсивно — ми можемо використовувати `mergeSort` для сортування двох менших списків:

```python
def mergeSort(nums):
    # Розміщує елементи nums у порядку зростання
    n = len(nums)
    # Нічого не робимо, якщо nums містить 0 або 1 елемент
    if n > 1:
        # Ділимо на два підсписки
        m = n // 2
        nums1, nums2 = nums[:m], nums[m:]
        # Рекурсивно сортуємо кожну частину
        mergeSort(nums1)
        mergeSort(nums2)
        # Зливаємо відсортовані частини назад у вихідний список
        merge(nums1, nums2, nums)
```

Базовий випадок: список з 0 або 1 елемента вже відсортований, нічого робити не потрібно.


### 13.3.3 Порівняння алгоритмів сортування

**Сортування вибором:** щоб знайти найменший елемент у списку з n елементів, потрібно n порівнянь; потім n−1, n−2 і т.д. Загальна кількість:

n + (n−1) + (n−2) + ... + 1 = n(n+1)/2

Кількість кроків пропорційна n². Це **квадратичний, або O(n²) алгоритм**. Подвоєння розміру списку учетверяє час.

**Сортування злиттям:** на кожному рівні злиття ми копіюємо n значень. Кількість рівнів — log₂n (бо ми ділимо список навпіл). Загальна робота: **n log₂n**. Це **алгоритм O(n log n)**.

Порівняння за часом на конкретному комп'ютері: для 3000 елементів сортування вибором займає понад 30 секунд, а сортування злиттям — близько ⅓ секунди. Сортування злиттям може відсортувати список із 20 000 елементів за менш ніж шість секунд; сортуванню вибором на це знадобляться приблизно 20 хвилин.

На невеликих списках (до ~50 елементів) сортування вибором може бути трохи швидшим через простішу реалізацію і менші накладні витрати.


## 13.4 Складні задачі

За допомогою підходу «розділяй і владарюй» ми розробили хороші алгоритми для пошуку і сортування. Однак не всі задачі мають ефективні розв'язання.


### 13.4.1 Вежа Ханоя

Одне дуже елегантне застосування рекурсивного розв'язання — знаменита математична головоломка, яку зазвичай називають **Вежею Ханоя** або Вежею Брахми.

Головоломка складається з трьох стрижнів і купи дисків різного радіуса, нанизаних на перший стрижень у вигляді піраміди. Потрібно перемістити всі диски на третій стрижень, дотримуючись правил:

1. Можна переміщати лише один диск за раз.
2. Диск можна ставити лише на один із трьох стрижнів.
3. Більший диск ніколи не можна класти на менший.

**Рекурсивне розв'язання:** щоб перемістити вежу висотою n:

1. Перемістити вежу висотою n−1 з джерела на тимчасовий стрижень.
2. Перемістити один диск (найбільший) з джерела на призначення.
3. Перемістити вежу висотою n−1 з тимчасового стрижня на призначення.

Базовий випадок: вежа висотою 1 — просто переміщуємо один диск.

```python
def moveTower(n, source, dest, temp):
    if n == 1:
        print("Перемістити диск з", source, "на", dest + ".")
    else:
        moveTower(n - 1, source, temp, dest)
        moveTower(1, source, dest, temp)
        moveTower(n - 1, temp, dest, source)


def hanoi(n):
    moveTower(n, "A", "C", "B")
```

```
>>> hanoi(3)
Перемістити диск з A на C.
Перемістити диск з A на B.
Перемістити диск з C на B.
Перемістити диск з A на C.
Перемістити диск з B на A.
Перемістити диск з B на C.
Перемістити диск з A на C.
```

Вирішення задачі Ханоя — «тривіальний» алгоритм, що складається лише з дев'яти рядків коду. Але наскільки він ефективний?

| кількість дисків | кроків у розв'язанні |
|---|---|
| 1 | 1 |
| 2 | 3 |
| 3 | 7 |
| 4 | 15 |
| 5 | 31 |

Взагалі розв'язання задачі розміром n потребує 2ⁿ − 1 кроків. Комп'ютерні вчені називають це **експоненційним алгоритмом**. Якби ченці починали з вежею лише з 64 дисків і переміщали по одному диску кожну секунду, 24 години на добу, без помилок — це зайняло б понад 580 мільярдів років.

Навіть попри те, що алгоритм для Вежі Ханоя легко виразити, він належить до класу **нерозв'язних на практиці задач** (intractable problems). Це задачі, що теоретично розв'язані, але потребують забагато обчислень для практичного розв'язання.


### 13.4.2 Проблема зупинки

Тепер розглянемо задачу, що є ще складнішою, ніж нерозв'язна на практиці — а саме **нерозв'язну в принципі**.

Уявіть, що ваш менеджер просить вас написати програму, яка аналізує вихідний код і виявляє, чи містить він нескінченний цикл. Специфікація:

**Програма: Аналізатор зупинки**  
Введення: файл програми Python і її вхідні дані.  
Виведення: «OK», якщо програма врешті зупиниться, або «НЕСПРАВНА», якщо програма зациклена.

Насправді це знаменита **проблема зупинки**, і вона нерозв'язна. Не існує жодного можливого алгоритму, що відповідає цій специфікації!

Як це довести? Методом **доведення від супротивного**: припустімо, що алгоритм існує, і покажемо, що це призводить до суперечності.

Припустімо, що є функція:

```python
def terminates(program, inputData):
    # program і inputData — обидва рядки
    # Повертає True, якщо program завершиться при запуску з inputData
```

Використовуючи цю функцію, напишемо таку програму:

```python
# turing.py

def terminates(program, inputData):
    # program і inputData — обидва рядки
    # Повертає True, якщо program завершиться при запуску з inputData

def main():
    # Зчитати програму зі стандартного введення
    lines = []
    print("Введіть програму (введіть 'done' для завершення).")
    line = input("")
    while line != "done":
        lines.append(line)
        line = input("")
    testProg = "\n".join(lines)

    # Якщо програма зупиняється при собі як введенні — зайти в нескінченний цикл
    if terminates(testProg, testProg):
        while True:
            pass  # оператор pass нічого не робить

main()
```

Оператор `pass` нічого не робить; якщо `terminates` повертає `True`, `turing.py` входить у нескінченний цикл.

Тепер поставимо ключове питання: що станеться, якщо ми запустимо `turing.py` і, коли попросять ввести програму, введемо вміст самого `turing.py`?

- Якщо `terminates` повертає `True` (тобто `turing.py` зупиниться при собі як введенні) → `turing.py` входить у нескінченний цикл → він не зупиняється! **Суперечність.**
- Якщо `terminates` повертає `False` (тобто `turing.py` зациклюється) → `turing.py` одразу завершується → він зупиняється! **Суперечність.**

Обидва варіанти призводять до суперечності. Отже, припущення про існування функції `terminates` хибне. Жодного алгоритму для розв'язання проблеми зупинки не існує.

Це доведення первісно розробив **Алан Тьюрінг** — британський математик, якого багато хто вважає «батьком комп'ютерних наук».


### 13.4.3 Висновок

Сподіваємося, цей розділ дав вам уявлення про те, що таке комп'ютерні науки. Як показали приклади, комп'ютерні науки — це значно більше, ніж «просто» програмування. Найважливіший комп'ютер для будь-якого комп'ютерного фахівця — все ще той, що знаходиться між вухами.


## 13.5 Підсумок розділу

- Однією з ключових підгалузей комп'ютерних наук є аналіз алгоритмів. Комп'ютерні вчені аналізують часову ефективність алгоритму, розглядаючи кількість кроків відносно розміру введення.

- Пошук — процес знаходження конкретного елемента в колекції. Лінійний пошук переглядає колекцію з початку до кінця і потребує часу, лінійно пропорційного розміру колекції. Якщо колекція відсортована, можна використати бінарний пошук, що потребує часу, пропорційного log від розміру колекції.

- Бінарний пошук — приклад підходу «розділяй і владарюй» до розробки алгоритмів. Цей підхід нерідко дає ефективні розв'язання.

- Визначення або функція є рекурсивною, якщо вона посилається на себе. Щоб бути коректним, рекурсивне визначення повинно відповідати двом властивостям:
  1. Повинен бути один або більше базових випадків, що не потребують рекурсії.
  2. Всі ланцюжки рекурсії повинні врешті досягати базового випадку.

- Послідовності можна розглядати як рекурсивні структури, що містять перший елемент і решту послідовності. Рекурсивні функції можна писати, слідуючи цьому підходу.

- Рекурсія є більш загальною, ніж ітерація. Вибір між рекурсією і циклом включає розгляд ефективності та елегантності.

- Сортування — розміщення колекції в порядку. Сортування вибором потребує часу, пропорційного квадрату розміру колекції. Сортування злиттям — алгоритм «розділяй і владарюй», що може відсортувати колекцію за час n log n.

- Задачі, що є розв'язними теоретично, але не на практиці, називаються нерозв'язними на практиці (intractable). Розв'язання знаменитої Вежі Ханої може бути виражено простим рекурсивним алгоритмом, але цей алгоритм нерозв'язний на практиці.

- Деякі задачі нерозв'язні в принципі. Проблема зупинки — один приклад нерозв'язної задачі.

- Варто розглянути можливість стати комп'ютерним ученим.


## 13.6 Вправи

### Питання для повторення

**Правда чи хиба**

1. Лінійний пошук потребує кількості кроків, пропорційної розміру списку, що переглядається.
2. Оператор Python `in` виконує бінарний пошук.
3. Бінарний пошук — це алгоритм O(n log n).
4. Кількість разів, коли n можна ділити на 2, дорівнює exp(n).
5. Всі правильні рекурсивні визначення повинні мати рівно один нерекурсивний базовий випадок.
6. Послідовність можна розглядати як рекурсивну колекцію даних.
7. Слово довжиною n має n! анаграм.
8. Цикли є більш загальними, ніж рекурсія.
9. Сортування злиттям — приклад алгоритму O(n log n).
10. Експоненційні алгоритми загалом вважаються нерозв'язними на практиці.

   Алгоритм: якщо степінь 0 — малювати відрізок заданої довжини; інакше ділимо довжину і степінь на 3 і 1 відповідно, малюємо чотири кривих менших степенів, між якими черепаха повертає на певні кути.

9. Інша цікава рекурсивна крива (дивіться попередню задачу) — C-крива. Вона утворюється так само, як крива Коха, але кожен відрізок замінюється двома відрізками довжиною `length/√2`, що утворюють кут 90 градусів. Написати програму для малювання C-кривої.

10. Автоматизовані програми перевірки орфографії аналізують документи і виявляють слова, що можуть бути написані з помилкою. Написати програму для перевірки орфографії текстового файлу. Для цього вам знадобиться великий файл англійських слів в алфавітному порядку (можна пошукати файл `words` у `/usr/dict` або в Інтернеті).  
    Ваша програма запрошує файл для аналізу, а потім намагається знайти кожне слово у файлі за допомогою бінарного пошуку. Якщо слово не знайдено у словнику, виводить його на екран як потенційно неправильне.

11. Написати програму для розв'язання задач з анаграмами. Вам знадобиться великий словник англійських слів (дивіться попередню задачу). Користувач вводить переставлене слово, а ваша програма генерує всі анаграми цього слова і перевіряє, які з них є в словнику. Анаграми, що є в словнику, виводяться як розв'язання головоломки.
