Одна з найкращих якостей Python — його послідовність. Попрацювавши з Python певний час, ви починаєте робити обґрунтовані, правильні здогадки щодо можливостей, з якими ще не стикалися.

Однак якщо ви вивчали іншу об'єктно-орієнтовану мову до Python, вам може здатися дивним писати `len(collection)` замість `collection.len()`. Ця уявна дивина — лише верхівка айсберга, який, якщо правильно його розуміти, є ключем до всього, що ми називаємо «пітонічним» (Pythonic). Айсберг називається моделлю даних Python, і це той API, що дозволяє нашим власним об'єктам добре взаємодіяти з найбільш ідіоматичними можливостями мови.

Модель даних можна сприймати як опис Python як фреймворку. Вона формалізує інтерфейси будівельних блоків самої мови — таких як послідовності, функції, ітератори, coroutine, класи, менеджери контексту тощо.

Використовуючи фреймворк, ми витрачаємо багато часу на написання методів, які викликає сам фреймворк. Те саме відбувається, коли ми використовуємо модель даних Python для побудови нових класів. Інтерпретатор Python викликає спеціальні методи для виконання базових операцій над об'єктами, що часто ініціюються спеціальним синтаксисом. Імена спеціальних методів завжди записуються з подвійними підкресленнями на початку і в кінці. Наприклад, синтаксис `obj[key]` підтримується спеціальним методом `__getitem__`. Щоб обчислити `my_collection[key]`, інтерпретатор викликає `my_collection.__getitem__(key)`.

Ми реалізуємо спеціальні методи, коли хочемо, щоб наші об'єкти підтримували та взаємодіяли з фундаментальними конструкціями мови, зокрема:

- колекціями;
- доступом до атрибутів;
- ітерацією (зокрема асинхронною ітерацією з використанням `async for`);
- перевантаженням операторів;
- викликом функцій і методів;
- рядковим представленням і форматуванням;
- асинхронним програмуванням з використанням `await`;
- створенням і знищенням об'єктів;
- керованими контекстами з використанням інструкцій `with` або `async with`.

---

> **Магічні методи та dunder**
>
> Термін «магічний метод» — це сленгова назва спеціального методу, але як говорити про конкретний метод, наприклад `__getitem__`? Я навчився казати «dunder-getitem» від автора і викладача Стіва Голдена. «Dunder» — скорочення від «double underscore before and after» (подвійне підкреслення до і після). Саме тому спеціальні методи також відомі як dunder-методи. Розділ «Lexical Analysis» у *The Python Language Reference* застерігає: «Будь-яке використання імен виду `__*__` в будь-якому контексті, що не відповідає явно задокументованому використанню, може призвести до поломки без попередження».

## Пітонічна колода карт

Приклад 1-1 простий, але вражає своєю ємністю. Він демонструє силу реалізації лише двох спеціальних методів — `__getitem__` і `__len__`.

**Приклад 1-1.** Колода як послідовність гральних карт

```python
import collections

Card = collections.namedtuple('Card', ['rank', 'suit'])

class FrenchDeck:
    ranks = [str(n) for n in range(2, 11)] + list('JQKA')
    suits = 'spades diamonds clubs hearts'.split()

    def __init__(self):
        self._cards = [Card(rank, suit) for suit in self.suits
                                        for rank in self.ranks]

    def __len__(self):
        return len(self._cards)

    def __getitem__(self, position):
        return self._cards[position]
```

Перше, на що слід звернути увагу, — це використання `collections.namedtuple` для побудови простого класу, що представляє окремі картки. Ми використовуємо `namedtuple` для побудови класів об'єктів, які є лише наборами атрибутів без власних методів, подібно до запису бази даних. У прикладі ми застосовуємо його для зручного представлення карток у колоді, як показано в сеансі консолі:

```python
>>> beer_card = Card('7', 'diamonds')
>>> beer_card
Card(rank='7', suit='diamonds')
```

Але суть цього прикладу — клас `FrenchDeck`. Він короткий, але дієвий. По-перше, як і будь-яка стандартна колекція Python, колода відповідає на функцію `len()`, повертаючи кількість карток у ній:

```python
>>> deck = FrenchDeck()
>>> len(deck)
52
```

Читати конкретні картки з колоди — скажімо, першу або останню — легко завдяки методу `__getitem__`:

```python
>>> deck[0]
Card(rank='2', suit='spades')
>>> deck[-1]
Card(rank='A', suit='hearts')
```

Чи потрібно нам створювати метод для вибору випадкової картки? Ні. Python вже має функцію для отримання випадкового елемента з послідовності: `random.choice`. Ми можемо використати її з екземпляром колоди:

```python
>>> from random import choice
>>> choice(deck)
Card(rank='3', suit='hearts')
>>> choice(deck)
Card(rank='K', suit='spades')
>>> choice(deck)
Card(rank='2', suit='clubs')
```

Ми щойно побачили дві переваги використання спеціальних методів для взаємодії з моделлю даних Python:

- Користувачам ваших класів не потрібно запам'ятовувати довільні імена методів для стандартних операцій. («Як отримати кількість елементів? Це `.size()`, `.length()` чи щось інше?»)
- Простіше скористатися багатою стандартною бібліотекою Python і уникнути повторного винаходу колеса — як-от функція `random.choice`.

Але це ще не все.

Оскільки наш `__getitem__` делегує оператору `[]` на `self._cards`, наша колода автоматично підтримує зрізи. Ось як ми дивимося на три верхні картки нової колоди, а потім вибираємо лише тузів, починаючи з індексу 12 і пропускаючи по 13 карток:

```python
>>> deck[:3]
[Card(rank='2', suit='spades'), Card(rank='3', suit='spades'),
Card(rank='4', suit='spades')]
>>> deck[12::13]
[Card(rank='A', suit='spades'), Card(rank='A', suit='diamonds'),
Card(rank='A', suit='clubs'), Card(rank='A', suit='hearts')]
```

Лише реалізувавши спеціальний метод `__getitem__`, наша колода також стає ітерабельною:

```python
>>> for card in deck:  # doctest: +ELLIPSIS
...     print(card)
Card(rank='2', suit='spades')
Card(rank='3', suit='spades')
Card(rank='4', suit='spades')
...
```

Ми також можемо ітерувати колоду у зворотному напрямку:

```python
>>> for card in reversed(deck):  # doctest: +ELLIPSIS
...     print(card)
Card(rank='A', suit='hearts')
Card(rank='K', suit='hearts')
Card(rank='Q', suit='hearts')
...
```

---

> **Ellipsis у doctests**
>
> Де можливо, я витягував лістинги консолі Python у цій книзі з doctest для забезпечення точності. Коли виведення було надто довгим, пропущена частина позначалася трьома крапками (`...`), як в останньому рядку попереднього коду. У таких випадках я використовував директиву `# doctest: +ELLIPSIS`, щоб doctest пройшов успішно. Якщо ви випробовуєте ці приклади в інтерактивній консолі, коментарі doctest можна повністю опустити.

---

Ітерація нерідко є неявною. Якщо колекція не має методу `__contains__`, оператор `in` виконує послідовне сканування. Це саме наш випадок: `in` працює з нашим класом `FrenchDeck`, тому що він є ітерабельним. Переконайтеся:

```python
>>> Card('Q', 'hearts') in deck
True
>>> Card('7', 'beasts') in deck
False
```

А що з сортуванням? Поширена система ранжування карток — за рангом (з тузами як найвищими), потім за мастю в порядку: піки (найвища), червоне серце, бубни і трефи (найнижча). Ось функція, яка ранжує картки за цим правилом, повертаючи 0 для двійки треф і 51 для туза пік:

```python
suit_values = dict(spades=3, hearts=2, diamonds=1, clubs=0)

def spades_high(card):
    rank_value = FrenchDeck.ranks.index(card.rank)
    return rank_value * len(suit_values) + suit_values[card.suit]
```

За наявності `spades_high` ми можемо тепер вивести нашу колоду в порядку зростання рангу:

```python
>>> for card in sorted(deck, key=spades_high):  # doctest: +ELLIPSIS
...     print(card)
Card(rank='2', suit='clubs')
Card(rank='2', suit='diamonds')
Card(rank='2', suit='hearts')
... (46 карток пропущено)
Card(rank='A', suit='diamonds')
Card(rank='A', suit='hearts')
Card(rank='A', suit='spades')
```

Хоча `FrenchDeck` неявно наслідує від класу `object`, більшість його функціональності не успадкована, а виникає завдяки використанню моделі даних і композиції. Реалізувавши спеціальні методи `__len__` і `__getitem__`, наш `FrenchDeck` поводиться як стандартна послідовність Python, що дозволяє йому скористатися основними можливостями мови (наприклад, ітерацією і зрізами) та стандартною бібліотекою, як показано в прикладах із `random.choice`, `reversed` і `sorted`. Завдяки композиції реалізації `__len__` і `__getitem__` можуть делегувати всю роботу об'єкту-списку `self._cards`.

---

> **А що з перемішуванням?**
>
> У нинішньому вигляді `FrenchDeck` не можна перемішати, бо він є незмінним: картки та їхні позиції не можна змінити, не порушивши інкапсуляцію і не звернувшись до атрибута `_cards` безпосередньо. У Розділі 13 ми виправимо це, додавши однорядковий метод `__setitem__`.

## Як використовуються спеціальні методи

Перше, що слід знати про спеціальні методи: вони призначені для виклику інтерпретатором Python, а не вами безпосередньо. Ви не пишете `my_object.__len__()`. Ви пишете `len(my_object)`, і якщо `my_object` є екземпляром класу, визначеного користувачем, Python викликає реалізований вами метод `__len__`.

Однак для вбудованих типів, як-от `list`, `str`, `bytearray`, або розширень — масивів NumPy — інтерпретатор робить скорочення. Змінні за розміром колекції Python, написані на C, містять C-структуру[^2] під назвою `PyVarObject`, яка має поле `ob_size`, що зберігає кількість елементів у колекції. Тому, якщо `my_object` є екземпляром одного з таких вбудованих типів, `len(my_object)` просто зчитує значення поля `ob_size`, і це набагато швидше, ніж виклик методу.

[^2]: C-структура — це запис із іменованими полями.

Найчастіше виклик спеціального методу є неявним. Наприклад, інструкція `for i in x:` насправді спричиняє виклик `iter(x)`, який, своєю чергою, може викликати `x.__iter__()`, якщо такий метод є, або скористатися `x.__getitem__()`, як у прикладі з `FrenchDeck`.

Зазвичай ваш код не повинен містити багато прямих викликів спеціальних методів. Якщо ви не займаєтеся активно метапрограмуванням, вам слід частіше реалізовувати спеціальні методи, ніж явно їх викликати. Єдиний спеціальний метод, який код користувача нерідко викликає напряму, — це `__init__`, для виклику ініціалізатора суперкласу у власній реалізації `__init__`.

Якщо потрібно викликати спеціальний метод, зазвичай краще викликати відповідну вбудовану функцію (наприклад, `len`, `iter`, `str` тощо). Ці вбудовані функції викликають відповідний спеціальний метод, але нерідко надають додаткові послуги і — для вбудованих типів — є швидшими за виклики методів. Дивіться, наприклад, «Using iter with a Callable» на стор. 598 у Розділі 17.

У наступних розділах ми розглянемо найважливіші способи застосування спеціальних методів:

- емуляцію числових типів;
- рядкове представлення об'єктів;
- булеве значення об'єкта;
- реалізацію колекцій.

## Емуляція числових типів

Кілька спеціальних методів дозволяють об'єктам користувача реагувати на оператори на зразок `+`. Ми розглянемо це докладніше в Розділі 16, але тут наша мета — додатково проілюструвати використання спеціальних методів на іншому простому прикладі.

Ми реалізуємо клас для представлення двовимірних векторів — тобто евклідових векторів, подібних до тих, що використовуються в математиці та фізиці (дивіться Рисунок 1-1).

> Вбудований тип `complex` можна використовувати для представлення двовимірних векторів, але наш клас можна розширити для представлення n-вимірних векторів. Це ми зробимо в Розділі 17.

**Рисунок 1-1.** Приклад додавання двовимірних векторів: `Vector(2, 4) + Vector(2, 1)` дає `Vector(4, 5)`.

Ми почнемо з проєктування API для такого класу, написавши змодельований сеанс консолі, який пізніше можна використати як doctest. Наступний фрагмент перевіряє додавання векторів, зображене на Рисунку 1-1:

```python
>>> v1 = Vector(2, 4)
>>> v2 = Vector(2, 1)
>>> v1 + v2
Vector(4, 5)
```

Зверніть увагу: оператор `+` повертає новий `Vector`, відображений у зручному форматі в консолі.

Вбудована функція `abs` повертає абсолютне значення цілих чисел і чисел з рухомою крапкою, а також модуль комплексних чисел — тому для узгодженості наш API теж використовує `abs` для обчислення модуля вектора:

```python
>>> v = Vector(3, 4)
>>> abs(v)
5.0
```

Ми також можемо реалізувати оператор `*` для скалярного множення (тобто множення вектора на число, щоб отримати новий вектор того самого напрямку зі збільшеним модулем):

```python
>>> v * 3
Vector(9, 12)
>>> abs(v * 3)
15.0
```

Приклад 1-2 — клас `Vector`, що реалізує описані операції за допомогою спеціальних методів `__repr__`, `__abs__`, `__add__` і `__mul__`.

**Приклад 1-2.** Простий клас двовимірного вектора

```python
"""
vector2d.py: спрощений клас, що демонструє деякі спеціальні методи

Він спрощений з дидактичних міркувань. У ньому відсутня належна
обробка помилок, особливо в методах ``__add__`` і ``__mul__``.
Цей приклад значно розширюється в подальших розділах книги.

Додавання::

    >>> v1 = Vector(2, 4)
    >>> v2 = Vector(2, 1)
    >>> v1 + v2
    Vector(4, 5)

Абсолютне значення::

    >>> v = Vector(3, 4)
    >>> abs(v)
    5.0

Скалярне множення::

    >>> v * 3
    Vector(9, 12)
    >>> abs(v * 3)
    15.0

"""

import math

class Vector:

    def __init__(self, x=0, y=0):
        self.x = x
        self.y = y

    def __repr__(self):
        return f'Vector({self.x!r}, {self.y!r})'

    def __abs__(self):
        return math.hypot(self.x, self.y)

    def __bool__(self):
        return bool(abs(self))

    def __add__(self, other):
        x = self.x + other.x
        y = self.y + other.y
        return Vector(x, y)

    def __mul__(self, scalar):
        return Vector(self.x * scalar, self.y * scalar)
```

Ми реалізували п'ять спеціальних методів на додаток до знайомого `__init__`. Зверніть увагу: жоден із них не викликається безпосередньо всередині класу або в типовому використанні класу, проілюстрованому doctests. Як уже згадувалося, інтерпретатор Python є єдиним, хто регулярно викликає більшість спеціальних методів.

Приклад 1-2 реалізує два оператори: `+` і `*`, щоб показати базове використання `__add__` і `__mul__`. В обох випадках методи створюють і повертають новий екземпляр `Vector` та не змінюють жодного з операндів — `self` і `other` лише зчитуються. Це очікувана поведінка інфіксних операторів: створювати нові об'єкти і не торкатися своїх операндів. Я докладніше зупинюся на цьому в Розділі 16.

> Як реалізовано в Прикладі 1-2, можна множити `Vector` на число, але не число на `Vector`, що порушує комутативність скалярного множення. Ми виправимо це за допомогою спеціального методу `__rmul__` у Розділі 16.

У наступних розділах ми обговоримо інші спеціальні методи у `Vector`.

## Рядкове представлення

Спеціальний метод `__repr__` викликається вбудованою функцією `repr` для отримання рядкового представлення об'єкта з метою інспекції. Без власного `__repr__` консоль Python відображатиме екземпляр `Vector` як `<Vector object at 0x10e100070>`.

Інтерактивна консоль і відлагоджувач викликають `repr` для результатів обчислених виразів, так само як замінник `%r` у класичному форматуванні з оператором `%` та поле перетворення `!r` у синтаксисі нових рядків форматування, що використовується в f-рядках та методі `str.format`.

Зверніть увагу: f-рядок у нашому `__repr__` використовує `!r`, щоб отримати стандартне представлення атрибутів, що відображаються. Це хороша практика, оскільки вона показує принципову різницю між `Vector(1, 2)` і `Vector('1', '2')` — останній не працюватиме в контексті цього прикладу, тому що аргументи конструктора мають бути числами, а не рядками `str`.

Рядок, повернутий `__repr__`, має бути однозначним і, якщо можливо, відповідати вихідному коду, необхідному для відтворення представленого об'єкта. Саме тому наше представлення `Vector` виглядає як виклик конструктора класу (наприклад, `Vector(3, 4)`).

На відміну від нього, `__str__` викликається вбудованою функцією `str()` і неявно використовується функцією `print`. Він має повертати рядок, придатний для відображення кінцевим користувачам.

Іноді той самий рядок, що повертається `__repr__`, є зручним для користувача, і вам не потрібно реалізовувати `__str__`, оскільки реалізація, успадкована від класу `object`, викликає `__repr__` як запасний варіант. Приклад 5-2 — один із кількох прикладів у цій книзі з власним `__str__`.

> Програмісти з попереднім досвідом роботи в мовах, що мають метод `toString`, зазвичай реалізують `__str__`, а не `__repr__`. Якщо ви реалізуєте в Python лише один із цих спеціальних методів, обирайте `__repr__`.
>
> «What is the difference between `__str__` and `__repr__` in Python?» — запитання на Stack Overflow з чудовими відповідями від Python-розробників Алекса Мартеллі та Мартейна Пітерса.

## Булеве значення власного типу

Хоча Python має тип `bool`, він приймає будь-який об'єкт у булевому контексті, наприклад у виразі, що керує інструкцією `if` або `while`, або як операнди для `and`, `or` і `not`. Щоб визначити, чи є значення `x` істинним або хибним, Python застосовує `bool(x)`, що повертає `True` або `False`.

За замовчуванням екземпляри класів, визначених користувачем, вважаються істинними, якщо не реалізовано ні `__bool__`, ні `__len__`. По суті, `bool(x)` викликає `x.__bool__()` і використовує результат. Якщо `__bool__` не реалізовано, Python намагається викликати `x.__len__()`, і якщо той повертає нуль, `bool` повертає `False`. В іншому разі `bool` повертає `True`.

Наша реалізація `__bool__` концептуально проста: вона повертає `False`, якщо модуль вектора дорівнює нулю, інакше — `True`. Ми перетворюємо модуль на булеве значення за допомогою `bool(abs(self))`, тому що від `__bool__` очікується повернення булевого значення. Поза методами `__bool__` рідко потрібно явно викликати `bool()`, адже будь-який об'єкт можна використовувати в булевому контексті.

Зверніть увагу: спеціальний метод `__bool__` дозволяє вашим об'єктам дотримуватися правил перевірки істинності, визначених у розділі «Built-in Types» документації *The Python Standard Library*.

> Більш швидка реалізація `Vector.__bool__`:
>
> ```python
> def __bool__(self):
>     return bool(self.x or self.y)
> ```
>
> Вона важча для читання, але дозволяє уникнути зайвих обчислень через `abs`, `__abs__`, зведення в квадрат і добування кореня. Явне перетворення на `bool` необхідне, тому що `__bool__` має повертати булеве значення, а `or` повертає один із операндів як є: `x or y` обчислюється як `x`, якщо той є істинним, інакше результат — `y`, яким би він не був.

## Collection API

Рисунок 1-2 документує інтерфейси найважливіших типів колекцій у мові. Усі класи на діаграмі є ABCs — абстрактними базовими класами. ABCs і модуль `collections.abc` розглядаються в Розділі 13. Мета цього короткого розділу — дати панорамний вигляд найважливіших інтерфейсів колекцій Python, показавши, як вони будуються зі спеціальних методів.

**Рисунок 1-2.** UML-діаграма класів із фундаментальними типами колекцій. Імена методів курсивом є абстрактними, тому вони мають бути реалізовані конкретними підкласами, такими як `list` і `dict`. Решта методів мають конкретні реалізації, тому підкласи можуть їх успадковувати.

Кожен із верхніх ABCs має єдиний спеціальний метод. ABC `Collection` (новий у Python 3.6) об'єднує три основні інтерфейси, які має реалізовувати кожна колекція:

- `Iterable` — для підтримки `for`, розпакування та інших форм ітерації;
- `Sized` — для підтримки вбудованої функції `len`;
- `Container` — для підтримки оператора `in`.

Python не вимагає, щоб конкретні класи фактично наслідували від будь-якого з цих ABCs. Будь-який клас, що реалізує `__len__`, задовольняє інтерфейс `Sized`.

Три дуже важливі спеціалізації `Collection`:

- `Sequence` — формалізує інтерфейс вбудованих типів, таких як `list` і `str`;
- `Mapping` — реалізований `dict`, `collections.defaultdict` тощо;
- `Set` — інтерфейс вбудованих типів `set` і `frozenset`.

Лише `Sequence` є `Reversible`, тому що послідовності підтримують довільне впорядкування свого вмісту, тоді як відображення та множини — ні.

> Починаючи з Python 3.7, тип `dict` офіційно є «впорядкованим», але це лише означає, що зберігається порядок вставки ключів. Ви не можете довільно перерозподіляти ключі в `dict`.

Усі спеціальні методи в ABC `Set` реалізують інфіксні оператори. Наприклад, `a & b` обчислює перетин множин `a` і `b` та реалізований у спеціальному методі `__and__`.

Наступні два розділи докладно розглядатимуть послідовності, відображення і множини зі стандартної бібліотеки.

Тепер розглянемо основні категорії спеціальних методів, визначених у моделі даних Python.

## Чому len не є методом

Я поставив це запитання розробнику ядра Реймонду Геттінґеру в 2013 році, і ключем до його відповіді стала цитата з «Дзену Python»: «Практичність перевершує чистоту» (practicality beats purity). У розділі «Як використовуються спеціальні методи» (стор. 8) я описав, як `len(x)` виконується дуже швидко, коли `x` є екземпляром вбудованого типу. Для вбудованих об'єктів CPython жоден метод не викликається: довжина просто зчитується з поля C-структури. Отримання кількості елементів у колекції — поширена операція, яка має ефективно працювати для таких базових і різноманітних типів, як `str`, `list`, `memoryview` тощо.

Іншими словами, `len` не викликається як метод, тому що отримує спеціальну обробку як частина моделі даних Python, так само як `abs`. Але завдяки спеціальному методу `__len__` ви також можете змусити `len` працювати з власними об'єктами. Це справедливий компроміс між потребою в ефективних вбудованих об'єктах і послідовністю мови.

Ще з «Дзену Python»: «Особливі випадки недостатньо особливі, щоб порушувати правила» (Special cases aren't special enough to break the rules).

> Якщо сприймати `abs` і `len` як унарні оператори, ви, мабуть, легше пробачите їм функціональний вигляд, на противагу синтаксису виклику методу, якого можна очікувати в об'єктно-орієнтованій мові. Насправді мова ABC — прямий предок Python, що заклав багато його можливостей, — мала оператор `#`, який був еквівалентом `len` (ви б писали `#s`). У ролі інфіксного оператора, записаного як `x#s`, він підраховував кількість входжень `x` у `s`, що в Python отримується як `s.count(x)` для будь-якої послідовності `s`.

## Підсумок розділу

Реалізуючи спеціальні методи, ваші об'єкти можуть поводитися як вбудовані типи, забезпечуючи виразний стиль програмування, який спільнота вважає пітонічним.

Базова вимога до об'єкта Python — надавати зручні рядкові представлення себе: одне для відлагодження і протоколювання, інше для презентації кінцевим користувачам. Саме для цього в моделі даних існують спеціальні методи `__repr__` і `__str__`.

Емуляція послідовностей, показана на прикладі `FrenchDeck`, — одне з найпоширеніших застосувань спеціальних методів. Наприклад, бібліотеки баз даних нерідко повертають результати запитів, загорнуті в колекції, подібні до послідовностей. Максимальне використання існуючих типів послідовностей — тема Розділу 2. Реалізація власних послідовностей буде розглянута в Розділі 12, де ми створимо багатовимірне розширення класу `Vector`.

Завдяки перевантаженню операторів Python пропонує багатий вибір числових типів — від вбудованих до `decimal.Decimal` і `fractions.Fraction`, усі з підтримкою інфіксних арифметичних операторів. Бібліотеки наукових обчислень NumPy підтримують інфіксні оператори для матриць і тензорів. Реалізація операторів — зокрема зворотних операторів і скорочених присвоєнь — буде показана в Розділі 16 через вдосконалення прикладу з `Vector`.

Використання і реалізація більшості спеціальних методів, що залишилися в моделі даних Python, розглядаються протягом усієї книги.

### Модель даних чи об'єктна модель?

Те, що документація Python називає «моделлю даних Python», більшість авторів назвала б «об'єктною моделлю Python». *Python in a Nutshell*, 3rd ed. Мартеллі, Равенскрофт і Голдена та *Python Essential Reference*, 4th ed. Девіда Бізлі — найкращі книги, що охоплюють модель даних Python, — але в них вона іменується «об'єктною моделлю». За визначенням у Вікіпедії, перше значення «об'єктної моделі» таке: «Властивості об'єктів загалом у конкретній мові комп'ютерного програмування». Саме про це йдеться в моделі даних Python. У цій книзі я використовуватиму «модель даних», тому що документація надає перевагу цьому терміну при посиланні на об'єктну модель Python і тому що це назва найбільш релевантного для наших обговорень розділу *The Python Language Reference*.

### Методи для маґлів

*The Original Hacker's Dictionary* визначає «магічне» як «те, що ще не пояснено або надто складне для пояснення» чи «можливість, що не афішується широко і дозволяє робити щось, що інакше неможливо».

Спільнота Ruby називає свої еквіваленти спеціальних методів «магічними методами». Чимало людей у спільноті Python теж прийняли цей термін. Я вважаю, що спеціальні методи — це протилежність магії. Python і Ruby надають своїм користувачам багатий протокол метаоб'єктів, що є повністю задокументованим і дозволяє таким маґлам, як ви і я, емулювати багато можливостей, доступних розробникам ядра, що пишуть інтерпретатори цих мов.

Для порівняння розглянемо Go. Деякі об'єкти в цій мові мають «магічні» можливості в тому сенсі, що ми не можемо їх емулювати у власних типах, визначених користувачем. Наприклад, масиви Go, рядки та відображення підтримують використання дужок для доступу до елементів, як у `a[i]`. Але немає жодного способу змусити нотацію `[]` працювати з новим типом колекції, який ви визначаєте. Ба більше, у Go немає концепції ітерабельного інтерфейсу або об'єкта-ітератора на рівні користувача, тому синтаксис `for/range` обмежений підтримкою п'яти «магічних» вбудованих типів, зокрема масивів, рядків і відображень.

Можливо, в майбутньому розробники Go вдосконалять його протокол метаоб'єктів. Але наразі він значно обмеженіший, ніж те, що є у нас у Python або Ruby.

### Метаоб'єкти

*The Art of the Metaobject Protocol* (AMOP) — моя улюблена назва комп'ютерної книги. Але я згадую її тому, що термін «протокол метаоб'єктів» корисний для роздумів про модель даних Python і подібні можливості в інших мовах. Частина «метаоб'єкт» відноситься до об'єктів, що є будівельними блоками самої мови. У цьому контексті «протокол» є синонімом «інтерфейсу». Отже, протокол метаоб'єктів — це вишукана назва об'єктної моделі: API для основних конструкцій мови.

Багатий протокол метаоб'єктів дозволяє розширювати мову для підтримки нових парадигм програмування. Ґреґор Кічалес, перший автор книги AMOP, пізніше став піонером аспектно-орієнтованого програмування та початковим автором AspectJ — розширення Java, що реалізує цю парадигму. Аспектно-орієнтоване програмування набагато простіше реалізувати в динамічній мові на кшталт Python, і деякі фреймворки це роблять. Найважливіший приклад — `zope.interface`, частина фреймворку, на якому побудовано систему керування контентом Plone.
