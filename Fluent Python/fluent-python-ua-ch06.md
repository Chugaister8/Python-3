## Змінні — це не ящики

У 1997 році я відвідував літні курси з Java в MIT. Професор Лінн Стейн[^1] зауважила, що звична метафора «змінні як ящики» насправді заважає розумінню посилальних змінних в об'єктно-орієнтованих мовах. Змінні Python схожі на посилальні змінні в Java; краща метафора — думати про змінні як про позначки з іменами, прикріплені до об'єктів. Наступний приклад і рисунок допоможуть зрозуміти, чому.

[^1]: Лінн Андреа Стейн — нагороджений педагог у галузі інформатики, що нині викладає в Olin College of Engineering.

Приклад 6-1 — це проста взаємодія, яку ідея «змінні як ящики» не може пояснити.

**Приклад 6-1.** Змінні `a` і `b` містять посилання на один і той самий список, а не копії списку

```python
>>> a = [1, 2, 3]
>>> b = a
>>> a.append(4)
>>> b
[1, 2, 3, 4]
```

Створити список `[1, 2, 3]` і прив'язати змінну `a` до нього. Прив'язати змінну `b` до того самого значення, на яке посилається `a`. Змінити список, на який посилається `a`, додавши ще один елемент. Ефект можна побачити через змінну `b`. Якщо ми думаємо про `b` як про ящик, що зберігав копію `[1, 2, 3]` із ящика `a`, ця поведінка не має сенсу.

**Рисунок 6-1.** Якщо ви уявляєте змінні як ящики, ви не можете зрозуміти присвоєння в Python; замість цього думайте про змінні як про стікери, і Приклад 6-1 стає легко пояснити.

Таким чином, інструкція `b = a` не копіює вміст ящика `a` до ящика `b`. Вона прикріплює позначку `b` до об'єкта, який вже має позначку `a`.

Проф. Стейн також говорила про присвоєння дуже обдумано. Наприклад, говорячи про об'єкт-гойдалку в симуляції, вона казала: «Змінна `s` присвоюється гойдалці», але ніколи «Гойдалка присвоюється змінній `s`». З посилальними змінними набагато зрозуміліше говорити, що змінна присвоюється об'єкту, а не навпаки. Зрештою, об'єкт створюється до присвоєння.

Оскільки дієслово «присвоювати» використовується в суперечливих значеннях, корисною альтернативою є «прив'язувати»: інструкція присвоєння Python `x = …` прив'язує ім'я `x` до об'єкта, створеного або отриманого в правій частині. І об'єкт має існувати перш ніж до нього можна прив'язати ім'я, як доводить Приклад 6-2.

**Приклад 6-2.** Змінні прив'язуються до об'єктів лише після створення об'єктів

```python
>>> class Gizmo:
...     def __init__(self):
...         print(f'Gizmo id: {id(self)}')
...
>>> x = Gizmo()
Gizmo id: 4301489152
>>> y = Gizmo() * 10
Gizmo id: 4301489432
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
TypeError: unsupported operand type(s) for *: 'Gizmo' and 'int'
>>>
>>> dir()
['Gizmo', '__builtins__', '__doc__', '__loader__', '__name__',
'__package__', '__spec__', 'x']
```

Виведення `Gizmo id: …` є побічним ефектом створення екземпляра `Gizmo`. Множення екземпляра `Gizmo` призведе до винятку. Ось доказ того, що другий `Gizmo` фактично був інстанційований до спроби множення. Але змінна `y` ніколи не була створена, бо виняток стався під час обчислення правої частини присвоєння.

> Щоб зрозуміти присвоєння в Python, спочатку читайте праву частину: саме там об'єкт створюється або отримується. Після цього змінна в лівій частині прив'язується до об'єкта, як прикріплений стікер. Просто забудьте про ящики.

## Ідентичність, рівність та аліаси

Льюїс Керрол — псевдонім проф. Чарльза Лутвіджа Доджсона. Містер Керрол не лише є рівним проф. Доджсону — вони одна й та сама людина. Приклад 6-3 виражає цю ідею в Python.

**Приклад 6-3.** `charles` і `lewis` посилаються на один і той самий об'єкт

```python
>>> charles = {'name': 'Charles L. Dodgson', 'born': 1832}
>>> lewis = charles
>>> lewis is charles
True
>>> id(charles), id(lewis)
(4300473992, 4300473992)
>>> lewis['balance'] = 950
>>> charles
{'name': 'Charles L. Dodgson', 'born': 1832, 'balance': 950}
```

`lewis` є аліасом для `charles`. Оператор `is` і функція `id` підтверджують це. Додавання елемента до `lewis` — це те саме, що додавання елемента до `charles`.

Однак припустимо, самозванець — назвемо його д-р Олександр Педаченко — стверджує, що він є Чарльз Л. Доджсон, народжений у 1832 році. Його відомості можуть бути тими самими, але д-р Педаченко — не проф. Доджсон. Рисунок 6-2 ілюструє цей сценарій.

**Рисунок 6-2.** `charles` і `lewis` прив'язані до одного й того самого об'єкта; `alex` прив'язаний до окремого об'єкта з однаковим значенням.

**Приклад 6-4.** `alex` і `charles` порівнюються як рівні, але `alex` — не `charles`

```python
>>> alex = {'name': 'Charles L. Dodgson', 'born': 1832, 'balance': 950}
>>> alex == charles
True
>>> alex is not charles
True
```

`alex` посилається на об'єкт, що є копією об'єкта, присвоєного `charles`. Об'єкти порівнюються як рівні завдяки реалізації `__eq__` у класі `dict`. Але вони є різними об'єктами. Це пітонічний спосіб написання негативного порівняння ідентичності: `a is not b`.

Приклад 6-3 є прикладом аліасингу. У цьому коді `lewis` і `charles` — це аліаси: дві змінні, прив'язані до одного й того самого об'єкта. З іншого боку, `alex` не є аліасом для `charles`: ці змінні прив'язані до різних об'єктів. Об'єкти, прив'язані до `alex` і `charles`, мають однакове значення — це те, що порівнює `==` — але вони мають різні ідентичності.

У *The Python Language Reference*, «3.1. Objects, values and types» зазначається:

> Ідентичність об'єкта ніколи не змінюється після його створення; можна думати про неї як про адресу об'єкта в пам'яті. Оператор `is` порівнює ідентичності двох об'єктів; функція `id()` повертає ціле число, що представляє його ідентичність.

Реальний смисл ID об'єкта залежить від реалізації. У CPython `id()` повертає адресу об'єкта в пам'яті, але в іншому інтерпретаторі Python це може бути щось інше. Ключовий момент: ID гарантовано є унікальним цілочисельним ярликом, що не зміниться протягом існування об'єкта.

На практиці ми рідко використовуємо функцію `id()` під час програмування. Перевірки ідентичності найчастіше виконуються за допомогою оператора `is`, що порівнює ID об'єктів, тому нашому коду не потрібно явно викликати `id()`.

### Вибір між == та is

Оператор `==` порівнює значення об'єктів (дані, що вони зберігають), тоді як `is` порівнює їхні ідентичності.

Під час програмування ми найчастіше більше дбаємо про значення, ніж про ідентичності об'єктів, тому `==` з'являється в коді Python частіше, ніж `is`.

Однак якщо ви порівнюєте змінну з одинаком, то має сенс використовувати `is`. Найпоширенішим випадком є перевірка того, чи прив'язана змінна до `None`. Ось рекомендований спосіб:

```python
x is None
```

І правильний спосіб записати заперечення:

```python
x is not None
```

`None` — найпоширеніший одинак, який ми перевіряємо за допомогою `is`. Сентинельні об'єкти — ще один приклад одинаків, що перевіряються через `is`. Ось один зі способів створення і перевірки сентинельного об'єкта:

```python
END_OF_DATA = object()
# ... багато рядків
def traverse(...):
    # ... ще рядки
    if node is END_OF_DATA:
        return
    # тощо.
```

Оператор `is` є швидшим за `==`, бо його не можна перевантажити, тому Python не повинен знаходити і викликати спеціальні методи для його обчислення, а обчислення `is` — це просто порівняння двох цілочисельних ID. Натомість `a == b` є синтаксичним цукром для `a.__eq__(b)`. Метод `__eq__`, успадкований від `object`, порівнює ID об'єктів, тому дає той самий результат, що й `is`. Але більшість вбудованих типів перевизначає `__eq__` більш змістовними реалізаціями, що фактично враховують значення атрибутів об'єкта. Рівність може передбачати багато обробки — наприклад, при порівнянні великих колекцій або глибоко вкладених структур.

> Зазвичай ми більше зацікавлені в рівності об'єктів, ніж у ідентичності. Перевірка на `None` — єдиний поширений варіант використання оператора `is`. Більшість інших застосувань, які я бачу при перегляді коду, є помилками. Якщо ви не впевнені, використовуйте `==`. Це зазвичай те, що вам потрібно, і також працює з `None` — хоча й не так швидко.

## Відносна незмінність кортежів

Кортежі, як і більшість колекцій Python — списки, `dict`, множини тощо — є контейнерами: вони зберігають посилання на об'єкти.[^2] Якщо елементи, на які є посилання, є мутабельними, вони можуть змінюватися навіть якщо сам кортеж не змінюється. Іншими словами, незмінність кортежів насправді стосується фізичного вмісту структури даних кортежу (тобто посилань, що він зберігає) і не поширюється на об'єкти, на які є посилання.

[^2]: На відміну від них, плоскі послідовності, наприклад `str`, `bytes` та `array.array`, не містять посилань, а безпосередньо зберігають свій вміст — символи, байти та числа — у суміжній пам'яті.

Приклад 6-5 ілюструє ситуацію, в якій значення кортежу змінюється в результаті змін мутабельного об'єкта, на який є посилання в ньому. Те, що в кортежі ніколи не може змінитися, — це ідентичності елементів, що він містить.

**Приклад 6-5.** `t1` і `t2` спочатку порівнюються як рівні, але зміна мутабельного елемента всередині кортежу `t1` робить його іншим

```python
>>> t1 = (1, 2, [30, 40])
>>> t2 = (1, 2, [30, 40])
>>> t1 == t2
True
>>> id(t1[-1])
4302515784
>>> t1[-1].append(99)
>>> t1
(1, 2, [30, 40, 99])
>>> id(t1[-1])
4302515784
>>> t1 == t2
False
```

`t1` є незмінним, але `t1[-1]` є мутабельним. Побудувати кортеж `t2`, елементи якого рівні елементам `t1`. Хоча вони є різними об'єктами, `t1` і `t2` порівнюються як рівні — як і очікується. Перевірити ідентичність списку на `t1[-1]`. Змінити список `t1[-1]` на місці. Ідентичність `t1[-1]` не змінилася, змінилося лише його значення. Тепер `t1` і `t2` є різними.

Ця відносна незмінність кортежів стоїть за загадкою «Загадка з +=» (стор. 54). Це також причина, через яку деякі кортежі не є гешованими, як ми бачили в «Що таке гешованість» (стор. 84).

## Копії є поверхневими за замовчуванням

Найпростіший спосіб скопіювати список (або більшість вбудованих мутабельних колекцій) — використати вбудований конструктор відповідного типу. Наприклад:

```python
>>> l1 = [3, [55, 44], (7, 8, 9)]
>>> l2 = list(l1)
>>> l2
[3, [55, 44], (7, 8, 9)]
>>> l2 == l1
True
>>> l2 is l1
False
```

`list(l1)` створює копію `l1`. Копії є рівними... ...але посилаються на два різних об'єкти.

Для списків та інших мутабельних послідовностей скорочення `l2 = l1[:]` також створює копію. Однак використання конструктора або `[:]` виробляє **поверхневу копію** (тобто зовнішній контейнер дублюється, але копія заповнюється посиланнями на ті самі елементи, що зберігає вихідний контейнер). Це заощаджує пам'ять і не спричиняє проблем, якщо всі елементи є незмінними. Але якщо є мутабельні елементи, це може призводити до неприємних сюрпризів.

У Прикладі 6-6 ми створюємо поверхневу копію списку, що містить ще один список і кортеж, а потім вносимо зміни, щоб побачити, як вони впливають на об'єкти, на які є посилання.

**Приклад 6-6.** Створення поверхневої копії списку, що містить ще один список; скопіюйте і вставте цей код, щоб переглянути його анімацію в Online Python Tutor

```python
l1 = [3, [66, 55, 44], (7, 8, 9)]
l2 = list(l1)
l1.append(100)
l1[1].remove(55)
print('l1:', l1)
print('l2:', l2)
l2[1] += [33, 22]
l2[2] += (10, 11)
print('l1:', l1)
print('l2:', l2)
```

`l2` є поверхневою копією `l1`. Додавання `100` до `l1` не впливає на `l2`. Тут ми видаляємо `55` із внутрішнього списку `l1[1]`. Це впливає на `l2`, бо `l2[1]` прив'язаний до того самого списку, що й `l1[1]`. Для мутабельного об'єкта, на який посилається `l2[1]`, оператор `+=` змінює список на місці. Ця зміна видима у `l1[1]`, що є аліасом для `l2[1]`. `+=` на кортежі створює новий кортеж і перевв'язує змінну `l2[2]`. Це те саме, що й `l2[2] = l2[2] + (10, 11)`. Тепер кортежі на останній позиції `l1` і `l2` більше не є одним об'єктом.

## Глибокі та поверхневі копії довільних об'єктів

Робота з поверхневими копіями не завжди є проблемою, але іноді потрібно робити **глибокі копії** (тобто дублікати, що не спільно використовують посилання на вбудовані об'єкти). Модуль `copy` надає функції `deepcopy` та `copy`, що повертають глибокі та поверхневі копії довільних об'єктів.

**Приклад 6-8.** `Bus` підбирає і висаджує пасажирів

```python
class Bus:
    def __init__(self, passengers=None):
        if passengers is None:
            self.passengers = []
        else:
            self.passengers = list(passengers)

    def pick(self, name):
        self.passengers.append(name)

    def drop(self, name):
        self.passengers.remove(name)
```

**Приклад 6-9.** Ефекти використання `copy` проти `deepcopy`

```python
>>> import copy
>>> bus1 = Bus(['Alice', 'Bill', 'Claire', 'David'])
>>> bus2 = copy.copy(bus1)
>>> bus3 = copy.deepcopy(bus1)
>>> id(bus1), id(bus2), id(bus3)
(4301498296, 4301499416, 4301499752)
>>> bus1.drop('Bill')
>>> bus2.passengers
['Alice', 'Claire', 'David']
>>> id(bus1.passengers), id(bus2.passengers), id(bus3.passengers)
(4302658568, 4302658568, 4302657800)
>>> bus3.passengers
['Alice', 'Bill', 'Claire', 'David']
```

Використовуючи `copy` і `deepcopy`, ми створюємо три різних екземпляри `Bus`. Після того як `bus1` висаджує `'Bill'`, він також відсутній у `bus2`. Перевірка атрибутів `passengers` показує, що `bus1` і `bus2` спільно використовують той самий об'єкт-список, бо `bus2` є поверхневою копією `bus1`. `bus3` є глибокою копією `bus1`, тому його атрибут `passengers` посилається на інший список.

Зверніть увагу, що робити глибокі копії — це не проста справа в загальному випадку. Об'єкти можуть мати циклічні посилання, що можуть призвести до нескінченного циклу при наївному алгоритмі. Функція `deepcopy` пам'ятає вже скопійовані об'єкти для коректної обробки циклічних посилань. Це демонструє Приклад 6-10.

**Приклад 6-10.** Циклічні посилання: `b` посилається на `a`, а потім додається до `a`; `deepcopy` все одно вміє скопіювати `a`

```python
>>> a = [10, 20]
>>> b = [a, 30]
>>> a.append(b)
>>> a
[10, 20, [[...], 30]]
>>> from copy import deepcopy
>>> c = deepcopy(a)
>>> c
[10, 20, [[...], 30]]
```

Крім того, глибока копія може бути надто глибокою в деяких випадках. Наприклад, об'єкти можуть посилатися на зовнішні ресурси або одинаки, які не слід копіювати. Можна контролювати поведінку як `copy`, так і `deepcopy`, реалізувавши спеціальні методи `__copy__()` і `__deepcopy__()`, як описано в документації модуля `copy`.

## Параметри функцій як посилання

Єдиний режим передачі параметрів у Python — **виклик із спільним доступом** (call by sharing). Це той самий режим, що використовується в більшості об'єктно-орієнтованих мов, включно з JavaScript, Ruby та Java (для посилальних типів Java; примітивні типи використовують виклик за значенням). Виклик із спільним доступом означає, що кожний формальний параметр функції отримує копію кожного посилання в аргументах. Іншими словами, параметри всередині функції стають аліасами фактичних аргументів.

Результат цієї схеми полягає в тому, що функція може змінювати будь-який мутабельний об'єкт, переданий як параметр, але не може змінити ідентичність цих об'єктів (тобто не може повністю замінити об'єкт іншим). Приклад 6-11 показує просту функцію з оператором `+=` для одного з її параметрів.

**Приклад 6-11.** Функція може змінювати будь-який отриманий мутабельний об'єкт

```python
>>> def f(a, b):
...     a += b
...     return a
...
>>> x = 1
>>> y = 2
>>> f(x, y)
3
>>> x, y
(1, 2)
>>> a = [1, 2]
>>> b = [3, 4]
>>> f(a, b)
[1, 2, 3, 4]
>>> a, b
([1, 2, 3, 4], [3, 4])
>>> t = (10, 20)
>>> u = (30, 40)
>>> f(t, u)
(10, 20, 30, 40)
>>> t, u
((10, 20), (30, 40))
```

Число `x` не змінилося. Список `a` змінився. Кортеж `t` не змінився.

### Мутабельні типи як значення за замовчуванням параметрів: погана ідея

Необов'язкові параметри зі значеннями за замовчуванням — чудова особливість визначень функцій Python, що дозволяє нашим API еволюціонувати зберігаючи зворотну сумісність. Однак слід уникати мутабельних об'єктів як значень за замовчуванням для параметрів.

Щоб проілюструвати цю проблему, у Прикладі 6-12 ми беремо клас `Bus` із Прикладу 6-8 і змінюємо його метод `__init__` для створення `HauntedBus`. Тут ми намагалися бути кмітливими і замість значення за замовчуванням `passengers=None` використали `passengers=[]`, тим самим уникаючи `if` у попередньому `__init__`. Ця «кмітливість» заводить нас у халепу.

**Приклад 6-12.** Простий клас для ілюстрації небезпеки мутабельного значення за замовчуванням

```python
class HauntedBus:
    """A bus model haunted by ghost passengers"""
    # Модель автобусу, що переслідується примарами пасажирів

    def __init__(self, passengers=[]):
        self.passengers = passengers

    def pick(self, name):
        self.passengers.append(name)

    def drop(self, name):
        self.passengers.remove(name)
```

Коли аргумент `passengers` не передається, цей параметр прив'язується до об'єкта списку за замовчуванням, який спочатку є порожнім. Це присвоєння робить `self.passengers` аліасом для `passengers`, який сам є аліасом для списку за замовчуванням, коли аргумент `passengers` не вказано. Коли методи `.remove()` і `.append()` використовуються з `self.passengers`, ми фактично змінюємо список за замовчуванням, який є атрибутом об'єкта функції.

**Приклад 6-13.** Автобуси, що переслідуються примарами пасажирів

```python
>>> bus1 = HauntedBus(['Alice', 'Bill'])
>>> bus1.passengers
['Alice', 'Bill']
>>> bus1.pick('Charlie')
>>> bus1.drop('Alice')
>>> bus1.passengers
['Bill', 'Charlie']
>>> bus2 = HauntedBus()
>>> bus2.pick('Carrie')
>>> bus2.passengers
['Carrie']
>>> bus3 = HauntedBus()
>>> bus3.passengers
['Carrie']
>>> bus3.pick('Dave')
>>> bus2.passengers
['Carrie', 'Dave']
>>> bus2.passengers is bus3.passengers
True
>>> bus1.passengers
['Bill', 'Charlie']
```

`bus1` стартує з двома пасажирами. Поки що все нормально. `bus2` стартує порожнім, тому до `self.passengers` присвоюється список за замовчуванням. `bus3` також стартує порожнім, знову присвоюється той самий список за замовчуванням. Список за замовчуванням більше не порожній! Тепер `Dave`, підібраний `bus3`, з'являється і в `bus2`. Проблема: `bus2.passengers` і `bus3.passengers` посилаються на той самий список. Але `bus1.passengers` є окремим списком.

Проблема полягає в тому, що екземпляри `HauntedBus`, які не отримують початковий список пасажирів, в кінцевому підсумку спільно використовують той самий список.

Після запуску рядків Прикладу 6-13 можна перевірити об'єкт `HauntedBus.__init__` і побачити «примарних студентів», що переслідують його атрибут `__defaults__`:

```python
>>> HauntedBus.__init__.__defaults__
(['Carrie', 'Dave'],)
```

Нарешті, можна переконатися, що `bus2.passengers` є аліасом, прив'язаним до першого елемента атрибута `HauntedBus.__init__.__defaults__`:

```python
>>> HauntedBus.__init__.__defaults__[0] is bus2.passengers
True
```

Проблема з мутабельними значеннями за замовчуванням пояснює, чому `None` зазвичай використовується як значення за замовчуванням для параметрів, що можуть отримувати мутабельні значення. У Прикладі 6-8 `__init__` перевіряє, чи є аргумент `passengers` рівним `None`. Якщо так, `self.passengers` прив'язується до нового порожнього списку. Якщо `passengers` не є `None`, правильна реалізація прив'язує **копію** цього аргументу до `self.passengers`.

### Захисне програмування з мутабельними параметрами

Коли ви пишете функцію, що отримує мутабельний параметр, слід ретельно подумати, чи очікує виклик, що переданий аргумент буде змінений.

Наприклад, якщо ваша функція отримує `dict` і потребує його змінити під час обробки, чи має цей побічний ефект бути видимим поза функцією чи ні? Це справді залежить від контексту. Це питання узгодженості очікувань розробника функції та викликаючого коду.

Останній приклад з автобусом у цьому розділі показує, як `TwilightBus` порушує очікування, спільно використовуючи свій список пасажирів із клієнтами.

**Приклад 6-14.** Пасажири зникають, коли їх висаджує `TwilightBus`

```python
>>> basketball_team = ['Sue', 'Tina', 'Maya', 'Diana', 'Pat']
>>> bus = TwilightBus(basketball_team)
>>> bus.drop('Tina')
>>> bus.drop('Pat')
>>> basketball_team
['Sue', 'Maya', 'Diana']
```

`basketball_team` містить п'ять імен. `TwilightBus` завантажується командою. Автобус висаджує одного студента, потім ще одного. Висаджені пасажири зникли з баскетбольної команди!

`TwilightBus` порушує «Принцип найменшого здивування» — найкращу практику дизайну інтерфейсу.[^3] Безперечно, дивно, що коли автобус висаджує студента, його ім'я видаляється зі списку баскетбольної команди.

[^3]: Дивіться Principle of least astonishment в англійській Вікіпедії.

**Приклад 6-15.** Простий клас для демонстрації небезпеки зміни отриманих аргументів

```python
class TwilightBus:
    """A bus model that makes passengers vanish"""
    # Модель автобусу, що змушує пасажирів зникати

    def __init__(self, passengers=None):
        if passengers is None:
            self.passengers = []
        else:
            self.passengers = passengers  # ← ПРОБЛЕМА

    def pick(self, name):
        self.passengers.append(name)

    def drop(self, name):
        self.passengers.remove(name)
```

Тут ми обережно створюємо новий порожній список, коли `passengers` є `None`. Однак це присвоєння робить `self.passengers` аліасом для `passengers`, який сам є аліасом для фактичного аргументу, переданого до `__init__` (тобто `basketball_team` у Прикладі 6-14). Коли методи `.remove()` і `.append()` використовуються з `self.passengers`, ми фактично змінюємо вихідний список, отриманий як аргумент конструктора.

Проблема тут у тому, що автобус утворює аліас для списку, що передається конструктору. Натомість він повинен зберігати власний список пасажирів. Виправлення просте: в `__init__`, якщо параметр `passengers` вказано, `self.passengers` слід ініціалізувати **копією** його, як ми правильно зробили в Прикладі 6-8:

```python
def __init__(self, passengers=None):
    if passengers is None:
        self.passengers = []
    else:
        self.passengers = list(passengers)
```

Зробити копію списку `passengers` або перетворити його на список, якщо він ним не є.

Тепер наша внутрішня обробка списку пасажирів не впливатиме на аргумент, що використовувався для ініціалізації автобусу. Як бонус, це рішення є більш гнучким: тепер аргумент, переданий до параметра `passengers`, може бути кортежем або будь-яким іншим ітерабельним об'єктом — множиною або навіть результатом запиту до бази даних, — бо конструктор `list` приймає будь-який ітерабельний об'єкт.

> Якщо метод явно не призначений для зміни об'єкта, отриманого як аргумент, слід двічі подумати, перш ніж утворювати аліас об'єкта-аргументу шляхом простого присвоєння його змінній екземпляра у вашому класі. Якщо сумніваєтеся — зробіть копію. Ваші клієнти будуть щасливіші. Звичайно, копіювання не є безкоштовним: є вартість у CPU та пам'яті. Однак API, що спричиняє непомітні помилки, зазвичай є більшою проблемою, ніж трохи повільніший або такий, що використовує більше ресурсів.

## del та збирання сміття

> Об'єкти ніколи не знищуються явно; однак, коли вони стають недосяжними, вони можуть бути утилізовані збирачем сміття.
>
> — Глава «Data Model» у *The Python Language Reference*

Перший дивний факт про `del` полягає в тому, що це не функція, а інструкція. Ми пишемо `del x`, а не `del(x)` — хоча останнє також працює, але лише тому, що вирази `x` і `(x)` зазвичай означають одне й те саме в Python.

Другий несподіваний факт: `del` видаляє посилання, а не об'єкти. Збирач сміття Python може видалити об'єкт із пам'яті як непрямий результат `del`, якщо видалена змінна була останнім посиланням на об'єкт. Перев'язування змінної також може спричинити зменшення кількості посилань на об'єкт до нуля, що призведе до його знищення.

```python
>>> a = [1, 2]
>>> b = a
>>> del a
>>> b
[1, 2]
>>> b = [3]
```

Створити об'єкт `[1, 2]` і прив'язати `a` до нього. Прив'язати `b` до того самого об'єкта `[1, 2]`. Видалити посилання `a`. `[1, 2]` не постраждав, бо `b` все ще вказує на нього. Перев'язування `b` на інший об'єкт видаляє останнє посилання на `[1, 2]`. Тепер збирач сміття може видалити цей об'єкт.

> Існує спеціальний метод `__del__`, але він не спричиняє знищення екземпляра і не повинен викликатися вашим кодом. `__del__` викликається інтерпретатором Python, коли екземпляр ось-ось буде знищений, щоб дати йому можливість звільнити зовнішні ресурси. Вам рідко потрібно реалізовувати `__del__` у власному коді, але деякі Python-розробники витрачають час на його написання без вагомих причин. Дивіться документацію спеціального методу `__del__` у главі «Data Model» у *The Python Language Reference*.

У CPython основним алгоритмом збирання сміття є підрахунок посилань. По суті, кожний об'єкт підтримує лічильник кількості посилань, що вказують на нього. Як тільки цей лічильник досягає нуля, об'єкт негайно знищується: CPython викликає метод `__del__` на об'єкті (якщо він визначений) і потім звільняє пам'ять, виділену для об'єкта.

Щоб продемонструвати кінець існування об'єкта, Приклад 6-16 використовує `weakref.finalize` для реєстрації функції зворотного виклику, яка буде викликана при знищенні об'єкта.

**Приклад 6-16.** Спостереження за кінцем об'єкта, на який більше немає посилань

```python
>>> import weakref
>>> s1 = {1, 2, 3}
>>> s2 = s1
>>> def bye():
...     print('...like tears in the rain.')
...
>>> ender = weakref.finalize(s1, bye)
>>> ender.alive
True
>>> del s1
>>> ender.alive
True
>>> s2 = 'spam'
...like tears in the rain.
>>> ender.alive
False
```

`s1` і `s2` є аліасами, що посилаються на ту саму множину `{1, 2, 3}`. Ця функція не повинна бути прив'язаним методом знищуваного об'єкта або утримувати інше посилання на нього. Зареєструвати зворотний виклик `bye` на об'єкті, на який посилається `s1`. Атрибут `.alive` є `True` до виклику об'єкта `finalize`. Як обговорювалося, `del` не видалив об'єкт, лише посилання `s1` на нього. Перев'язування останнього посилання `s2` робить `{1, 2, 3}` недосяжним. Об'єкт знищується, викликається зворотний виклик `bye`, і `ender.alive` стає `False`.

## Хитрощі Python із незмінними об'єктами

> Цей необов'язковий розділ обговорює деякі деталі Python, що насправді не є важливими для користувачів Python і можуть не застосовуватися до інших реалізацій Python або навіть до майбутніх версій CPython. Тим не менш я бачив, як люди стикалися з цими крайніми випадками, і тоді починали неправильно використовувати оператор `is`, тому вважав, що їх варто згадати.

Мене здивувало дізнатися, що для кортежу `t`, `t[:]` не робить копію, а повертає посилання на той самий об'єкт. Ви також отримаєте посилання на той самий кортеж, якщо напишете `tuple(t)`.[^4] Приклад 6-17 доводить це.

[^4]: Це чітко задокументовано. Введіть `help(tuple)` у консолі Python, щоб прочитати: «If the argument is a tuple, the return value is the same object.» Я думав, що знав все про кортежі, перш ніж написав цю книгу.

**Приклад 6-17.** Кортеж, побудований з іншого, насправді є тим самим точним кортежем

```python
>>> t1 = (1, 2, 3)
>>> t2 = tuple(t1)
>>> t2 is t1
True
>>> t3 = t1[:]
>>> t3 is t1
True
```

`t1` і `t2` прив'язані до того самого об'єкта. І `t3` теж.

Та сама поведінка спостерігається з екземплярами `str`, `bytes` та `frozenset`. Зверніть увагу, що `frozenset` не є послідовністю, тому `fs[:]` не працює. Але `fs.copy()` має той самий ефект: воно «обманює» і повертає посилання на той самий об'єкт, а не копію.[^5]

[^5]: Нешкідлива брехня методу `copy`, що не копіює, виправдана сумісністю інтерфейсу: це робить `frozenset` більш сумісним із `set`. В будь-якому разі для кінцевого користувача не має значення, чи є два ідентичних незмінних об'єкти тим самим чи копіями.

**Приклад 6-18.** Рядкові літерали можуть створювати спільні об'єкти

```python
>>> t1 = (1, 2, 3)
>>> t3 = (1, 2, 3)
>>> t3 is t1
False
>>> s1 = 'ABC'
>>> s2 = 'ABC'
>>> s2 is s1
True
```

Створення нового кортежу з нуля. `t1` і `t3` є рівними, але не є тим самим об'єктом. Створення другого `str` з нуля. Несподіванка: `s1` і `s2` посилаються на той самий `str`!

Спільне використання рядкових літералів — це техніка оптимізації, що називається **інтернуванням** (interning). CPython використовує схожу техніку з малими цілими числами, щоб уникнути непотрібного дублювання чисел, що часто зустрічаються в програмах, наприклад 0, 1, –1 тощо. Зверніть увагу: CPython не інтернує всі рядки або цілі числа, і критерії, що він використовує для цього, є недокументованою деталлю реалізації.

> Ніколи не покладайтеся на інтернування `str` або `int`! Завжди використовуйте `==` замість `is` для порівняння рядків або цілих чисел на рівність. Інтернування — це оптимізація для внутрішнього використання інтерпретатора Python.

## Підсумок розділу

Кожний об'єкт Python має ідентичність, тип і значення. Лише значення об'єкта може змінюватися з часом.[^7]

[^7]: Насправді тип об'єкта може бути змінений шляхом простого присвоєння іншого класу його атрибуту `__class__`, але це є чистим злом, і я шкодую, що написав це виноску.

Якщо дві змінні посилаються на незмінні об'єкти, що мають рівні значення (`a == b` є `True`), на практиці рідко має значення, чи посилаються вони на копії або є аліасами, що посилаються на той самий об'єкт, бо значення незмінного об'єкта не змінюється — з одним винятком. Виняток — незмінні колекції, наприклад кортежі: якщо незмінна колекція містить посилання на мутабельні елементи, її значення фактично може змінюватися при зміні значення мутабельного елемента. На практиці цей сценарій не є настільки поширеним. Те, що ніколи не змінюється в незмінній колекції, — це ідентичності об'єктів у ній. Клас `frozenset` не страждає від цієї проблеми, тому що може містити лише гешовані елементи, а значення гешованих об'єктів за визначенням ніколи не може змінюватися.

Той факт, що змінні зберігають посилання, має багато практичних наслідків у програмуванні Python:

- Просте присвоєння не створює копій.
- Скорочені присвоєння з `+=` або `*=` створюють нові об'єкти, якщо ліва змінна прив'язана до незмінного об'єкта, але можуть змінювати мутабельний об'єкт на місці.
- Присвоєння нового значення існуючій змінній не змінює об'єкт, що раніше був до неї прив'язаний. Це називається **перев'язуванням**: змінна тепер прив'язана до іншого об'єкта. Якщо ця змінна була останнім посиланням на попередній об'єкт, цей об'єкт буде утилізований збирачем сміття.
- Параметри функцій передаються як аліаси, що означає: функція може змінювати будь-який мутабельний об'єкт, отриманий як аргумент. Немає способу запобігти цьому, крім як створювати локальні копії або використовувати незмінні об'єкти (наприклад, передавати кортеж замість списку).
- Використання мутабельних об'єктів як значень за замовчуванням для параметрів функцій є небезпечним, бо якщо параметри змінюються на місці, значення за замовчуванням змінюється, що впливає на кожний майбутній виклик, що покладається на значення за замовчуванням.

У CPython об'єкти видаляються, як тільки кількість посилань на них досягає нуля. Вони також можуть бути видалені, якщо утворюють групи з циклічними посиланнями, але без зовнішніх посилань.

В деяких ситуаціях може бути корисним зберігати посилання на об'єкт, яке само по собі не підтримує об'єкт «живим». Один із прикладів — клас, який хоче відстежувати всі свої поточні екземпляри. Це можна зробити за допомогою слабких посилань — низькорівневого механізму, що лежить в основі більш корисних колекцій `WeakValueDictionary`, `WeakKeyDictionary`, `WeakSet` і функції `finalize` з модуля `weakref`. Докладніше — у «Weak References» на fluentpython.com.

## Трибуна

### Рівне ставлення до всіх об'єктів

Я вивчав Java до того, як відкрив Python. Оператор `==` у Java ніколи не здавався мені правильним. Програмістам значно частіше важлива рівність, ніж ідентичність, але для об'єктів (не примітивних типів) Java `==` порівнює посилання, а не значення об'єктів. Навіть для такої базової речі, як порівняння рядків, Java змушує вас використовувати метод `.equals`. Навіть тоді є ще одна пастка: якщо написати `a.equals(b)` і `a` є `null`, отримаєте виняток null pointer. Розробники Java відчули необхідність перевантажити `+` для рядків, то чому б не перевантажити й `==`?

Python робить це правильно. Оператор `==` порівнює значення об'єктів; `is` порівнює посилання. І оскільки Python підтримує перевантаження операторів, `==` розумно працює з усіма об'єктами стандартної бібліотеки, зокрема `None`, що є правильним об'єктом — на відміну від `null` у Java.

### Мутабельність

Цей розділ не був би потрібний, якби всі об'єкти Python були незмінними. Коли ви маєте справу зі змінними об'єктами, неважливо, зберігають чи змінні самі об'єкти чи посилання на спільні об'єкти. Якщо `a == b` є `true` і жоден об'єкт не може змінитися, вони могли б так само бути тим самим. Саме тому інтернування рядків є безпечним. Ідентичність об'єкта стає важливою лише тоді, коли об'єкти є мутабельними.

У «чистому» функціональному програмуванні всі дані є незмінними: додавання до колекції фактично створює нову колекцію. Elixir — одна з доступних для вивчення практичних функціональних мов, у якій усі вбудовані типи є незмінними, зокрема списки.

Python, однак, не є функціональною мовою — тим менше чистою. Екземпляри класів, визначених користувачем, є мутабельними за замовчуванням у Python, як і в більшості об'єктно-орієнтованих мов.

### Знищення об'єктів та збирання сміття

У Python немає механізму для безпосереднього знищення об'єкта, і це насправді є чудовою особливістю: якби ви могли знищити об'єкт у будь-який час, що сталося б із наявними посиланнями на нього?

Збирання сміття в CPython виконується перш за все за рахунком посилань, що є простим у реалізації, але схильне до витоків пам'яті при наявності циклічних посилань. Тому у версії 2.0 (жовтень 2000 року) був реалізований генераційний збирач сміття, що здатний видаляти недосяжні об'єкти, що підтримуються в живих циклічними посиланнями.

Але підрахунок посилань залишається базовим рівнем і спричиняє негайне видалення об'єктів із нульовою кількістю посилань. Це означає, що в CPython — принаймні зараз — безпечно писати:

```python
open('test.txt', 'wt', encoding='utf-8').write('1, 2, 3')
```

Цей код є безпечним, тому що лічильник посилань файлового об'єкта буде нульовим після повернення методу `write`, і Python негайно закриє файл до знищення представляючого його в пам'яті об'єкта. Однак той самий рядок не є безпечним у Jython або IronPython, що використовують збирач сміття своїх хост-середовищ (Java VM та .NET CLR), які є більш складними, але не покладаються на підрахунок посилань і можуть довше зволікати зі знищенням об'єкта та закриттям файлу. У всіх випадках, включно з CPython, найкращою практикою є явне закриття файлу, а найнадійнішим способом є використання інструкції `with`, яка гарантує закриття файлу навіть при виникненні винятків під час його відкриття:

```python
with open('test.txt', 'wt', encoding='utf-8') as fp:
    fp.write('1, 2, 3')
```

### Передача параметрів: виклик із спільним доступом

Популярний спосіб пояснення того, як передача параметрів працює в Python, — фраза: «Параметри передаються за значенням, але значення є посиланнями». Це не є помилкою, але спричиняє плутанину, тому що найбільш поширені режими передачі параметрів у старіших мовах — виклик за значенням (функція отримує копію аргументу) і виклик за посиланням (функція отримує вказівник на аргумент). У Python функція отримує копію аргументів, але аргументи завжди є посиланнями. Тому значення об'єктів, на які є посилання, може бути змінено, якщо вони є мутабельними, але їхня ідентичність не може. Крім того, оскільки функція отримує копію посилання в аргументі, перев'язування його у тілі функції не має ефекту поза функцією. Я прийняв термін «виклик із спільним доступом» після вивчення теми в *Programming Language Pragmatics*, 3rd ed., by Michael L. Scott (Morgan Kaufmann), розділ «8.3.1: Parameter Modes».
