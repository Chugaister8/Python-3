# Розділ 4. Текст Unicode проти байтів

> Люди використовують текст. Комп'ютери говорять байтами.
>
> — Естер Нам та Тревіс Фішер, «Character Encoding and Unicode in Python»[^1]

[^1]: Слайд 12 доповіді PyCon 2014 «Character Encoding and Unicode in Python» (слайди, відео).

Python 3 запровадив чітке розмежування між рядками людського тексту та послідовностями необроблених байтів. Неявне перетворення послідовностей байтів у текст Unicode залишилось у минулому. Цей розділ присвячений рядкам Unicode, бінарним послідовностям та кодуванням, що використовуються для перетворення між ними.

Залежно від того, чим ви займаєтесь у Python, може здатися, що розуміння Unicode не є важливим. Це малоймовірно, але навіть у такому разі уникнути розмежування `str` і байтів неможливо. Як бонус, ви дізнаєтесь, що спеціалізовані типи бінарних послідовностей надають можливості, яких не мав «універсальний» тип `str` у Python 2.

У цьому розділі ми розглянемо такі теми:

- символи, кодові точки та байтові представлення;
- унікальні особливості бінарних послідовностей: `bytes`, `bytearray` і `memoryview`;
- кодування для повного Unicode та застарілих наборів символів;
- уникнення та обробка помилок кодування;
- найкращі практики при роботі з текстовими файлами;
- пастка типового кодування та проблеми стандартного вводу-виводу;
- безпечне порівняння тексту Unicode з нормалізацією;
- утилітні функції для нормалізації, згортання регістру та видалення діакритики;
- правильне сортування тексту Unicode з `locale` та бібліотекою `pyuca`;
- метадані символів у базі даних Unicode;
- двохрежимні API, що обробляють `str` і `bytes`.

## Питання символів

Концепція «рядка» досить проста: рядок — це послідовність символів. Проблема криється у визначенні «символу».

У 2021 році найкраще визначення «символу» — це символ Unicode. Відповідно, елементи, які ми отримуємо зі `str` у Python 3, є символами Unicode, так само як елементи об'єкта `unicode` в Python 2 — і не необроблені байти, які ми отримували зі `str` у Python 2.

Стандарт Unicode явно розмежовує ідентичність символів та конкретні байтові представлення:

- Ідентичність символу — його **кодова точка** — це число від 0 до 1 114 111 (у десятковому записі), що показується в стандарті Unicode у вигляді від 4 до 6 шістнадцяткових цифр з префіксом «U+», від U+0000 до U+10FFFF. Наприклад, кодова точка для літери A — U+0041, знак євро — U+20AC, а музичний символ скрипкового ключа призначений кодовій точці U+1D11E. Близько 13% дійсних кодових точок мають призначені символи в Unicode 13.0.0 — стандарті, що використовується в Python 3.10.0b4.
- Фактичні байти, що представляють символ, залежать від кодування, що використовується. **Кодування** — це алгоритм, що перетворює кодові точки на послідовності байтів і навпаки. Кодова точка для літери A (U+0041) кодується як один байт `\x41` у кодуванні UTF-8, або як байти `\x41\x00` у кодуванні UTF-16LE. Як інший приклад: UTF-8 вимагає трьох байтів — `\xe2\x82\xac` — для кодування знака євро (U+20AC), тоді як у UTF-16LE та сама кодова точка кодується двома байтами: `\xac\x20`.

Перетворення з кодових точок у байти — це **кодування**; перетворення з байтів у кодові точки — це **декодування**. Дивіться Приклад 4-1.

**Приклад 4-1.** Кодування та декодування

```python
>>> s = 'café'
>>> len(s)
4
>>> b = s.encode('utf8')
>>> b
b'caf\xc3\xa9'
>>> len(b)
5
>>> b.decode('utf8')
'café'
```

`str` `'café'` має чотири символи Unicode. Кодувати `str` у байти за допомогою кодування UTF-8. Літерали `bytes` мають префікс `b`. Байти `b` має п'ять байтів (кодова точка для «é» кодується двома байтами в UTF-8). Декодувати байти у `str` за допомогою кодування UTF-8.

> Якщо потрібна підказка для розрізнення `.decode()` від `.encode()`, переконайте себе, що послідовності байтів можуть бути криптичними дампами пам'яті машини, тоді як об'єкти Unicode `str` — «людський» текст. Тому має сенс, що ми **де**кодуємо байти у `str`, щоб отримати зрозумілий людині текст, і **кодуємо** `str` у байти для зберігання або передачі.

Хоча `str` у Python 3 — це майже тип `unicode` з Python 2 із новим іменем, `bytes` у Python 3 — не просто перейменований старий `str`, а також існує тісно пов'язаний тип `bytearray`. Тому варто поглянути на типи бінарних послідовностей перш ніж переходити до питань кодування/декодування.

## Основи байтів

Нові типи бінарних послідовностей у багатьох відношеннях відрізняються від `str` у Python 2. Перше, що слід знати: є два основні вбудовані типи для бінарних послідовностей — незмінний тип `bytes`, введений у Python 3, і мутабельний `bytearray`, доданий ще в Python 2.6.[^2] Документація Python іноді використовує загальний термін «byte string» для позначення обох типів. Я уникаю цього заплутаного терміна.

[^2]: Python 2.6 і 2.7 також мали `bytes`, але це був просто псевдонім для типу `str`.

Кожен елемент у `bytes` або `bytearray` — це ціле число від 0 до 255, а не рядок із одного символу, як у `str` Python 2. Однак зріз бінарної послідовності завжди виробляє бінарну послідовність того самого типу — зокрема зрізи довжиною 1. Дивіться Приклад 4-2.

**Приклад 4-2.** П'ятибайтова послідовність як `bytes` і як `bytearray`

```python
>>> cafe = bytes('café', encoding='utf_8')
>>> cafe
b'caf\xc3\xa9'
>>> cafe[0]
99
>>> cafe[:1]
b'c'
>>> cafe_arr = bytearray(cafe)
>>> cafe_arr
bytearray(b'caf\xc3\xa9')
>>> cafe_arr[-1:]
bytearray(b'\xa9')
```

`bytes` можна побудувати зі `str` із вказаним кодуванням. Кожен елемент — ціле число в `range(256)`. Зрізи `bytes` — це теж `bytes`, навіть зрізи з одного байта. Для `bytearray` немає літерального синтаксису: вони відображаються як `bytearray()` з літералом `bytes` як аргументом. Зріз `bytearray` — це теж `bytearray`.

> Те, що `my_bytes[0]` повертає `int`, але `my_bytes[:1]` повертає послідовність `bytes` довжиною 1, дивує лише тому, що ми звикли до типу `str` Python, де `s[0] == s[:1]`. Для всіх інших типів послідовностей Python 1 елемент і зріз довжиною 1 — різні речі.

Хоча бінарні послідовності є справді послідовностями цілих чисел, їхній літеральний запис відображає той факт, що ASCII-текст часто вбудований у них. Тому використовуються чотири різних варіанти відображення залежно від значення кожного байта:

- Для байтів із десятковими кодами від 32 до 126 — від пробілу до `~` (тильди) — використовується сам символ ASCII.
- Для байтів, що відповідають табуляції, символу нового рядка, символу повернення каретки та `\`, використовуються escape-послідовності `\t`, `\n`, `\r` та `\\`.
- Якщо в послідовності байтів є обидва обмежувачі рядків `'` і `"`, вся послідовність обмежується `'`, а будь-які `'` всередині екрануються як `\'`.[^3]
- Для всіх інших значень байтів використовується шістнадцяткова escape-послідовність (наприклад, `\x00` — нульовий байт).

[^3]: Цікавинка: символ ASCII «одинарна лапка», який Python використовує за замовчуванням як обмежувач рядків, насправді називається APOSTROPHE у стандарті Unicode. Справжні одинарні лапки є асиметричними: ліва — U+2018, права — U+2019.

Ось чому в Прикладі 4-2 ми бачимо `b'caf\xc3\xa9'`: перші три байти `b'caf'` — у друкованому діапазоні ASCII, останні два — ні.

Обидва типи `bytes` і `bytearray` підтримують усі методи `str`, крім тих, що виконують форматування (`format`, `format_map`) та залежать від даних Unicode, зокрема `casefold`, `isdecimal`, `isidentifier`, `isnumeric`, `isprintable` та `encode`. Це означає, що знайомі методи рядків, як-от `endswith`, `replace`, `strip`, `translate`, `upper` та десятки інших, можна використовувати з бінарними послідовностями — тільки з аргументами `bytes`, а не `str`. Крім того, функції регулярних виразів модуля `re` також працюють із бінарними послідовностями, якщо регекс скомпільований із бінарної послідовності, а не `str`. Починаючи з Python 3.5, оператор `%` знову працює з бінарними послідовностями.[^4]

[^4]: Він не працював у Python 3.0–3.4, спричиняючи великий біль розробникам, що мали справу з бінарними даними. Відновлення задокументовано в PEP 461 — Adding % formatting to bytes and bytearray.

Бінарні послідовності мають метод класу, якого немає у `str`, — `fromhex`, що будує бінарну послідовність, розбираючи пари шістнадцяткових цифр, необов'язково розділені пробілами:

```python
>>> bytes.fromhex('31 4B CE A9')
b'1K\xce\xa9'
```

Інші способи побудови екземплярів `bytes` або `bytearray` — це виклик їхніх конструкторів із:

- рядком `str` та ключовим аргументом кодування;
- ітерабельним об'єктом, що надає елементи зі значеннями від 0 до 255;
- об'єктом, що реалізує буферний протокол (наприклад, `bytes`, `bytearray`, `memoryview`, `array.array`), який копіює байти з вихідного об'єкта у нову бінарну послідовність.

> До Python 3.5 також можна було викликати `bytes` або `bytearray` з одним цілим числом для створення бінарної послідовності такого розміру, ініціалізованої нульовими байтами. Ця сигнатура була застарілою в Python 3.5 і вилучена в Python 3.6. Дивіться PEP 467 — Minor API improvements for binary sequences.

Побудова бінарної послідовності з буфероподібного об'єкта — це низькорівнева операція, яка може передбачати перетворення типів. Дивіться демонстрацію в Прикладі 4-3.

**Приклад 4-3.** Ініціалізація `bytes` із необробленими даними масиву

```python
>>> import array
>>> numbers = array.array('h', [-2, -1, 0, 1, 2])
>>> octets = bytes(numbers)
>>> octets
b'\xfe\xff\xff\xff\x00\x00\x01\x00\x02\x00'
```

Типовий код `'h'` створює масив коротких цілих чисел (16 біт). `octets` містить копію байтів, що складають `numbers`. Це 10 байтів, що представляють 5 коротких цілих чисел.

Створення об'єкта `bytes` або `bytearray` з будь-якого буфероподібного джерела завжди копіює байти. Натомість об'єкти `memoryview` дозволяють спільно використовувати пам'ять між бінарними структурами даних, як ми бачили в «Memory Views» (стор. 62).

## Базові кодеки/декодери

Дистрибутив Python містить понад 100 кодеків (кодерів/декодерів) для перетворення тексту в байти і навпаки. Кожний кодек має ім'я, наприклад `'utf_8'`, і часто псевдоніми, такі як `'utf8'`, `'utf-8'` та `'U8'`, які можна використовувати як аргумент `encoding` у функціях на зразок `open()`, `str.encode()`, `bytes.decode()` тощо.

Приклад 4-4 показує той самий текст, закодований у вигляді трьох різних послідовностей байтів.

**Приклад 4-4.** Рядок `"El Niño"`, закодований трьома кодеками, що виробляють дуже різні послідовності байтів

```python
>>> for codec in ['latin_1', 'utf_8', 'utf_16']:
...     print(codec, 'El Niño'.encode(codec), sep='\t')
...
latin_1    b'El Ni\xf1o'
utf_8      b'El Ni\xc3\xb1o'
utf_16     b'\xff\xfeE\x00l\x00 \x00N\x00i\x00\xf1\x00o\x00'
```

**Рисунок 4-1.** Дванадцять символів, їхні кодові точки та байтове представлення (у шістнадцятковому записі) у 7 різних кодуваннях (зірочки означають, що символ не може бути представлений у цьому кодуванні).

Усі ці зірочки на Рисунку 4-1 чітко показують, що деякі кодування, наприклад ASCII і навіть багатобайтовий GB2312, не можуть представити кожний символ Unicode. Кодування UTF, однак, розроблені для обробки будь-якої кодової точки Unicode.

Кодування, показані на Рисунку 4-1, були вибрані як репрезентативна вибірка:

**latin1, він же iso8859_1**
Важливий, бо є основою для інших кодувань, зокрема cp1252 та самого Unicode (зверніть увагу, як значення байтів latin1 з'являються у байтах cp1252 і навіть у кодових точках).

**cp1252**
Корисне надмножинне розширення latin1, створене Microsoft, яке додає корисні символи, наприклад фігурні лапки та € (євро). Деякі програми Windows називають його «ANSI», але це ніколи не було справжнім стандартом ANSI.

**cp437**
Оригінальний набір символів IBM PC із символами для малювання рамок. Несумісний із latin1, що з'явився пізніше.

**gb2312**
Застарілий стандарт для кодування спрощених китайських ієрогліфів, що використовуються в материковому Китаї; одне з кількох широко застосовуваних багатобайтових кодувань для азійських мов.

**utf-8**
Найпоширеніше 8-бітне кодування в інтернеті. Станом на липень 2021 року, «W3Techs: Usage statistics of character encodings for websites» стверджує, що 97% сайтів використовують UTF-8 — порівняно з 81,4%, коли я писав цей абзац у першому виданні у вересні 2014 року.

**utf-16le**
Одна з форм 16-бітної схеми кодування UTF; усі кодування UTF-16 підтримують кодові точки вище U+FFFF через escape-послідовності, що називаються «сурогатними парами».

> UTF-16 замінив оригінальне 16-бітне кодування Unicode 1.0 — UCS-2 — ще в 1996 році. UCS-2 досі використовується в багатьох системах, попри те що є застарілим з минулого сторіччя, бо підтримує лише кодові точки до U+FFFF. Станом на 2021 рік, понад 57% виділених кодових точок знаходяться вище U+FFFF, зокрема важливі емодзі.

## Розуміння проблем кодування/декодування

Хоча існує загальний виняток `UnicodeError`, помилка, про яку повідомляє Python, зазвичай є конкретнішою: або `UnicodeEncodeError` (при перетворенні `str` у бінарні послідовності), або `UnicodeDecodeError` (при читанні бінарних послідовностей у `str`). Завантаження модулів Python також може викликати `SyntaxError`, якщо кодування джерела є неочікуваним. Ми покажемо, як обробляти всі ці помилки в наступних розділах.

> Перше, на що слід звернути увагу при отриманні помилки Unicode, — це точний тип винятку. Це `UnicodeEncodeError`, `UnicodeDecodeError` або якась інша помилка (наприклад, `SyntaxError`), що згадує проблему кодування? Щоб розв'язати проблему, її спочатку потрібно зрозуміти.

### Робота з UnicodeEncodeError

Більшість кодеків не UTF обробляють лише невелику підмножину символів Unicode. При перетворенні тексту в байти, якщо символ не визначений у цільовому кодуванні, буде викликаний `UnicodeEncodeError`, якщо не забезпечена спеціальна обробка шляхом передачі аргументу `errors` до методу або функції кодування. Поведінка обробників помилок показана в Прикладі 4-5.

**Приклад 4-5.** Кодування у байти: успіх та обробка помилок

```python
>>> city = 'São Paulo'
>>> city.encode('utf_8')
b'S\xc3\xa3o Paulo'
>>> city.encode('utf_16')
b'\xff\xfeS\x00\xe3\x00o\x00 \x00P\x00a\x00u\x00l\x00o\x00'
>>> city.encode('iso8859_1')
b'S\xe3o Paulo'
>>> city.encode('cp437')
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
  File "/.../lib/python3.4/encodings/cp437.py", line 12, in encode
    return codecs.charmap_encode(input,errors,encoding_map)
UnicodeEncodeError: 'charmap' codec can't encode character '\xe3' in
position 1: character maps to <undefined>
>>> city.encode('cp437', errors='ignore')
b'So Paulo'
>>> city.encode('cp437', errors='replace')
b'S?o Paulo'
>>> city.encode('cp437', errors='xmlcharrefreplace')
b'S&#227;o Paulo'
```

Кодування UTF обробляють будь-який `str`. `iso8859_1` також працює для рядка `'São Paulo'`. `cp437` не може закодувати `'ã'` («a» з тильдою). Обробник помилок за замовчуванням — `'strict'` — викликає `UnicodeEncodeError`. Обробник `errors='ignore'` пропускає символи, що не можуть бути закодовані; це зазвичай дуже погана ідея, що призводить до мовчазної втрати даних. При кодуванні `errors='replace'` замінює незакодовувані символи на `'?'`; дані також втрачаються, але користувачі отримають підказку, що щось пішло не так. `'xmlcharrefreplace'` замінює незакодовувані символи XML-сутностями. Якщо не можна використовувати UTF і не можна дозволити собі втрату даних, це єдиний варіант.

> Обробка помилок у кодеках є розширюваною. Можна зареєструвати додаткові рядки для аргументу `errors`, передавши ім'я та функцію обробки помилок до функції `codecs.register_error`. Дивіться документацію `codecs.register_error`.

ASCII є спільною підмножиною для всіх кодувань, про які мені відомо, тому кодування завжди має вдаватися, якщо текст складається виключно з ASCII-символів. Python 3.7 додав новий булевий метод `str.isascii()` для перевірки того, чи є ваш текст Unicode на 100% чистим ASCII. Якщо так, ви повинні мати можливість закодувати його у байти в будь-якому кодуванні без виникнення `UnicodeEncodeError`.

### Робота з UnicodeDecodeError

Не кожний байт є дійсним символом ASCII, і не кожна послідовність байтів є дійсним UTF-8 або UTF-16; тому при припущенні одного з цих кодувань під час перетворення бінарної послідовності в текст ви отримаєте `UnicodeDecodeError`, якщо будуть виявлені несподівані байти.

З іншого боку, багато застарілих 8-бітних кодувань, наприклад `'cp1252'`, `'iso8859_1'` та `'koi8_r'`, здатні декодувати будь-який потік байтів, зокрема випадковий шум, без повідомлення про помилки. Тому якщо ваша програма припускає неправильне 8-бітне кодування, вона мовчки декодуватиме сміття.

> Спотворені символи відомі як gremlins або мозибаке (文字化け — японською «перетворений текст»).

Приклад 4-6 ілюструє, як використання неправильного кодека може призвести до gremlin-символів або `UnicodeDecodeError`.

**Приклад 4-6.** Декодування зі `str` у байти: успіх та обробка помилок

```python
>>> octets = b'Montr\xe9al'
>>> octets.decode('cp1252')
'Montréal'
>>> octets.decode('iso8859_7')
'Montrιal'
>>> octets.decode('koi8_r')
'MontrИal'
>>> octets.decode('utf_8')
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
UnicodeDecodeError: 'utf-8' codec can't decode byte 0xe9 in position 5:
    invalid continuation byte
>>> octets.decode('utf_8', errors='replace')
'Montr\ufffdal'
```

Слово «Montréal», закодоване у latin1; `'\xe9'` — байт для «é». Декодування за допомогою Windows 1252 вдається, бо це надмножина latin1. ISO-8859-7 призначений для грецької мови, тому байт `'\xe9'` неправильно інтерпретується, і помилок не виникає. KOI8-R — для російської. Тепер `'\xe9'` означає кирилічну літеру «И». Кодек `'utf_8'` виявляє, що `octets` не є дійсним UTF-8, і викликає `UnicodeDecodeError`. За допомогою обробки помилок `'replace'` `\xe9` замінюється на «» (кодова точка U+FFFD) — офіційний СИМВОЛ ЗАМІНИ Unicode, призначений для представлення невідомих символів.

### SyntaxError при завантаженні модулів із неочікуваним кодуванням

UTF-8 є типовим кодуванням джерела для Python 3, так само як ASCII було типовим для Python 2. Якщо завантажити модуль `.py`, що містить дані не в UTF-8, без оголошення кодування, отримаєте повідомлення на зразок:

```
SyntaxError: Non-UTF-8 code starting with '\xe1' in file ola.py on line
1, but no encoding declared; see https://python.org/dev/peps/pep-0263/
for details
```

Оскільки UTF-8 широко застосовується в системах GNU/Linux та macOS, вірогідним сценарієм є відкриття файлу `.py`, створеного на Windows з cp1252. Зверніть увагу, що ця помилка виникає навіть у Python для Windows, тому що типовим кодуванням джерела Python 3 є UTF-8 на всіх платформах.

Щоб виправити цю проблему, додайте магічний коментар кодування на початку файлу, як показано в Прикладі 4-7.

**Приклад 4-7.** `ola.py`: «Olá, Mundo!» португальською

```python
# coding: cp1252
print('Olá, Mundo!')
```

> Оскільки вихідний код Python 3 більше не обмежений ASCII і типово використовує чудове кодування UTF-8, найкращим «виправленням» для вихідного коду у застарілих кодуваннях, наприклад `'cp1252'`, є конвертація в UTF-8 вже зараз, без турботи про коментарі кодування. Якщо ваш редактор не підтримує UTF-8, час його змінити.

### Як визначити кодування послідовності байтів

Як знайти кодування послідовності байтів? Коротка відповідь: ніяк. Вам мають про нього повідомити.

Деякі комунікаційні протоколи та формати файлів, наприклад HTTP і XML, містять заголовки, що явно вказують, яким чином закодовано вміст. Можна бути впевненими, що деякі потоки байтів не є ASCII, бо вони містять значення байтів вище 127, а спосіб побудови UTF-8 і UTF-16 також обмежує можливі послідовності байтів.

---

> **Лайфхак Лео для вгадування декодування UTF-8**
>
> *(Наступні абзаци є нотаткою технічного рецензента Леонардо Рошаеля у чернетці цієї книги.)*
>
> За задумом UTF-8 майже неможливо, щоб випадкова послідовність байтів або навіть невипадкова послідовність байтів із кодування не UTF-8 була декодована як сміття у UTF-8 замість того, щоб викликати `UnicodeDecodeError`. Причина в тому, що escape-послідовності UTF-8 ніколи не використовують ASCII-символи, і ці escape-послідовності мають бітові шаблони, що дуже ускладнюють для випадкових даних бути дійсним UTF-8 випадково.
>
> Тому якщо вдається декодувати деякі байти, що містять коди > 127, як UTF-8, це, мабуть, і є UTF-8.
>
> При роботі з бразильськими онлайн-сервісами, деякі з яких були прив'язані до застарілих бекендів, мені іноді доводилося реалізовувати стратегію декодування: спробувати декодувати через UTF-8 і при `UnicodeDecodeError` декодувати через cp1252. Це було некрасиво, але ефективно.

---

Однак, якщо припустити, що потік байтів є людським звичайним текстом, можна спробувати визначити його кодування за допомогою евристики та статистики. Наприклад, якщо байти `b'\x00'` зустрічаються часто, це, мабуть, 16- або 32-бітне кодування, а не 8-бітна схема, бо нульові символи у звичайному тексті є помилками. Коли послідовність байтів `b'\x20\x00'` з'являється часто, це, швидше за все, символ пробілу (U+0020) у кодуванні UTF-16LE, а не загадковий символ U+2000 EN QUAD — хоч би що це означало.

Саме так пакет «Chardet — The Universal Character Encoding Detector» вгадує одне з більш ніж 30 підтримуваних кодувань. Chardet — це бібліотека Python, яку можна використовувати у своїх програмах, але вона також включає утиліту командного рядка `chardetect`. Ось що вона повідомляє про вихідний файл цього розділу:

```
$ chardetect 04-text-byte.asciidoc
04-text-byte.asciidoc: utf-8 with confidence 0.99
```

### BOM: корисний gremlin

У Прикладі 4-4 ви могли помітити кілька додаткових байтів на початку послідовності, закодованої в UTF-16. Ось вони знову:

```python
>>> u16 = 'El Niño'.encode('utf_16')
>>> u16
b'\xff\xfeE\x00l\x00 \x00N\x00i\x00\xf1\x00o\x00'
```

Байти `b'\xff\xfe'`. Це BOM — **мітка порядку байтів** (byte-order mark), що вказує на «little-endian»-порядок байтів процесора Intel, де виконувалося кодування.

На little-endian-машині для кожної кодової точки менш значущий байт стоїть першим: літера `'E'`, кодова точка U+0045 (десяткове 69), кодується у зсувах байтів 2 і 3 як 69 і 0:

```python
>>> list(u16)
[255, 254, 69, 0, 108, 0, 32, 0, 78, 0, 105, 0, 241, 0, 111, 0]
```

На big-endian-процесорі кодування було б оберненим; `'E'` кодувалося б як 0 і 69.

Щоб уникнути плутанини, кодування UTF-16 додає перед кодованим текстом спеціальний невидимий символ ZERO WIDTH NO-BREAK SPACE (U+FEFF). На little-endian-системі він кодується як `b'\xff\xfe'` (десяткове 255, 254). Оскільки за задумом у Unicode немає символу U+FFFE, послідовність байтів `b'\xff\xfe'` має означати ZERO WIDTH NO-BREAK SPACE у little-endian-кодуванні, тому кодек знає, який порядок байтів використовувати.

Існує варіант UTF-16 — UTF-16LE, який є явно little-endian, та інший — UTF-16BE, явно big-endian. Якщо їх використовувати, BOM не генерується:

```python
>>> u16le = 'El Niño'.encode('utf_16le')
>>> list(u16le)
[69, 0, 108, 0, 32, 0, 78, 0, 105, 0, 241, 0, 111, 0]
>>> u16be = 'El Niño'.encode('utf_16be')
>>> list(u16be)
[0, 69, 0, 108, 0, 32, 0, 78, 0, 105, 0, 241, 0, 111]
```

Якщо BOM присутній, кодек UTF-16 має відфільтрувати його, щоб ви отримали лише фактичний текстовий вміст файлу без початкового ZERO WIDTH NO-BREAK SPACE. Стандарт Unicode говорить, що якщо файл є UTF-16 і не має BOM, слід вважати, що він є UTF-16BE (big-endian). Однак архітектура Intel x86 є little-endian, тому в мережі є чимало little-endian UTF-16 без BOM.

Вся ця проблема порядку байтів стосується лише кодувань, що використовують слова довше одного байта, наприклад UTF-16 і UTF-32. Однією з великих переваг UTF-8 є те, що воно виробляє однакову послідовність байтів незалежно від порядку байтів машини, тому BOM не потрібен. Тим не менш деякі програми Windows (зокрема Notepad) все одно додають BOM до файлів UTF-8, а Excel залежить від BOM для виявлення файлу UTF-8, інакше він вважає, що вміст закодований за допомогою кодової сторінки Windows. Це кодування UTF-8 з BOM називається UTF-8-SIG у реєстрі кодеків Python. Символ U+FEFF, закодований у UTF-8-SIG, є тримайбайтовою послідовністю `b'\xef\xbb\xbf'`. Тому якщо файл починається з цих трьох байтів, він, швидше за все, є файлом UTF-8 з BOM.

---

> **Порада Калеба щодо UTF-8-SIG**
>
> Калеб Геттінґ — один із технічних рецензентів — рекомендує завжди використовувати кодек UTF-8-SIG при читанні файлів UTF-8. Це нешкідливо, бо UTF-8-SIG правильно читає файли як із BOM, так і без нього і не повертає сам BOM. При записі я рекомендую використовувати UTF-8 для загальної сумісності. Наприклад, скрипти Python можна зробити виконуваними в системах Unix, якщо вони починаються з коментаря `#!/usr/bin/env python3`. Для цього перші два байти файлу мають бути `b'#!'`, але BOM порушує цю конвенцію. Якщо є особлива потреба експортувати дані в додатки, що вимагають BOM, використовуйте UTF-8-SIG, але майте на увазі, що документація кодеків Python говорить: «У UTF-8 використання BOM не рекомендується і його загалом слід уникати».

## Обробка текстових файлів

Найкрача практика для текстового вводу-виводу — «Юнікодний сендвіч» (Рисунок 4-2).[^5] Це означає, що байти слід декодувати в `str` якнайраніше на вході (наприклад, при відкритті файлу для читання). «Начинкою» сендвіча є бізнес-логіка вашої програми, де обробка тексту виконується виключно на об'єктах `str`. Ніколи не слід кодувати чи декодувати всередині іншої обробки. На виході рядки `str` кодуються у байти якнайпізніше. Більшість вебфреймворків працюють саме так, і при їхньому використанні ми рідко торкаємося байтів. Наприклад, у Django ваші представлення мають виводити Unicode `str`; Django сам піклується про кодування відповіді у байти, використовуючи UTF-8 за замовчуванням.

[^5]: Вперше я побачив термін «Юнікодний сендвіч» у чудовій доповіді Неда Батчелдера «Pragmatic Unicode» на US PyCon 2012.

**Рисунок 4-2.** Юнікодний сендвіч: поточна найкраща практика для обробки тексту.

Python 3 спрощує дотримання поради про Юнікодний сендвіч, тому що вбудована функція `open()` виконує необхідне декодування при читанні та кодування при записі файлів у текстовому режимі, тому все, що ви отримуєте з `my_file.read()` та передаєте до `my_file.write(text)`, є об'єктами `str`.

Тому використання текстових файлів здається простим. Але якщо покладатися на типові кодування, можна потрапити в неприємності.

Розгляньте консольний сеанс у Прикладі 4-8. Чи можете ви помітити помилку?

**Приклад 4-8.** Проблема кодування платформи (якщо спробуєте це на своїй машині, проблема може проявитись або ні)

```python
>>> open('cafe.txt', 'w', encoding='utf_8').write('café')
4
>>> open('cafe.txt').read()
'cafÃ©'
```

Помилка: я вказав кодування UTF-8 при записі файлу, але не вказав його при читанні, тому Python припустив типове кодування файлів Windows — кодову сторінку 1252 — і кінцеві байти у файлі були декодовані як символи `'Ã©'` замість `'é'`.

Я запускав Приклад 4-8 на Python 3.8.1, 64 біти, на Windows 10 (збірка 18363). Ті самі інструкції, виконувані на сучасних GNU/Linux або macOS, працюють чудово, бо їхнє типове кодування — UTF-8, що створює хибне враження, ніби все гаразд. Якщо аргумент `encoding` було б опущено при відкритті файлу для запису, використовувалося б типове кодування locale, і ми б правильно прочитали файл, використовуючи те саме кодування. Але тоді цей скрипт генерував би файли з різним байтовим вмістом залежно від платформи або навіть від налаштувань locale на тій самій платформі, створюючи проблеми сумісності.

> Код, що має запускатися на кількох машинах або у різний час, ніколи не повинен покладатися на типові кодування. Завжди передавайте явний аргумент `encoding=` при відкритті текстових файлів, бо типове кодування може змінюватися від машини до машини або від дня до дня.

Цікава деталь у Прикладі 4-8: функція запису в першій інструкції повідомляє, що було записано чотири символи, але в наступному рядку читається п'ять символів. Приклад 4-9 — розширена версія Прикладу 4-8, що пояснює це та інші деталі.

**Приклад 4-9.** Ретельніший розгляд Прикладу 4-8 на Windows виявляє помилку і спосіб її виправити

```python
>>> fp = open('cafe.txt', 'w', encoding='utf_8')
>>> fp
<_io.TextIOWrapper name='cafe.txt' mode='w' encoding='utf_8'>
>>> fp.write('café')
4
>>> fp.close()
>>> import os
>>> os.stat('cafe.txt').st_size
5
>>> fp2 = open('cafe.txt')
>>> fp2
<_io.TextIOWrapper name='cafe.txt' mode='r' encoding='cp1252'>
>>> fp2.encoding
'cp1252'
>>> fp2.read()
'cafÃ©'
>>> fp3 = open('cafe.txt', encoding='utf_8')
>>> fp3
<_io.TextIOWrapper name='cafe.txt' mode='r' encoding='utf_8'>
>>> fp3.read()
'café'
>>> fp4 = open('cafe.txt', 'rb')
>>> fp4
<_io.BufferedReader name='cafe.txt'>
>>> fp4.read()
b'caf\xc3\xa9'
```

За замовчуванням `open` використовує текстовий режим і повертає об'єкт `TextIOWrapper` із конкретним кодуванням. Метод `write` на `TextIOWrapper` повертає кількість записаних символів Unicode. `os.stat` повідомляє, що файл має 5 байтів; UTF-8 кодує `'é'` як 2 байти, 0xc3 і 0xa9. Відкриття текстового файлу без явного кодування повертає `TextIOWrapper` із кодуванням, встановленим за замовчуванням locale. Об'єкт `TextIOWrapper` має атрибут `encoding`, який можна перевірити: у цьому випадку `cp1252`. У кодуванні Windows cp1252 байт 0xc3 — це «Ã» (A з тильдою), а 0xa9 — знак авторського права. Відкриття того самого файлу з правильним кодуванням. Очікуваний результат: ті самі чотири символи Unicode для `'café'`. Прапор `'rb'` відкриває файл для читання у бінарному режимі. Повернутий об'єкт є `BufferedReader`, а не `TextIOWrapper`. Читання повертає байти, як і очікується.

> Не відкривайте текстові файли у бінарному режимі, якщо тільки вам не потрібно аналізувати вміст файлу для визначення кодування — і в такому разі слід використовувати Chardet замість того, щоб винаходити велосипед (дивіться «Як визначити кодування послідовності байтів» (стор. 128)). Звичайний код повинен використовувати бінарний режим лише для відкриття бінарних файлів, наприклад растрових зображень.

Це не означає, що всі ваші улюблені емодзі відображатимуться у консолі: це також залежить від шрифту, який використовує консоль.

[^6]: Джерело: «Windows Command-Line: Unicode and UTF-8 Output Text Buffer».

Ще однією зміною став PEP 529 — Change Windows filesystem encoding to UTF-8, також впроваджений у Python 3.6, який змінив кодування файлової системи (для представлення назв каталогів і файлів) з пропрієтарного Microsoft MBCS на UTF-8.

Це означає, що скрипт на зразок Прикладу 4-12 працює при виведенні у консоль, але може зламатися при перенаправленні виведення у файл.

**Приклад 4-12.** `stdout_check.py`

```python

print()

print('sys.stdout.encoding:', sys.stdout.encoding)
print()

test_chars = [
    '\N{HORIZONTAL ELLIPSIS}',        # є в cp1252, немає в cp437
    '\N{INFINITY}',                   # є в cp437, немає в cp1252
    '\N{CIRCLED NUMBER FORTY TWO}',   # немає ні в cp437, ні в cp1252
]

    print(f'Trying to output {name(char)}:')
    print(char)
```

> У Прикладі 4-12 я використовую escape-послідовність `'\N{}'` для літералів Unicode, де пишу офіційне ім'я символу всередині `\N{}`. Це досить багатослівно, але явно й безпечно: Python викликає `SyntaxError`, якщо ім'я не існує, — набагато краще, ніж записувати шістнадцяткове число, яке може бути неправильним, але ви дізнаєтесь про це значно пізніше. Так чи інакше, напевно захотілося б написати коментар із поясненням кодів символів, тому багатослівність `\N{}` легко прийняти.

Для підведення підсумку цього неприємного питання типових кодувань розглянемо різні кодування в Прикладі 4-11:

- Якщо пропустити аргумент `encoding` при відкритті файлу, типове значення визначається `locale.getpreferredencoding()` (`'cp1252'` у Прикладі 4-11).
- Кодування `sys.stdout|stdin|stderr` раніше задавалося змінною середовища `PYTHONIOENCODING` до Python 3.6 — тепер ця змінна ігнорується, якщо тільки `PYTHONLEGACYWINDOWSSTDIO` не встановлено непорожнім рядком. Інакше кодування стандартного вводу-виводу є UTF-8 для інтерактивного вводу-виводу, або визначається `locale.getpreferredencoding()`, якщо виведення/введення перенаправляється у файл/із файлу.
- `sys.getdefaultencoding()` використовується Python внутрішньо для неявних перетворень бінарних даних у/зі `str`. Зміна цього параметра не підтримується.
- `sys.getfilesystemencoding()` використовується для кодування/декодування імен файлів (не вмісту файлів). Він використовується, коли `open()` отримує аргумент типу `str` для імені файлу; якщо ім'я файлу передається як аргумент типу `bytes`, воно передається без змін до API ОС.

> На GNU/Linux та macOS всі ці кодування за замовчуванням встановлені на UTF-8, і так було вже кілька років, тому вводу-виводу обробляє всі символи Unicode. На Windows не тільки в одній системі використовуються різні кодування, але й зазвичай це кодові сторінки на зразок `'cp850'` або `'cp1252'`, що підтримують лише ASCII із 127 додатковими символами, що різняться від кодування до кодування. Тому користувачі Windows значно частіше стикаються з помилками кодування, якщо не бути особливо уважними.

Підсумовуючи, найважливішим параметром кодування є той, що повертає `locale.getpreferredencoding()`: він є типовим при відкритті текстових файлів і для `sys.stdout/stdin/stderr`, коли вони перенаправляються у файли. Однак у документації (частково) написано:

```
locale.getpreferredencoding(do_setlocale=True)
```

> Повертає кодування, що використовується для текстових даних відповідно до уподобань користувача. Уподобання користувача виражаються по-різному в різних системах і можуть бути недоступні програмно в деяких системах, тому ця функція повертає лише здогад. [...]

Тому найкраща порада щодо типових кодувань: **не покладайтеся на них**.

Ви уникнете багато болю, якщо будете дотримуватися поради про Юнікодний сендвіч і завжди явно вказуватимете кодування у своїх програмах. На жаль, Unicode є болісним навіть якщо правильно перетворювати байти у `str`. Наступні два розділи охоплюють теми, що є простими в ASCII-просторі, але стають досить складними на планеті Unicode: нормалізація тексту (тобто приведення тексту до однорідного представлення для порівнянь) та сортування.

## Нормалізація Unicode для надійних порівнянь

Порівняння рядків ускладнюється тим, що Unicode має комбіновані символи: діакритичні знаки та інші позначки, що прикріплюються до попереднього символу і виглядають як одне ціле при виведенні на друк.

Наприклад, слово «café» може бути складено двома способами, використовуючи чотири або п'ять кодових точок, але результат виглядає однаково:

```python
>>> s1 = 'café'
>>> s2 = 'cafe\N{COMBINING ACUTE ACCENT}'
>>> s1, s2
('café', 'café')
>>> len(s1), len(s2)
(4, 5)
>>> s1 == s2
False
```

Розміщення COMBINING ACUTE ACCENT (U+0301) після «e» відображається як «é». У стандарті Unicode послідовності на зразок `'é'` і `'e\u0301'` називаються «канонічними еквівалентами», і програми мають трактувати їх як однакові. Але Python бачить дві різних послідовності кодових точок і вважає їх нерівними.

Рішення — `unicodedata.normalize()`. Перший аргумент цієї функції — один із чотирьох рядків: `'NFC'`, `'NFD'`, `'NFKC'` та `'NFKD'`. Почнемо з перших двох.

Форма нормалізації C (NFC) складає кодові точки для отримання найкоротшого еквівалентного рядка, тоді як NFD розкладає, розкриваючи складені символи на базові символи та окремі комбіновані символи. Обидві ці нормалізації роблять порівняння такими, як слід, як показує наступний приклад:

```python
>>> from unicodedata import normalize
>>> s1 = 'café'
>>> s2 = 'cafe\N{COMBINING ACUTE ACCENT}'
>>> len(s1), len(s2)
(4, 5)
>>> len(normalize('NFC', s1)), len(normalize('NFC', s2))
(4, 4)
>>> len(normalize('NFD', s1)), len(normalize('NFD', s2))
(5, 5)
>>> normalize('NFC', s1) == normalize('NFC', s2)
True
>>> normalize('NFD', s1) == normalize('NFD', s2)
True
```

Драйвери клавіатур зазвичай генерують складені символи, тому текст, введений користувачами, за замовчуванням буде у форматі NFC. Однак для безпеки доцільно нормалізувати рядки за допомогою `normalize('NFC', user_text)` перед збереженням. NFC також є формою нормалізації, рекомендованою W3C у «Character Model for the World Wide Web: String Matching and Searching».

Деякі одиничні символи нормалізуються NFC в інший одиничний символ. Символ для одиниці омського електричного опору (Ω) нормалізується до грецької великої омеги. Вони візуально ідентичні, але порівнюються як нерівні, тому нормалізація є необхідною для уникнення несподіванок:

```python
>>> from unicodedata import normalize, name
>>> ohm = '\u2126'
>>> name(ohm)
'OHM SIGN'
>>> ohm_c = normalize('NFC', ohm)
>>> name(ohm_c)
'GREEK CAPITAL LETTER OMEGA'
>>> ohm == ohm_c
False
>>> normalize('NFC', ohm) == normalize('NFC', ohm_c)
True
```

Інші дві форми нормалізації — NFKC і NFKD, де літера K означає «сумісність». Це більш потужні форми нормалізації, що впливають на так звані «символи сумісності». Хоча однією з цілей Unicode є наявність єдиної «канонічної» кодової точки для кожного символу, деякі символи з'являються більш ніж один раз для сумісності з наявними стандартами. Наприклад, МІКРОЗНАК, µ (U+00B5), був доданий до Unicode для підтримки двостороннього перетворення з latin1, що його включає, хоча той самий символ є частиною грецького алфавіту з кодовою точкою U+03BC (ГРЕЦЬКА МАЛА ЛІТЕРА МЮ). Тому знак мікро вважається «символом сумісності».

У формах NFKC і NFKD кожний символ сумісності замінюється «розкладанням сумісності» з одного або кількох символів, що вважаються «кращим» представленням, навіть якщо при цьому відбувається деяка втрата форматування — в ідеалі форматування має бути відповідальністю зовнішньої розмітки, а не частиною Unicode. Наприклад, розкладання сумісності дробу-половини `'½'` (U+00BD) — це послідовність трьох символів `'1/2'`, а розкладання сумісності знаку мікро `'µ'` (U+00B5) — це мала грецька мю `'μ'` (U+03BC).[^7]

[^7]: Цікаво, що знак мікро вважається «символом сумісності», а символ ом — ні. В підсумку: NFC не чіпає знак мікро, але змінює символ ом на велику омегу, тоді як NFKC та NFKD змінюють і ом, і мікро на грецькі символи.

Ось як NFKC працює на практиці:

```python
>>> from unicodedata import normalize, name
>>> half = '\N{VULGAR FRACTION ONE HALF}'
>>> print(half)
½
>>> normalize('NFKC', half)
'1⁄2'
>>> for char in normalize('NFKC', half):
...     print(char, name(char), sep='\t')
...
1	DIGIT ONE
⁄	FRACTION SLASH
2	DIGIT TWO
>>> four_squared = '4²'
>>> normalize('NFKC', four_squared)
'42'
>>> micro = 'µ'
>>> micro_kc = normalize('NFKC', micro)
>>> micro, micro_kc
('µ', 'μ')
>>> ord(micro), ord(micro_kc)
(181, 956)
>>> name(micro), name(micro_kc)
('MICRO SIGN', 'GREEK SMALL LETTER MU')
```

Хоча `'1⁄2'` є прийнятною заміною для `'½'`, а знак мікро насправді є малою грецькою мю, перетворення `'4²'` на `'42'` змінює значення. Програма може зберігати `'4²'` як `'4<sup>2</sup>'`, але функція `normalize` нічого не знає про форматування. Тому NFKC або NFKD можуть втрачати або спотворювати інформацію, але можуть виробляти зручні проміжні представлення для пошуку та індексування.

> NFKC та NFKD нормалізація спричиняє втрату даних і має застосовуватися лише в особливих випадках, наприклад при пошуку та індексуванні, а не для постійного зберігання тексту.

### Згортання регістру

Згортання регістру — це по суті приведення всього тексту до нижнього регістру з деякими додатковими трансформаціями. Воно підтримується методом `str.casefold()`.

Для будь-якого рядка `s`, що містить лише символи latin1, `s.casefold()` виробляє той самий результат, що й `s.lower()`, з лише двома винятками: знак мікро `'µ'` замінюється на грецьку малу мю (яка виглядає так само в більшості шрифтів), а German Eszett або «гостре s» (ß) стає «ss»:

```python
>>> micro = 'µ'
>>> name(micro)
'MICRO SIGN'
>>> micro_cf = micro.casefold()
>>> name(micro_cf)
'GREEK SMALL LETTER MU'
>>> micro, micro_cf
('µ', 'μ')
>>> eszett = 'ß'
>>> name(eszett)
'LATIN SMALL LETTER SHARP S'
>>> eszett_cf = eszett.casefold()
>>> eszett, eszett_cf
('ß', 'ss')
```

Існує майже 300 кодових точок, для яких `str.casefold()` і `str.lower()` повертають різні результати.

### Утилітні функції для порівняння нормалізованих рядків

Як ми бачили, NFC та NFD безпечні у використанні й дозволяють розумні порівняння рядків Unicode. NFC є найкращою нормалізованою формою для більшості застосунків. `str.casefold()` — правильний вибір для нечутливих до регістру порівнянь.

Якщо ви працюєте з текстом багатьох мов, пара функцій, наприклад `nfc_equal` і `fold_equal` у Прикладі 4-13, є корисним доповненням до вашого набору інструментів.

**Приклад 4-13.** `normeq.py`: нормалізоване порівняння рядків Unicode

```python
"""
Utility functions for normalized Unicode string comparison.
Using Normal Form C, case sensitive:

>>> s1 = 'café'
>>> s2 = 'cafe\u0301'
>>> s1 == s2
False
>>> nfc_equal(s1, s2)
True
>>> nfc_equal('A', 'a')
False

Using Normal Form C with case folding:

>>> s3 = 'Straße'
>>> s4 = 'strasse'
>>> s3 == s4
False
>>> nfc_equal(s3, s4)
False
>>> fold_equal(s3, s4)
True
>>> fold_equal(s1, s2)
True
>>> fold_equal('A', 'a')
True

"""
from unicodedata import normalize

def nfc_equal(str1, str2):
    return normalize('NFC', str1) == normalize('NFC', str2)

def fold_equal(str1, str2):
    return (normalize('NFC', str1).casefold() ==
            normalize('NFC', str2).casefold())
```

### Крайня «нормалізація»: видалення діакритики

Секретний соус Google Search включає безліч прийомів, але один із них — очевидно, ігнорування діакритичних знаків (наприклад, акцентів, седилів тощо), принаймні в деяких контекстах. Видалення діакритичних знаків не є належною формою нормалізації, оскільки нерідко змінює значення слів і може призводити до хибних спрацювань при пошуку. Але це допомагає справлятися з деякими фактами: люди іноді ледачі або неосвічені щодо правильного використання діакритики, а правила правопису з часом змінюються — акценти з'являються і зникають у живих мовах.

Поза пошуком, позбавлення діакритичних знаків також дає більш читабельні URL, принаймні у мовах на латинській основі. Подивіться на URL статті Вікіпедії про місто Сан-Паулу:

```
https://en.wikipedia.org/wiki/S%C3%A3o_Paulo
```

Частина `%C3%A3` — це URL-закодоване, UTF-8 представлення одної літери «ã» («a» з тильдою). Наступний URL значно легше впізнати, навіть якщо він не є правильним написанням:

```
https://en.wikipedia.org/wiki/Sao_Paulo
```

Для видалення всіх діакритичних знаків зі `str` можна використати функцію на зразок Прикладу 4-14.

**Приклад 4-14.** `simplify.py`: функція для видалення всіх комбінованих знаків

```python

                     if not unicodedata.combining(c))
    return unicodedata.normalize('NFC', shaved)

Часто причиною видалення діакритичних знаків є прагнення перетворити латинський текст у чистий ASCII, але `shave_marks` також змінює нелатинські символи, наприклад грецькі літери, які ніколи не стануть ASCII лише від видалення акцентів. Тому має сенс аналізувати кожний базовий символ і видаляти прикріплені знаки лише якщо базовий символ є літерою латинського алфавіту. Саме це робить Приклад 4-16.

**Приклад 4-16.** Функція для видалення комбінованих знаків із латинських символів (інструкції `import` опущені, бо це частина модуля `simplify.py` із Прикладу 4-14)

```python
def shave_marks_latin(txt):
    """Remove all diacritic marks from Latin base characters"""
    # Видалити всі діакритичні знаки з латинських базових символів
    norm_txt = unicodedata.normalize('NFD', txt)
    latin_base = False
    preserve = []
    for c in norm_txt:
        if unicodedata.combining(c) and latin_base:
            continue  # ігнорувати діакритику на латинському базовому символі
        preserve.append(c)
        # якщо це не комбінований символ, це новий базовий символ
        if not unicodedata.combining(c):
            latin_base = c in string.ascii_letters
    shaved = ''.join(preserve)
    return unicodedata.normalize('NFC', shaved)
```

Розкласти всі символи на базові та комбіновані. Пропускати комбіновані знаки, якщо базовий символ є латинським. Інакше зберігати поточний символ. Визначити новий базовий символ і перевірити, чи є він латинським. Повторно скласти всі символи.

## Сортування тексту Unicode

Python сортує послідовності будь-якого типу, порівнюючи елементи в кожній послідовності по одному. Для рядків це означає порівняння кодових точок. На жаль, це дає неприйнятні результати для тих, хто використовує символи не-ASCII.

Розглянемо сортування списку фруктів, що вирощуються в Бразилії:

```python
>>> fruits = ['caju', 'atemoia', 'cajá', 'açaí', 'acerola']
>>> sorted(fruits)
['acerola', 'atemoia', 'açaí', 'caju', 'cajá']
```

Правила сортування відрізняються для різних локалей, але в португальській та багатьох мовах, що використовують латинський алфавіт, акценти та седилі рідко впливають на сортування.[^8] Тому «cajá» сортується як «caja» і має стояти перед «caju».

[^8]: Діакритика впливає на сортування лише у рідкісному випадку, коли вона є єдиною відмінністю між двома словами — тоді слово з діакритикою сортується після звичайного слова.

Відсортований список фруктів має бути:

```python
['açaí', 'acerola', 'atemoia', 'cajá', 'caju']
```

Стандартний спосіб сортування тексту не-ASCII у Python — використання функції `locale.strxfrm`, яка, відповідно до документації модуля `locale`, «перетворює рядок на такий, що можна використовувати в порівняннях з урахуванням locale».

Для ввімкнення `locale.strxfrm` спочатку потрібно встановити відповідний locale для вашого застосунку. Послідовність команд у Прикладі 4-19 може вам підійти.

**Приклад 4-19.** `locale_sort.py`: використання функції `locale.strxfrm` як ключа сортування

```python
import locale
my_locale = locale.setlocale(locale.LC_COLLATE, 'pt_BR.UTF-8')
print(my_locale)
fruits = ['caju', 'atemoia', 'cajá', 'açaí', 'acerola']
sorted_fruits = sorted(fruits, key=locale.strxfrm)
print(sorted_fruits)
```

Запуск Прикладу 4-19 на GNU/Linux (Ubuntu 19.10) із встановленим locale `pt_BR.UTF-8` дає правильний результат:

```
'pt_BR.UTF-8'
['açaí', 'acerola', 'atemoia', 'cajá', 'caju']
```

Тому потрібно викликати `setlocale(LC_COLLATE, «ваш_locale»)` перед використанням `locale.strxfrm` як ключа при сортуванні.

Однак є деякі застереження:

- Оскільки налаштування locale є глобальними, виклик `setlocale` у бібліотеці не рекомендується. Ваш застосунок або фреймворк має встановлювати locale при запуску процесу і не змінювати його після цього.
- Locale має бути встановлено в ОС, інакше `setlocale` викликає виняток `locale.Error: unsupported locale setting`.
- Потрібно знати, як правильно написати ім'я locale.
- Locale має бути правильно реалізоване розробниками ОС. Я досяг успіху на Ubuntu 19.10, але не на macOS 10.14. На macOS виклик `setlocale(LC_COLLATE, 'pt_BR.UTF-8')` повертає рядок `'pt_BR.UTF-8'` без скарг. Але `sorted(fruits, key=locale.strxfrm)` давав той самий неправильний результат, що й `sorted(fruits)`. Я також спробував locale `fr_FR`, `es_ES` та `de_DE` на macOS, але `locale.strxfrm` ніколи не виконував свою роботу.[^9]

[^9]: Знову ж таки, я не зміг знайти рішення, але знайшов інших людей, які повідомляли про ту саму проблему. Алекс Мартеллі, один із технічних рецензентів, не мав проблем із використанням `setlocale` та `locale.strxfrm` на своєму Mac із macOS 10.9. Підсумок: результати можуть відрізнятися.

Тому стандартне бібліотечне рішення для інтернаціонального сортування працює, але, схоже, добре підтримується лише на GNU/Linux (можливо, також на Windows, якщо ви є експертом). Навіть тоді воно залежить від налаштувань locale, що створює проблеми при розгортанні.

На щастя, є простіше рішення: бібліотека `pyuca`, доступна на PyPI.

## База даних Unicode

Стандарт Unicode надає цілу базу даних — у вигляді кількох структурованих текстових файлів, — яка включає не лише таблицю відповідності кодових точок іменам символів, але й метадані про окремі символи та їхні взаємозв'язки. Наприклад, база даних Unicode фіксує, чи є символ друкованим, чи є він літерою, десятковою цифрою або іншим числовим символом. Саме так працюють методи `str` `isalpha`, `isprintable`, `isdecimal` та `isnumeric`. `str.casefold` також використовує інформацію з таблиці Unicode.

> Функція `unicodedata.category(char)` повертає двобуквену категорію `char` із бази даних Unicode. Методи `str` вищого рівня простіші у використанні. Наприклад, `label.isalpha()` повертає `True`, якщо кожний символ у `label` належить до однієї з цих категорій: `Lm`, `Lt`, `Lu`, `Ll` або `Lo`. Щоб дізнатися, що означають ці коди, дивіться «General Category» у статті Wikipedia «Unicode character property».

Завдяки багатому API множин Python, нам не потрібен вкладений цикл `for` та ще один `if` для реалізації цієї перевірки.

**Приклад 4-21.** `cf.py`: утиліта пошуку символів

```python
#!/usr/bin/env python3
import sys
import unicodedata

START, END = ord(' '), sys.maxunicode + 1

def find(*query_words, start=START, end=END):
    query = {w.upper() for w in query_words}
    for code in range(start, end):
        char = chr(code)
        name = unicodedata.name(char, None)
        if name and query.issubset(name.split()):
            print(f'U+{code:04X}\t{char}\t{name}')

def main(words):
    if words:
        find(*words)
    else:
        print('Please provide words to find.')

if __name__ == '__main__':
    main(sys.argv[1:])
```

Встановити типові значення для діапазону кодових точок для пошуку. `find` приймає `query_words` та необов'язкові аргументи лише за ключовим словом для обмеження діапазону пошуку, щоб полегшити тестування. Перетворити `query_words` на множину рядків у верхньому регістрі. Отримати символ Unicode для `code`. Отримати ім'я символу або `None`, якщо кодова точка не призначена. Якщо ім'я є, розбити його на список слів і перевірити, що множина запиту є підмножиною цього списку. Вивести рядок із кодовою точкою у форматі U+9999, символом та його іменем.

### Числове значення символів

Модуль `unicodedata` включає функції для перевірки того, чи представляє символ Unicode число, і якщо так, його числове значення для людини — на відміну від номера кодової точки. Приклад 4-22 показує використання `unicodedata.name()` та `unicodedata.numeric()`, а також методів `.isdecimal()` і `.isnumeric()` типу `str`.

**Приклад 4-22.** Демонстрація числових метаданих символів Unicode (виноски описують кожний стовпець виведення)

```python
import unicodedata
import re

re_digit = re.compile(r'\d')
sample = '1\xbc\xb2\u0969\u136b\u216b\u2466\u2480\u3285'
for char in sample:
    print(f'U+{ord(char):04x}',
          char.center(6),
          're_dig' if re_digit.match(char) else '-',
          'isdig'  if char.isdigit()       else '-',
          'isnum'  if char.isnumeric()     else '-',
          f'{unicodedata.numeric(char):5.2f}',
          unicodedata.name(char),
          sep='\t')
```

Кодова точка у форматі U+0000. Символ, вирівняний по центру в рядку довжиною 6. Показати `re_dig`, якщо символ відповідає регексу `r'\d'`. Показати `isdig`, якщо `char.isdigit()` є `True`. Показати `isnum`, якщо `char.isnumeric()` є `True`. Числове значення з шириною 5 і двома десятковими знаками. Ім'я символу Unicode.

**Рисунок 4-7.** Термінал macOS показує числові символи та метадані про них; `re_dig` означає, що символ відповідає регулярному виразу `r'\d'`.

Шостий стовпець Рисунку 4-7 — результат виклику `unicodedata.numeric(char)` для символу. Він показує, що Unicode знає числове значення символів, що представляють числа. Тому якщо хочете створити застосунок для роботи з електронними таблицями, що підтримує тамільські цифри або римські числа — вперед!

Рисунок 4-7 показує, що регулярний вираз `r'\d'` відповідає цифрі «1» та цифрі Деванагарі 3, але не деяким іншим символам, що вважаються цифрами функцією `isdigit`. Модуль `re` не настільки розумний щодо Unicode, як міг би бути. Новий модуль `regex`, доступний на PyPI, розроблений для поступової заміни `re` та надає кращу підтримку Unicode.[^11] Ми повернемося до модуля `re` в наступному розділі.

[^11]: Хоча у цьому конкретному прикладі він не був кращим за `re` при розпізнаванні цифр.

## Двохрежимні API str і bytes

Стандартна бібліотека Python має функції, що приймають аргументи `str` або `bytes` і поводяться по-різному залежно від типу. Деякі приклади можна знайти у модулях `re` та `os`.

### str проти bytes у регулярних виразах

Якщо регулярний вираз побудований із `bytes`, шаблони на зразок `\d` і `\w` відповідають лише ASCII-символам; якщо ж ці шаблони задані як `str`, вони відповідають цифрам або літерам Unicode поза ASCII. Приклад 4-23 і Рисунок 4-8 порівнюють, як букви, ASCII-цифри, надрядкові та тамільські цифри відповідають шаблонам `str` і `bytes`.

**Приклад 4-23.** `ramanujan.py`: порівняння поведінки простих регулярних виразів `str` і `bytes`

```python
import re

re_numbers_str = re.compile(r'\d+')
re_words_str = re.compile(r'\w+')
re_numbers_bytes = re.compile(rb'\d+')
re_words_bytes = re.compile(rb'\w+')

text_str = ("Ramanujan saw \u0be7\u0bed\u0be8\u0bef"
            " as 1729 = 1³ + 12³ = 9³ + 10³.")
text_bytes = text_str.encode('utf_8')

print(f'Text\n  {text_str!r}')
print('Numbers')
print('  str  :', re_numbers_str.findall(text_str))
print('  bytes:', re_numbers_bytes.findall(text_bytes))
print('Words')
print('  str  :', re_words_str.findall(text_str))
print('  bytes:', re_words_bytes.findall(text_bytes))
```

Перші два регулярних вирази є типу `str`. Останні два — типу `bytes`. Текст Unicode для пошуку, що містить тамільські цифри для числа 1729 (логічний рядок продовжується до токена правої дужки). Цей рядок об'єднується з попереднім під час компіляції. Для пошуку за допомогою `bytes`-регулярних виразів потрібен `bytes`-рядок. Шаблон `str` `r'\d+'` відповідає тамільським та ASCII-цифрам. Шаблон `bytes` `rb'\d+'` відповідає лише ASCII-байтам для цифр. Шаблон `str` `r'\w+'` відповідає літерам, надрядковим символам, тамільським та ASCII-цифрам. Шаблон `bytes` `rb'\w+'` відповідає лише ASCII-байтам для літер та цифр.

**Рисунок 4-8.** Знімок екрана запуску `ramanujan.py` із Прикладу 4-23.

Приклад 4-23 — тривіальний приклад для демонстрації одного факту: регулярні вирази можна використовувати для `str` і `bytes`, але в другому випадку байти поза діапазоном ASCII трактуються як нецифри та несловесні символи.

Для `str`-регулярних виразів є прапорець `re.ASCII`, що змушує `\w`, `\W`, `\b`, `\B`, `\d`, `\D`, `\s` та `\S` виконувати відповідність лише з ASCII. Дивіться документацію модуля `re` для детальної інформації.

### str проти bytes у функціях os

Ядро GNU/Linux не підтримує Unicode, тому в реальному світі можна зустріти імена файлів, що складаються з байтових послідовностей, які не є дійсними в жодній розумній схемі кодування та не можуть бути декодовані у `str`. Файлові сервери з клієнтами, що використовують різні ОС, особливо схильні до цієї проблеми.

Щоб обійти цю проблему, всі функції модуля `os`, що приймають імена файлів або шляхів, приймають аргументи типу `str` або `bytes`. Якщо така функція викликається з аргументом типу `str`, аргумент буде автоматично перетворено за допомогою кодека, іменованого `sys.getfilesystemencoding()`, а відповідь ОС буде декодована тим самим кодеком. Це майже завжди те, що потрібно, відповідно до найкращої практики Юнікодного сендвіча.

Але якщо необхідно мати справу з (і, можливо, виправляти) іменами файлів, які не можна обробити таким чином, можна передавати аргументи типу `bytes` до функцій `os`, щоб отримати значення типу `bytes`. Ця можливість дозволяє обробляти будь-який файл або шлях, скільки б gremlin-символів там не було. Дивіться Приклад 4-24.

**Приклад 4-24.** `listdir` з аргументами і результатами типів `str` і `bytes`

```python
>>> os.listdir('.')
['abc.txt', 'digits-of-π.txt']
>>> os.listdir(b'.')
[b'abc.txt', b'digits-of-\xcf\x80.txt']
```

Друге ім'я файлу — «digits-of-π.txt» (із грецькою літерою пі). При аргументі типу `bytes` `listdir` повертає імена файлів як `bytes`: `b'\xcf\x80'` — це UTF-8-кодування грецької літери пі.

Для допомоги при ручній обробці послідовностей `str` або `bytes`, що є іменами файлів або шляхів, модуль `os` надає спеціальні функції кодування та декодування `os.fsencode(name_or_path)` та `os.fsdecode(name_or_path)`. Обидві ці функції приймають аргумент типу `str`, `bytes` або об'єкт, що реалізує інтерфейс `os.PathLike`, починаючи з Python 3.6.

## Підсумок розділу

Ми почали розділ, спростувавши уявлення, що 1 символ == 1 байт. Оскільки світ переходить на Unicode, нам потрібно тримати концепцію текстових рядків відокремленою від бінарних послідовностей, що їх представляють у файлах, а Python 3 забезпечує це розмежування.

Після короткого огляду типів даних бінарних послідовностей — `bytes`, `bytearray` і `memoryview` — ми перейшли до кодування та декодування, з огляду важливих кодеків, після чого розглянули підходи до запобігання або обробки сумнозвісних помилок `UnicodeEncodeError`, `UnicodeDecodeError` та `SyntaxError`, спричиненої неправильним кодуванням у вихідних файлах Python.

Потім ми розглянули теорію і практику виявлення кодування за відсутності метаданих: теоретично це неможливо, але на практиці пакет Chardet справляється з цим досить добре для ряду популярних кодувань. Мітки порядку байтів були представлені як єдина підказка щодо кодування, що зазвичай зустрічається у файлах UTF-16 і UTF-32, а іноді і в файлах UTF-8.

У наступному розділі ми продемонстрували відкриття текстових файлів — просте завдання, за винятком однієї пастки: аргумент `encoding=` не є обов'язковим при відкритті текстового файлу, але повинен бути. Якщо не вказати кодування, отримаєте програму, яка генерує «звичайний текст», несумісний між платформами через конфліктуючі типові кодування. Ми потім розкрили різні параметри кодування, що Python використовує за замовчуванням, і способи їх виявлення. Сумний факт для користувачів Windows: ці параметри нерідко мають різні значення на одній машині, і ці значення є взаємно несумісними; користувачі GNU/Linux і macOS, навпаки, живуть у більш щасливому місці, де UTF-8 є типовим майже скрізь.

Unicode надає кілька способів представлення деяких символів, тому нормалізація є обов'язковою умовою для зіставлення тексту. Крім пояснення нормалізації та zgortannya регістру, ми представили деякі утилітні функції, які ви можете адаптувати до своїх потреб, включно з різкими трансформаціями, як-от видалення всіх акцентів. Потім ми побачили, як правильно сортувати текст Unicode, використовуючи стандартний модуль `locale` — із деякими застереженнями — та альтернативу, що не залежить від хитрих налаштувань locale: зовнішній пакет `pyuca`.

Ми використали базу даних Unicode для програмування утиліти командного рядка для пошуку символів за іменем — у 28 рядках коду завдяки потужності Python. Ми побіжно розглянули інші метадані Unicode та зробили короткий огляд двохрежимних API, де деякі функції можуть бути викликані з аргументами `str` або `bytes`, виробляючи різні результати.

## Трибуна

### Не-ASCII-імена у вихідному коді: варто їх використовувати?

Python 3 дозволяє не-ASCII-ідентифікатори у вихідному коді:

```python
>>> ação = 'PBR'  # ação = акція
>>> ε = 10**-6    # ε = епсилон
```

Деякі люди не люблять цю ідею. Найпоширеніший аргумент на користь ASCII-ідентифікаторів — полегшити читання та редагування коду для всіх. Цей аргумент упускає суть: вихідний код має бути зрозумілим і доступним для редагування своїй цільовій аудиторії, а не «для всіх». Якщо код належить транснаціональній корпорації або є відкритим вихідним кодом і ви хочете контриб'юторів з усього світу, ідентифікатори мають бути англійською — і тоді ASCII буде достатньо.

Але якщо ви викладач у Бразилії, вашим студентам буде легше читати код, що використовує імена змінних і функцій португальською, правильно написаних. І вони не матимуть труднощів із введенням седилів та акцентованих голосних на своїх локалізованих клавіатурах.

Тепер, коли Python може обробляти імена Unicode і UTF-8 є типовим кодуванням джерела, я не бачу сенсу в написанні ідентифікаторів португальською без акцентів, як ми раніше робили в Python 2 за необхідністю — якщо тільки не потрібно, щоб код також запускався на Python 2. Якщо імена є португальськими, відмова від акцентів не зробить код більш читабельним для кого б то не було.

Це моя точка зору як португаломовного бразильця, але я вважаю, що вона застосовна між кордонами та культурами: обирайте людську мову, яка робить код більш зрозумілим для команди, а потім використовуйте символи, необхідні для правильного написання.

### Що таке «звичайний текст»?

Для тих, хто щодня має справу з нeanglomовним текстом, «звичайний текст» не означає «ASCII». Глосарій Unicode визначає звичайний текст таким чином:

> Комп'ютерно-кодований текст, що складається лише з послідовності кодових точок із даного стандарту, без будь-якої іншої інформації про форматування чи структуру.

Це визначення починається дуже добре, але я не погоджуюся з частиною після коми. HTML є чудовим прикладом формату звичайного тексту, що несе інформацію про форматування та структуру. Але це все одно звичайний текст, тому що кожний байт у такому файлі присутній для представлення текстового символу, зазвичай за допомогою UTF-8. Немає байтів із нетекстовим значенням, як у документі `.png` або `.xls`, де більшість байтів представляють упаковані бінарні значення — наприклад, значення RGB та числа з рухомою крапкою. У звичайному тексті числа представляються як послідовності символів цифр.

Я пишу цю книгу у форматі звичайного тексту, що іронічно називається AsciiDoc, — частині видавничої платформи O'Reilly. Вихідні файли AsciiDoc є звичайним текстом, але вони є UTF-8, а не ASCII. Інакше написання цього розділу було б справді болісним. Попри назву, AsciiDoc — просто чудовий формат.

### Як кодові точки str зберігаються в RAM?

Офіційна документація Python уникає питання про те, як кодові точки `str` зберігаються в пам'яті. Це справді деталь реалізації. Теоретично це не має значення: яким би не було внутрішнє представлення, кожний `str` повинен бути закодований у байти на виході.

В пам'яті Python 3 зберігає кожний `str` як послідовність кодових точок, використовуючи фіксовану кількість байтів на кодову точку, для забезпечення ефективного прямого доступу до будь-якого символу або зрізу. Починаючи з Python 3.3, при створенні нового об'єкта `str` інтерпретатор перевіряє символи в ньому і вибирає найекономічніший формат пам'яті, підходящий для цього конкретного `str`: якщо є лише символи у діапазоні latin1, цей `str` використовуватиме лише один байт на кодову точку. В іншому разі може використовуватися два або чотири байти на кодову точку залежно від `str`. Це спрощення; для повних деталей дивіться PEP 393 — Flexible String Representation.

Гнучке представлення рядка схоже на те, як тип `int` працює в Python 3: якщо ціле число вміщується в машинне слово, воно зберігається в одному машинному слові. Інакше інтерпретатор переходить до представлення змінної довжини, подібного до типу `long` у Python 2. Приємно бачити поширення гарних ідей.

Однак завжди можна покладатися на Армін Ронашер, щоб знайти проблеми в Python 3. Він пояснив мені, чому це не така вже й гарна ідея на практиці: достатньо одного щура (RAT, U+1F400), щоб роздути інакше суто ASCII-текст у масив, що поглинає пам'ять, використовуючи чотири байти на символ, тоді як для кожного символу, крім щура, вистачило б одного байта. Крім того, через усі способи поєднання символів Unicode можливість швидкого отримання довільного символу за позицією є переоціненою, — а вилучення довільних зрізів із тексту Unicode є наївним у кращому випадку і часто помилковим, виробляючи мозибаке. З поширенням емодзі ці проблеми лише погіршуватимуться.
