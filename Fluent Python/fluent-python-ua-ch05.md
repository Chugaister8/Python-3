# Розділ 5. Конструктори класів даних

> Класи даних — як діти. Вони є прийнятним відправним пунктом, але щоб брати участь у дорослому об'єкті, їм потрібно взяти на себе певну відповідальність.
>
> — Мартін Фаулер та Кент Бек[^1]

[^1]: З *Refactoring*, перше видання, Глава 3, розділ «Bad Smells in Code, Data Class», стор. 87 (Addison-Wesley).

Python надає кілька способів побудови простого класу, що є лише набором полів із мінімальною або відсутньою додатковою функціональністю. Цей шаблон відомий як «клас даних», і `data classes` — одна з бібліотек, що його підтримують. У цьому розділі розглядаються три різних конструктори класів, які можна використовувати як скорочення для написання класів даних:

**`collections.namedtuple`**
Найпростіший спосіб — доступний із Python 2.6.

**`typing.NamedTuple`**
Альтернатива, що вимагає підказок типів на полях — з Python 3.5, синтаксис `class` доданий у 3.6.

**`@dataclasses.dataclass`**
Декоратор класу, що дозволяє більше налаштування, ніж попередні альтернативи, додаючи багато параметрів і можливу складність — із Python 3.7.

Після розгляду цих конструкторів класів ми обговоримо, чому клас даних є також назвою «запаху коду»: шаблону програмування, що може бути симптомом поганого об'єктно-орієнтованого дизайну.

> `typing.TypedDict` може здатися ще одним конструктором класів даних. Він використовує схожий синтаксис і описується одразу після `typing.NamedTuple` у документації модуля `typing` для Python 3.9. Однак `TypedDict` не будує конкретних класів, які можна інстанціювати. Це лише синтаксис для написання підказок типів для параметрів функцій та змінних, що прийматимуть значення відображень, які використовуються як записи з ключами-іменами полів. Ми розглянемо їх у Розділі 15, «TypedDict» (стор. 526).

## Огляд конструкторів класів даних

Розглянемо простий клас для представлення пари географічних координат, як показано в Прикладі 5-1.

**Приклад 5-1.** `class/coordinates.py`

```python
class Coordinate:
    def __init__(self, lat, lon):
        self.lat = lat
        self.lon = lon
```

Цей клас `Coordinate` виконує свою роботу зберігаючи атрибути широти і довготи. Написання шаблонного `__init__` дуже швидко набридає, особливо якщо ваш клас має більше кількох атрибутів: кожний із них згадується тричі! І цей шаблон не дає нам базових можливостей, яких ми очікували б від об'єкта Python:

```python
>>> from coordinates import Coordinate
>>> moscow = Coordinate(55.76, 37.62)
>>> moscow
<coordinates.Coordinate object at 0x107142f10>
>>> location = Coordinate(55.76, 37.62)
>>> location == moscow
False
>>> (location.lat, location.lon) == (moscow.lat, moscow.lon)
True
```

`__repr__`, успадкований від `object`, не є дуже корисним. Беззмістовний `==`; метод `__eq__`, успадкований від `object`, порівнює ID об'єктів. Для порівняння двох координат потрібне явне порівняння кожного атрибута.

Конструктори класів даних, розглянуті в цьому розділі, автоматично надають необхідні методи `__init__`, `__repr__` та `__eq__`, а також інші корисні можливості.

> Жоден із обговорюваних тут конструкторів класів не залежить від наслідування для виконання своєї роботи. І `collections.namedtuple`, і `typing.NamedTuple` будують класи, що є підкласами `tuple`. `@dataclass` — це декоратор класу, який жодним чином не впливає на ієрархію класів. Кожен із них використовує різні техніки метапрограмування для вставки методів та атрибутів даних у клас, що будується.

Ось клас `Coordinate`, побудований за допомогою `namedtuple` — фабричної функції, що будує підклас `tuple` із назвою та полями, які ви вказуєте:

```python
>>> from collections import namedtuple
>>> Coordinate = namedtuple('Coordinate', 'lat lon')
>>> issubclass(Coordinate, tuple)
True
>>> moscow = Coordinate(55.756, 37.617)
>>> moscow
Coordinate(lat=55.756, lon=37.617)
>>> moscow == Coordinate(lat=55.756, lon=37.617)
True
```

Корисний `__repr__`. Змістовний `__eq__`.

Новіший `typing.NamedTuple` надає ту саму функціональність, додаючи анотацію типу до кожного поля:

```python
>>> import typing
>>> Coordinate = typing.NamedTuple('Coordinate',
...                                [('lat', float), ('lon', float)])
>>> issubclass(Coordinate, tuple)
True
>>> typing.get_type_hints(Coordinate)
{'lat': <class 'float'>, 'lon': <class 'float'>}
```

> Іменований кортеж із підказками типів також можна побудувати, передавши поля як ключові аргументи:
>
> ```python
> Coordinate = typing.NamedTuple('Coordinate', lat=float, lon=float)
> ```
>
> Це більш читабельно, а також дозволяє передати відображення полів і типів як `**fields_and_types`.

Починаючи з Python 3.6, `typing.NamedTuple` також можна використовувати в інструкції `class` з анотаціями типів, описаними в PEP 526 — Syntax for Variable Annotations. Це значно більш читабельно і дозволяє легко перевизначати методи або додавати нові. Приклад 5-2 — це той самий клас `Coordinate` із парою атрибутів `float` та власним `__str__` для виведення координати у форматі `55.8°N, 37.6°E`.

**Приклад 5-2.** `typing_namedtuple/coordinates.py`

```python
from typing import NamedTuple

class Coordinate(NamedTuple):
    lat: float
    lon: float

    def __str__(self):
        ns = 'N' if self.lat >= 0 else 'S'
        we = 'E' if self.lon >= 0 else 'W'
        return f'{abs(self.lat):.1f}°{ns}, {abs(self.lon):.1f}°{we}'
```

> Хоча `NamedTuple` з'являється в інструкції `class` як суперклас, насправді ним не є. `typing.NamedTuple` використовує розширену функціональність метакласу[^2] для налаштування створення класу користувача. Переконайтеся:
>
> ```python
> >>> issubclass(Coordinate, typing.NamedTuple)
> False
> >>> issubclass(Coordinate, tuple)
> True
> ```

[^2]: Метакласи є однією з тем, що розглядаються в Розділі 24, «Class Metaprogramming».

Подібно до `typing.NamedTuple`, декоратор `dataclass` підтримує синтаксис PEP 526 для оголошення атрибутів екземпляра. Декоратор читає анотації змінних і автоматично генерує методи для вашого класу. Для порівняння перегляньте еквівалентний клас `Coordinate`, написаний за допомогою декоратора `dataclass`, як показано в Прикладі 5-3.

**Приклад 5-3.** `dataclass/coordinates.py`

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class Coordinate:
    lat: float
    lon: float

    def __str__(self):
        ns = 'N' if self.lat >= 0 else 'S'
        we = 'E' if self.lon >= 0 else 'W'
        return f'{abs(self.lat):.1f}°{ns}, {abs(self.lon):.1f}°{we}'
```

Зверніть увагу: тіло класів у Прикладах 5-2 і 5-3 є ідентичним — різниця лише в самій інструкції `class`. Декоратор `@dataclass` не залежить від наслідування або метакласу, тому не повинен заважати вашому власному використанню цих механізмів.[^3] Клас `Coordinate` у Прикладі 5-3 є підкласом `object`.

[^3]: Декоратори класів розглядаються в Розділі 24, «Class Metaprogramming», разом із метакласами. Обидва є способами налаштування поведінки класу, що виходять за межі можливого за допомогою наслідування.

## Основні можливості

Різні конструктори класів даних мають багато спільного, як підсумовано в Таблиці 5-1.

**Таблиця 5-1.** Вибрані можливості, порівняні для трьох конструкторів класів даних; `x` позначає екземпляр класу даних цього виду

| | namedtuple | NamedTuple | dataclass |
|---|:---:|:---:|:---:|
| Мутабельні екземпляри | НІ | НІ | ТАК |
| Синтаксис `class` | НІ | ТАК | ТАК |
| Побудова `dict` | `x._asdict()` | `x._asdict()` | `dataclasses.asdict(x)` |
| Отримати імена полів | `x._fields` | `x._fields` | `[f.name for f in dataclasses.fields(x)]` |
| Отримати значення за замовчуванням | `x._field_defaults` | `x._field_defaults` | `[f.default for f in dataclasses.fields(x)]` |
| Отримати типи полів | N/A | `x.__annotations__` | `x.__annotations__` |
| Новий екземпляр зі змінами | `x._replace(…)` | `x._replace(…)` | `dataclasses.replace(x, …)` |
| Новий клас під час виконання | `namedtuple(…)` | `NamedTuple(…)` | `dataclasses.make_dataclass(…)` |

> Класи, побудовані `typing.NamedTuple` і `@dataclass`, мають атрибут `__annotations__`, що містить підказки типів для полів. Однак читати безпосередньо з `__annotations__` не рекомендується. Натомість рекомендована найкраща практика для отримання цієї інформації — виклик `inspect.get_annotations(MyClass)` (додано в Python 3.10) або `typing.get_type_hints(MyClass)` (Python 3.5–3.9). Це пов'язано з тим, що ці функції надають додаткові послуги, наприклад розв'язання прямих посилань у підказках типів. Ми повернемося до цього питання значно пізніше в книзі, у «Problems with Annotations at Runtime» (стор. 538).

Тепер обговоримо ці основні можливості.

### Мутабельні екземпляри

Ключова відмінність між цими конструкторами класів полягає в тому, що `collections.namedtuple` і `typing.NamedTuple` будують підкласи `tuple`, тому екземпляри є незмінними. За замовчуванням `@dataclass` виробляє мутабельні класи. Але декоратор приймає ключовий аргумент `frozen` — показаний у Прикладі 5-3. Коли `frozen=True`, клас викликатиме виняток, якщо ви спробуєте присвоїти значення полю після ініціалізації екземпляра.

### Синтаксис class

Лише `typing.NamedTuple` і `dataclass` підтримують звичайний синтаксис інструкції `class`, що полегшує додавання методів і docstrings до класу, що створюється.

### Побудова dict

Обидва варіанти іменованих кортежів надають метод екземпляра (`._asdict`) для побудови об'єкта `dict` із полів екземпляра класу даних. Модуль `dataclasses` надає функцію для цього: `dataclasses.asdict`.

### Отримання імен і значень за замовчуванням полів

Усі три конструктори класів дозволяють отримувати імена полів і значення за замовчуванням, що можуть бути для них налаштовані. У класах іменованих кортежів ці метадані зберігаються в атрибутах класу `._fields` і `._fields_defaults`. Ті самі метадані з класу, декорованого `dataclass`, можна отримати за допомогою функції `fields` із модуля `dataclasses`. Вона повертає кортеж об'єктів `Field`, що мають кілька атрибутів, зокрема `name` і `default`.

### Отримання типів полів

Класи, визначені за допомогою `typing.NamedTuple` і `@dataclass`, мають атрибут класу `__annotations__` — відображення імен полів на типи. Як вже зазначалось, використовуйте функцію `typing.get_type_hints` замість безпосереднього читання `__annotations__`.

### Новий екземпляр зі змінами

Для екземпляра іменованого кортежу `x` виклик `x._replace(**kwargs)` повертає новий екземпляр із деякими замінами значень атрибутів відповідно до переданих ключових аргументів. Функція `dataclasses.replace(x, **kwargs)` на рівні модуля робить те саме для екземпляра класу, декорованого `@dataclass`.

### Новий клас під час виконання

Хоча синтаксис інструкції `class` є більш читабельним, він жорстко закодований. Фреймворк може потребувати побудови класів даних на льоту, під час виконання. Для цього можна використовувати синтаксис виклику функції за замовчуванням `collections.namedtuple`, що підтримується і `typing.NamedTuple`. Модуль `dataclasses` надає функцію `make_dataclass` для тієї самої цілі.

## Класичні іменовані кортежі

Функція `collections.namedtuple` — це фабрика, що будує підкласи `tuple`, розширені іменами полів, іменем класу та інформативним `__repr__`. Класи, побудовані за допомогою `namedtuple`, можна використовувати скрізь, де потрібні кортежі, і фактично багато функцій стандартної бібліотеки Python, що раніше повертали кортежі, тепер повертають іменовані кортежі для зручності — без жодного впливу на код користувача.

> Кожний екземпляр класу, побудованого `namedtuple`, займає рівно стільки пам'яті, скільки кортеж, тому що імена полів зберігаються в класі.

Приклад 5-4 показує, як можна визначити іменований кортеж для зберігання інформації про місто.

**Приклад 5-4.** Визначення та використання типу іменованого кортежу

```python
>>> from collections import namedtuple
>>> City = namedtuple('City', 'name country population coordinates')
>>> tokyo = City('Tokyo', 'JP', 36.933, (35.689722, 139.691667))
>>> tokyo
City(name='Tokyo', country='JP', population=36.933, coordinates=(35.689722,
139.691667))
>>> tokyo.population
36.933
>>> tokyo.coordinates
(35.689722, 139.691667)
>>> tokyo[1]
'JP'
```

Для створення іменованого кортежу потрібні два параметри: ім'я класу та список імен полів, який можна передати як ітерабельний об'єкт рядків або як єдиний рядок із роздільником-пробілом. Значення полів мають передаватися як окремі позиційні аргументи конструктору (на відміну від конструктора `tuple`, що приймає єдиний ітерабельний об'єкт). До полів можна звертатися за іменем або позицією.

Як підклас `tuple`, клас `City` успадковує корисні методи, зокрема `__eq__` та спеціальні методи операторів порівняння, включно з `__lt__`, що дозволяє сортувати списки екземплярів `City`.

Іменований кортеж пропонує кілька атрибутів і методів на додаток до тих, що успадковані від `tuple`. Приклад 5-5 показує найбільш корисні з них: атрибут класу `_fields`, метод класу `_make(iterable)` та метод екземпляра `_asdict()`.

**Приклад 5-5.** Атрибути та методи іменованого кортежу (продовження попереднього прикладу)

```python
>>> City._fields
('name', 'country', 'population', 'location')
>>> Coordinate = namedtuple('Coordinate', 'lat lon')
>>> delhi_data = ('Delhi NCR', 'IN', 21.935, Coordinate(28.613889, 77.208889))
>>> delhi = City._make(delhi_data)
>>> delhi._asdict()
{'name': 'Delhi NCR', 'country': 'IN', 'population': 21.935,
'location': Coordinate(lat=28.613889, lon=77.208889)}
>>> import json
>>> json.dumps(delhi._asdict())
'{"name": "Delhi NCR", "country": "IN", "population": 21.935,
"location": [28.613889, 77.208889]}'
```

`._fields` — кортеж із іменами полів класу. `._make()` будує `City` з ітерабельного об'єкта; `City(*delhi_data)` зробило б те саме. `._asdict()` повертає `dict`, побудований з екземпляра іменованого кортежу. `._asdict()` є корисним для серіалізації даних у форматі JSON, наприклад.

> Метод `_asdict` повертав `OrderedDict` до Python 3.7. Починаючи з Python 3.8, він повертає простий `dict` — що нормально тепер, коли можна покладатися на порядок вставки ключів. Якщо потрібен `OrderedDict`, документація `_asdict` рекомендує будувати його з результату: `OrderedDict(x._asdict())`.

Починаючи з Python 3.7, `namedtuple` приймає ключовий аргумент `defaults`, що надає ітерабельний об'єкт із N значеннями за замовчуванням для кожного з N найправіших полів класу. Приклад 5-6 показує, як визначити іменований кортеж `Coordinate` зі значенням за замовчуванням для поля `reference`.

**Приклад 5-6.** Атрибути та методи іменованого кортежу, продовження Прикладу 5-5

```python
>>> Coordinate = namedtuple('Coordinate', 'lat lon reference', defaults=['WGS84'])
>>> Coordinate(0, 0)
Coordinate(lat=0, lon=0, reference='WGS84')
>>> Coordinate._field_defaults
{'reference': 'WGS84'}
```

У «Синтаксис class» (стор. 168) я зазначав, що з синтаксисом `class`, підтримуваним `typing.NamedTuple` і `@dataclass`, простіше кодувати методи. До `namedtuple` також можна додавати методи, але це є хакерським підходом.

> **Хак для вставки методу в namedtuple**
>
> Згадайте, як ми будували клас `Card` у Прикладі 1-1 у Розділі 1:
>
> ```python
> Card = collections.namedtuple('Card', ['rank', 'suit'])
> ```
>
> Пізніше в Розділі 1 я написав функцію `spades_high` для сортування. Було б зручно, якби ця логіка була інкапсульована в методі `Card`, але додавання `spades_high` до `Card` без синтаксису `class` вимагає швидкого хака: визначіть функцію, а потім присвойте її атрибуту класу. Приклад 5-7 показує, як це зробити.
>
> **Приклад 5-7.** `frenchdeck.doctest`: Додавання атрибута класу та методу до `Card`, іменованого кортежу з «Пітонічна колода карт» (стор. 5)
>
> ```python
> >>> Card.suit_values = dict(spades=3, hearts=2, diamonds=1, clubs=0)
> >>> def spades_high(card):
> ...     rank_value = FrenchDeck.ranks.index(card.rank)
> ...     suit_value = card.suit_values[card.suit]
> ...     return rank_value * len(card.suit_values) + suit_value
> ...
> >>> Card.overall_rank = spades_high
> >>> lowest_card = Card('2', 'clubs')
> >>> highest_card = Card('A', 'spades')
> >>> lowest_card.overall_rank()
> 0
> >>> highest_card.overall_rank()
> 51
> ```
>
> Приєднати атрибут класу зі значеннями для кожної масті. `spades_high` стане методом; перший аргумент не обов'язково називати `self`. В будь-якому разі при виклику як методу він отримає отримувача. Приєднати функцію до класу `Card` як метод `overall_rank`. Це спрацьовує!
>
> Для читабельності та подальшого обслуговування значно краще кодувати методи всередині інструкції `class`. Але корисно знати, що цей хак можливий, адже він може знадобитися.[^4]

[^4]: Якщо ви знаєте Ruby, знаєте, що вставка методів є відомою, але суперечливою технікою серед розробників Ruby. У Python це не настільки поширено, бо не працює з жодним вбудованим типом — `str`, `list` тощо. Я вважаю це обмеження Python благом.

## Іменовані кортежі з підказками типів

Клас `Coordinate` зі значенням за замовчуванням для поля з Прикладу 5-6 можна написати за допомогою `typing.NamedTuple`, як показано в Прикладі 5-8.

**Приклад 5-8.** `typing_namedtuple/coordinates2.py`

```python
from typing import NamedTuple

class Coordinate(NamedTuple):
    lat: float
    lon: float
    reference: str = 'WGS84'
```

Кожне поле екземпляра має бути анотоване типом. Поле екземпляра `reference` анотоване типом і має значення за замовчуванням.

Класи, побудовані `typing.NamedTuple`, не мають жодних методів, крім тих, що генерує `collections.namedtuple`, та тих, що успадковуються від `tuple`. Єдина відмінність — наявність атрибута класу `__annotations__`, який Python повністю ігнорує під час виконання.

Оскільки головною особливістю `typing.NamedTuple` є анотації типів, перед тим як продовжити дослідження конструкторів класів даних, коротко розглянемо їх.

## Підказки типів 101

Підказки типів, вони ж анотації типів, — це способи оголошення очікуваного типу аргументів функцій, значень, що повертаються, змінних та атрибутів.

Перше, що потрібно знати про підказки типів: компілятор байткоду та інтерпретатор Python їх **взагалі не застосовують**.

> Це дуже короткий вступ до підказок типів — лише достатній для розуміння синтаксису та значення анотацій, що використовуються в оголошеннях `typing.NamedTuple` і `@dataclass`. Підказки типів для сигнатур функцій ми розглянемо в Розділі 8, а більш розширені анотації — у Розділі 15. Тут ми переважно побачимо підказки з простими вбудованими типами — `str`, `int` та `float`, які, мабуть, є найпоширенішими типами, що використовуються для анотування полів класів даних.

### Відсутність ефекту під час виконання

Думайте про підказки типів Python як про «документацію, яку можна перевірити за допомогою IDE та засобів перевірки типів».

Це пов'язано з тим, що підказки типів не впливають на поведінку програм Python під час виконання. Перегляньте Приклад 5-9.

**Приклад 5-9.** Python не застосовує підказки типів під час виконання

```python
>>> import typing
>>> class Coordinate(typing.NamedTuple):
...     lat: float
...     lon: float
...
>>> trash = Coordinate('Ni!', None)
>>> print(trash)
Coordinate(lat='Ni!', lon=None)
```

Я ж казав: перевірки типів під час виконання немає! Якщо надрукувати код Прикладу 5-9 у модулі Python, він запуститься і відобразить беззмістовну `Coordinate` без жодної помилки чи попередження.

Підказки типів призначені в першу чергу для підтримки сторонніх засобів перевірки типів, таких як Mypy або вбудований засіб перевірки типів IDE PyCharm. Це інструменти статичного аналізу: вони перевіряють вихідний код Python «у спокої», а не код, що виконується.

```
$ mypy nocheck_demo.py
nocheck_demo.py:8: error: Argument 1 to "Coordinate" has
incompatible type "str"; expected "float"
nocheck_demo.py:8: error: Argument 2 to "Coordinate" has
incompatible type "None"; expected "float"
```

### Синтаксис анотацій змінних

І `typing.NamedTuple`, і `@dataclass` використовують синтаксис анотацій змінних, визначений у PEP 526. Це короткий вступ до цього синтаксису в контексті визначення атрибутів в інструкціях `class`.

Базовий синтаксис анотації змінної:

```python
var_name: some_type
```

Розділ «Acceptable type hints» у PEP 484 пояснює, що є прийнятними типами, але в контексті визначення класу даних ці типи, найімовірніше, будуть корисними:

- конкретний клас, наприклад `str` або `FrenchDeck`;
- параметризований тип колекції, наприклад `list[int]`, `tuple[str, float]` тощо;
- `typing.Optional`, наприклад `Optional[str]` — для оголошення поля, що може бути `str` або `None`.

Змінну також можна ініціалізувати значенням. В оголошенні `typing.NamedTuple` або `@dataclass` це значення стане значенням за замовчуванням для цього атрибута, якщо відповідний аргумент пропущено у виклику конструктора:

```python
var_name: some_type = a_value
```

### Значення анотацій змінних

Ми бачили в «Відсутність ефекту під час виконання» (стор. 173), що підказки типів не мають ефекту під час виконання. Але під час імпорту — коли модуль завантажується — Python читає їх для побудови словника `__annotations__`, який `typing.NamedTuple` і `@dataclass` потім використовують для покращення класу.

Розпочнемо дослідження з простого класу в Прикладі 5-10, щоб потім побачити, які додаткові можливості надають `typing.NamedTuple` і `@dataclass`.

**Приклад 5-10.** `meaning/demo_plain.py`: простий клас із підказками типів

```python
class DemoPlainClass:
    a: int
    b: float = 1.1
    c = 'spam'
```

`a` стає записом у `__annotations__`, але в іншому відношенні відкидається: ніякого атрибута з іменем `a` в класі не створюється. `b` зберігається як анотація і також стає атрибутом класу зі значенням 1.1. `c` — просто звичайний атрибут класу, не анотація.

Можна перевірити це у консолі:

```python
>>> from demo_plain import DemoPlainClass
>>> DemoPlainClass.__annotations__
{'a': <class 'int'>, 'b': <class 'float'>}
>>> DemoPlainClass.a
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
AttributeError: type object 'DemoPlainClass' has no attribute 'a'
>>> DemoPlainClass.b
1.1
>>> DemoPlainClass.c
'spam'
```

> Спеціальний атрибут `__annotations__` створюється інтерпретатором для запису підказок типів, що з'являються у вихідному коді — навіть у простому класі.
>
> `a` залишається лише як анотація. Він не стає атрибутом класу, бо до нього не прив'язано жодного значення.[^5] `b` і `c` зберігаються як атрибути класу, бо вони прив'язані до значень.

[^5]: Python не має концепції «undefined» — одна з найдурніших помилок у дизайні JavaScript. Дякуємо Ґвідо!

Жоден із цих трьох атрибутів не буде в новому екземплярі `DemoPlainClass`.

### Огляд typing.NamedTuple

Тепер розглянемо клас, побудований за допомогою `typing.NamedTuple` (Приклад 5-11), використовуючи ті самі атрибути та анотації, що й `DemoPlainClass` із Прикладу 5-10.

**Приклад 5-11.** `meaning/demo_nt.py`: клас, побудований за допомогою `typing.NamedTuple`

```python
import typing

class DemoNTClass(typing.NamedTuple):
    a: int
    b: float = 1.1
    c = 'spam'
```

`a` стає анотацією і також атрибутом екземпляра. `b` — ще одна анотація, і також стає атрибутом екземпляра зі значенням за замовчуванням 1.1. `c` — просто звичайний атрибут класу; жодна анотація не посилатиметься на нього.

При перевірці `DemoNTClass`:

```python
>>> from demo_nt import DemoNTClass
>>> DemoNTClass.__annotations__
{'a': <class 'int'>, 'b': <class 'float'>}
>>> DemoNTClass.a
<_collections._tuplegetter object at 0x101f0f940>
>>> DemoNTClass.b
<_collections._tuplegetter object at 0x101f0f8b0>
>>> DemoNTClass.c
'spam'
```

Атрибути класу `a` і `b` є дескрипторами — розширена можливість, що розглядається в Розділі 23. Наразі думайте про них як про геттери властивостей: методи, що не вимагають явного оператора виклику `()` для отримання атрибута екземпляра. На практиці це означає, що `a` і `b` працюватимуть як атрибути екземпляра лише для читання — що має сенс, коли ми пам'ятаємо, що екземпляри `DemoNTClass` є лише вишуканими кортежами, а кортежі є незмінними.

### Огляд класу, декорованого dataclass

Тепер розглянемо Приклад 5-12.

**Приклад 5-12.** `meaning/demo_dc.py`: клас, декорований `@dataclass`

```python
from dataclasses import dataclass

@dataclass
class DemoDataClass:
    a: int
    b: float = 1.1
    c = 'spam'
```

```python
>>> from demo_dc import DemoDataClass
>>> DemoDataClass.__annotations__
{'a': <class 'int'>, 'b': <class 'float'>}
>>> DemoDataClass.__doc__
'DemoDataClass(a: int, b: float = 1.1)'
>>> DemoDataClass.a
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
AttributeError: type object 'DemoDataClass' has no attribute 'a'
>>> DemoDataClass.b
1.1
>>> DemoDataClass.c
'spam'
```

В `DemoDataClass` немає атрибута `a` — на відміну від `DemoNTClass`. Атрибут `a` існуватиме лише в екземплярах `DemoDataClass`. Це буде публічний атрибут, який можна отримувати і встановлювати, якщо клас не є заморозленим. А `b` і `c` існують як атрибути класу: `b` зберігає значення за замовчуванням для атрибута екземпляра `b`, тоді як `c` — лише атрибут класу, що не буде прив'язаний до екземплярів.

## Більше про @dataclass

Ми бачили лише прості приклади використання `@dataclass`. Декоратор приймає кілька ключових аргументів. Ось його сигнатура:

```python
@dataclass(*, init=True, repr=True, eq=True, order=False,
           unsafe_hash=False, frozen=False)
```

`*` на першій позиції означає, що решта параметрів є лише ключовими. Таблиця 5-2 описує їх.

**Таблиця 5-2.** Ключові параметри, прийняті декоратором `@dataclass`

| Параметр | Значення | Типово | Примітки |
|---|---|:---:|---|
| `init` | Генерувати `__init__` | `True` | Ігнорується, якщо `__init__` реалізовано користувачем. |
| `repr` | Генерувати `__repr__` | `True` | Ігнорується, якщо `__repr__` реалізовано користувачем. |
| `eq` | Генерувати `__eq__` | `True` | Ігнорується, якщо `__eq__` реалізовано користувачем. |
| `order` | Генерувати `__lt__`, `__le__`, `__gt__`, `__ge__` | `False` | Якщо `True`, викликає виняток при `eq=False` або якщо будь-який із методів порівняння вже визначений або успадкований. |
| `unsafe_hash` | Генерувати `__hash__` | `False` | Складна семантика з кількома застереженнями — дивіться документацію `dataclass`. |
| `frozen` | Зробити екземпляри «незмінними» | `False` | Екземпляри будуть достатньо захищені від випадкових змін, але насправді не незмінними.[^a] |

[^a]: `@dataclass` емулює незмінність, генеруючи `__setattr__` і `__delattr__`, які викликають `dataclass.FrozenInstanceError` — підклас `AttributeError` — коли користувач намагається встановити або видалити поле.

Параметри за замовчуванням є найбільш корисними налаштуваннями для типових випадків використання. Параметри, які ви, найімовірніше, захочете змінити:

**`frozen=True`**
Захищає від випадкових змін екземплярів класу.

**`order=True`**
Дозволяє сортування екземплярів класу даних.

Якщо аргументи `eq` і `frozen` обидва є `True`, `@dataclass` виробляє підходящий метод `__hash__`, тому екземпляри будуть гешованими. Якщо `frozen=False` (за замовчуванням), `@dataclass` встановить `__hash__` у `None`, сигналізуючи, що екземпляри є негешованими.

### Параметри поля

Ми вже бачили найбільш базовий параметр поля: надання (або не надання) значення за замовчуванням із підказкою типу. Поля екземпляра, які ви оголошуєте, стануть параметрами в згенерованому `__init__`. Python не дозволяє параметри без значень за замовчуванням після параметрів зі значеннями за замовчуванням, тому після оголошення поля зі значенням за замовчуванням всі наступні поля також мають мати значення за замовчуванням.

Мутабельні значення за замовчуванням є поширеним джерелом помилок для програмістів-початківців у Python. У визначеннях функцій мутабельне значення за замовчуванням легко пошкоджується, коли одне з викликань функції змінює значення за замовчуванням, змінюючи поведінку подальших викликів. `@dataclass` відхиляє визначення класу в Прикладі 5-13.

**Приклад 5-13.** `dataclass/club_wrong.py`: цей клас викликає `ValueError`

```python
@dataclass
class ClubMember:
    name: str
    guests: list = []
```

Якщо завантажити модуль із цим класом `ClubMember`:

```
$ python3 club_wrong.py
Traceback (most recent call last):
  ...
ValueError: mutable default <class 'list'> for field guests is not allowed:
use default_factory
```

Повідомлення `ValueError` пояснює проблему і пропонує рішення: використовуйте `default_factory`. Приклад 5-14 показує, як виправити `ClubMember`.

**Приклад 5-14.** `dataclass/club.py`: це визначення `ClubMember` працює

```python
from dataclasses import dataclass, field

@dataclass
class ClubMember:
    name: str
    guests: list = field(default_factory=list)
```

У полі `guests` Прикладу 5-14 замість літерального списку значення за замовчуванням встановлюється через виклик функції `dataclasses.field` із `default_factory=list`.

Параметр `default_factory` дозволяє надати функцію, клас або будь-який інший callable, що буде викликатися без аргументів для побудови значення за замовчуванням щоразу, коли створюється екземпляр класу даних. Таким чином, кожний екземпляр `ClubMember` матиме власний список — замість того, щоб всі екземпляри спільно використовували один і той самий список із класу.

> Добре, що `@dataclass` відхиляє визначення класів зі значенням `list` за замовчуванням у полі. Однак майте на увазі, що це часткове рішення, яке стосується лише `list`, `dict` та `set`. Інші мутабельні значення, що використовуються як значення за замовчуванням, не позначатимуться `@dataclass`. Це ваша відповідальність — зрозуміти проблему та пам'ятати про використання фабрики за замовчуванням для встановлення мутабельних значень за замовчуванням.

**Приклад 5-15.** `dataclass/club_generic.py`: це визначення `ClubMember` є більш точним

```python
from dataclasses import dataclass, field

@dataclass
class ClubMember:
    name: str
    guests: list[str] = field(default_factory=list)
```

`list[str]` означає «список рядків `str`». Новий синтаксис `list[str]` — це параметризований узагальнений тип: починаючи з Python 3.9, вбудований тип `list` приймає цей синтаксис із дужками для вказівки типу елементів списку.

Таблиця 5-3 перераховує всі ключові аргументи, прийняті функцією `field`.

**Таблиця 5-3.** Ключові аргументи, прийняті функцією `field`

| Параметр | Значення | Типово |
|---|---|:---:|
| `default` | Значення за замовчуванням для поля | `_MISSING_TYPE`[^a2] |
| `default_factory` | Функція з 0 параметрами для виробництва значення за замовчуванням | `_MISSING_TYPE` |
| `init` | Включити поле в параметри `__init__` | `True` |
| `repr` | Включити поле в `__repr__` | `True` |
| `compare` | Використати поле в методах порівняння `__eq__`, `__lt__` тощо | `True` |
| `hash` | Включити поле в обчислення `__hash__` | `None`[^b2] |
| `metadata` | Відображення з даними, визначеними користувачем; ігнорується `@dataclass` | `None` |

[^a2]: `dataclass._MISSING_TYPE` — це сентинельне значення, що вказує, що параметр не вказано. Він існує, щоб ми могли встановити `None` як фактичне значення за замовчуванням — поширений варіант використання.
[^b2]: Параметр `hash=None` означає, що поле буде використовуватися в `__hash__` лише якщо `compare=True`.

### Обробка після ініціалізації

Метод `__init__`, згенерований `@dataclass`, лише приймає передані аргументи та присвоює їх — або їхні значення за замовчуванням, якщо відсутні — атрибутам екземпляра, що є полями екземпляра. Але може знадобитися зробити більше для ініціалізації екземпляра.

У такому разі можна надати метод `__post_init__`. Коли цей метод існує, `@dataclass` додасть код до згенерованого `__init__` для виклику `__post_init__` як останнього кроку.

Типові варіанти використання `__post_init__` — перевірка та обчислення значень поля на основі інших полів. Розглянемо простий приклад, що використовує `__post_init__` для обох цілей.

**Приклад 5-16.** `dataclass/hackerclub.py`: doctests для `HackerClubMember`

```python
"""
``HackerClubMember`` objects accept an optional ``handle`` argument::

    >>> anna = HackerClubMember('Anna Ravenscroft', handle='AnnaRaven')
    >>> anna
    HackerClubMember(name='Anna Ravenscroft', guests=[], handle='AnnaRaven')

If ``handle`` is omitted, it's set to the first part of the member's name::

    >>> leo = HackerClubMember('Leo Rochael')
    >>> leo
    HackerClubMember(name='Leo Rochael', guests=[], handle='Leo')

Members must have a unique handle. The following ``leo2`` will not be created,
because its ``handle`` would be 'Leo', which was taken by ``leo``::

    >>> leo2 = HackerClubMember('Leo DaVinci')
    Traceback (most recent call last):
        ...
    ValueError: handle 'Leo' already exists.

To fix, ``leo2`` must be created with an explicit ``handle``::

    >>> leo2 = HackerClubMember('Leo DaVinci', handle='Neo')
    >>> leo2
    HackerClubMember(name='Leo DaVinci', guests=[], handle='Neo')
"""
```

**Приклад 5-17.** `dataclass/hackerclub.py`: код для `HackerClubMember`

```python
from dataclasses import dataclass
from club import ClubMember

@dataclass
class HackerClubMember(ClubMember):
    all_handles = set()
    handle: str = ''

    def __post_init__(self):
        cls = self.__class__
        if self.handle == '':
            self.handle = self.name.split()[0]
        if self.handle in cls.all_handles:
            msg = f'handle {self.handle!r} already exists.'
            raise ValueError(msg)
        cls.all_handles.add(self.handle)
```

`HackerClubMember` розширює `ClubMember`. `all_handles` — атрибут класу. `handle` — поле екземпляра типу `str` із порожнім рядком як значенням за замовчуванням; це робить його необов'язковим. Отримати клас екземпляра. Якщо `self.handle` є порожнім рядком, встановити його в перший фрагмент імені. Якщо `self.handle` є в `cls.all_handles` — викликати `ValueError`. Додати новий handle до `cls.all_handles`.

### Атрибути класу з підказками типів

Якщо перевірити Приклад 5-17 за допомогою Mypy:

```
$ mypy hackerclub.py
hackerclub.py:37: error: Need type annotation for "all_handles"
(hint: "all_handles: Set[<type>] = ...")
Found 1 error in 1 file (checked 1 source file)
```

Обхідний шлях, визначений у PEP 526 — Syntax for Variable Annotations, є незграбним. Для кодування змінної класу з підказкою типу потрібно використовувати псевдотип `typing.ClassVar`, що використовує нотацію `[]` узагальнень для задання типу змінної та оголошення її атрибутом класу.

Щоб задовольнити засіб перевірки типів і `@dataclass`, ось як слід оголошувати `all_handles` у Прикладі 5-17:

```python
all_handles: ClassVar[set[str]] = set()
```

Ця підказка типу говорить: `all_handles` є атрибутом класу типу `set-of-str` із порожньою множиною як значенням за замовчуванням. Для написання цієї анотації потрібно імпортувати `ClassVar` із модуля `typing`.

Декоратор `@dataclass` не дбає про типи в анотаціях, за двома винятками, і цей є одним із них: якщо тип є `ClassVar`, поле екземпляра для цього атрибута не генерується.

### Змінні ініціалізації, що не є полями

Іноді може знадобитися передати аргументи до `__init__`, які не є полями екземпляра. Такі аргументи називаються *змінними лише для ініціалізації* (init-only variables) в документації `dataclasses`. Для оголошення такого аргументу модуль `dataclasses` надає псевдотип `InitVar`, що використовує той самий синтаксис, що й `typing.ClassVar`.

**Приклад 5-18.** Приклад із документації модуля `dataclasses`

```python
@dataclass
class C:
    i: int
    j: int = None
    database: InitVar[DatabaseType] = None

    def __post_init__(self, database):
        if self.j is None and database is not None:
            self.j = database.lookup('j')

c = C(10, database=my_database)
```

`InitVar` запобігатиме тому, щоб `@dataclass` трактував `database` як звичайне поле. Він не буде встановлений як атрибут екземпляра, і функція `dataclasses.fields` не буде його перераховувати. Однак `database` буде одним із аргументів, що приймає згенерований `__init__`, і він також буде передаватися до `__post_init__`.

## Приклад @dataclass: запис ресурсу Dublin Core

Часто класи, побудовані за допомогою `@dataclass`, матимуть більше полів, ніж у дуже коротких прикладах, представлених донині. Dublin Core надає основу для більш типового прикладу `@dataclass`.

> Схема Dublin Core — це невеликий набір термінів словника, що можна використовувати для опису цифрових ресурсів (відео, зображення, вебсторінки тощо), а також фізичних ресурсів, таких як книги або CD, та об'єктів, наприклад художніх творів.[^8]
>
> — Dublin Core у Вікіпедії

[^8]: Джерело: стаття Dublin Core в англійській Вікіпедії.

Стандарт визначає 15 необов'язкових полів; клас `Resource` у Прикладі 5-19 використовує 8 із них.

**Приклад 5-19.** `dataclass/resource.py`: код для `Resource`, класу на основі термінів Dublin Core

```python
from dataclasses import dataclass, field
from typing import Optional
from enum import Enum, auto
from datetime import date

class ResourceType(Enum):
    BOOK = auto()
    EBOOK = auto()
    VIDEO = auto()

@dataclass
class Resource:
    """Media resource description."""
    # Опис медіаресурсу.
    identifier: str
    title: str = '<untitled>'
    creators: list[str] = field(default_factory=list)
    date: Optional[date] = None
    type: ResourceType = ResourceType.BOOK
    description: str = ''
    language: str = ''
    subjects: list[str] = field(default_factory=list)
```

Цей `Enum` надасть типобезпечні значення для поля `Resource.type`. `identifier` — єдине обов'язкове поле. `title` — перше поле зі значенням за замовчуванням. Значення `date` може бути екземпляром `datetime.date` або `None`. Значення за замовчуванням поля `type` — `ResourceType.BOOK`.

Приклад 5-20 показує doctest, що демонструє, як запис `Resource` виглядає у коді.

**Приклад 5-20.** `dataclass/resource.py`: doctest для `Resource`

```python
>>> description = 'Improving the design of existing code'
>>> book = Resource('978-0-13-475759-9', 'Refactoring, 2nd Edition',
...                 ['Martin Fowler', 'Kent Beck'], date(2018, 11, 19),
...                 ResourceType.BOOK, description, 'EN',
...                 ['computer programming', 'OOP'])
>>> book  # doctest: +NORMALIZE_WHITESPACE
Resource(identifier='978-0-13-475759-9', title='Refactoring, 2nd Edition',
creators=['Martin Fowler', 'Kent Beck'], date=datetime.date(2018, 11, 19),
type=<ResourceType.BOOK: 1>, description='Improving the design of existing code',
language='EN', subjects=['computer programming', 'OOP'])
```

`__repr__`, згенерований `@dataclass`, нормальний, але ми можемо зробити його більш читабельним. Приклад 5-21 — це код `__repr__` для виробництва відформатованого виведення. У цьому прикладі використовується `dataclass.fields` для отримання імен полів класу даних.

**Приклад 5-21.** `dataclass/resource_repr.py`: код для методу `__repr__`, реалізованого в класі `Resource`

```python
def __repr__(self):
    cls = self.__class__
    cls_name = cls.__name__
    indent = ' ' * 4
    res = [f'{cls_name}(']
    for f in fields(cls):
        value = getattr(self, f.name)
        res.append(f'{indent}{f.name} = {value!r},')
    res.append(')')
    return '\n'.join(res)
```

Починаємо список `res` для побудови вихідного рядка іменем класу та відкритою дужкою. Для кожного поля `f` у класі... ...отримуємо іменований атрибут екземпляра. Додаємо рядок із відступом і іменем поля та `repr(value)` — ось що робить `!r`. Додаємо закриваючу дужку. Будуємо багаторядковий рядок з `res` і повертаємо його.

## Клас даних як запах коду

Незалежно від того, реалізуєте ви клас даних самостійно або використовуєте один із конструкторів класів, описаних у цьому розділі, майте на увазі, що це може свідчити про проблему у вашому дизайні.

У *Refactoring: Improving the Design of Existing Code*, 2nd ed. (Addison-Wesley), Мартін Фаулер і Кент Бек представляють каталог «запахів коду» — шаблонів у коді, що можуть вказувати на необхідність рефакторингу. Запис під назвою «Data Class» починається так:

> Це класи, що мають поля, геттери та сеттери для полів — і більше нічого. Такі класи є тупими носіями даних і часто надмірно детально маніпулюються іншими класами.

Основна ідея об'єктно-орієнтованого програмування — розміщення поведінки та даних разом в одній кодовій одиниці: класі. Якщо клас широко використовується, але не має значної власної поведінки, можливо, код, що обробляє його екземпляри, розпорошений (і навіть дублюється) у методах і функціях по всій системі — рецепт для проблем із обслуговуванням.

Беручи це до уваги, є кілька типових сценаріїв, де доцільно мати клас даних із мінімальною поведінкою або без неї.

### Клас даних як риштування

У цьому сценарії клас даних є початковою, спрощеною реалізацією класу для швидкого старту нового проєкту або модуля. З часом клас повинен отримати власні методи замість того, щоб покладатися на методи інших класів для роботи зі своїми екземплярами. Риштування є тимчасовим; зрештою ваш власний клас може стати повністю незалежним від конструктора, що ви використали для початку.

Python також використовується для швидкого вирішення завдань і експериментів, і тоді залишати риштування на місці нормально.

### Клас даних як проміжне представлення

Клас даних може бути корисним для побудови записів, що готуються до експорту в JSON або інший формат обміну, або для зберігання даних, що щойно були імпортовані. Усі конструктори класів даних Python надають метод або функцію для перетворення екземпляра на простий `dict`, і завжди можна викликати конструктор із `dict` як ключовими аргументами, розгорнутими за допомогою `**`.

У цьому сценарії екземпляри класу даних мають оброблятися як незмінні об'єкти — навіть якщо поля є мутабельними, ви не повинні їх змінювати, поки вони знаходяться в цьому проміжному вигляді.

## Зіставлення зі зразком екземплярів класів

Зразки класів призначені для зіставлення екземплярів класів за типом — і необов'язково — за атрибутами. Суб'єктом зразка класу може бути екземпляр будь-якого класу, а не лише екземпляри класів даних.[^10]

[^10]: Я розмістив цей вміст тут, тому що це найраніший розділ, що фокусується на класах, визначених користувачем, і вважав, що зіставлення зі зразком для класів є надто важливим, щоб чекати до Частини II книги. Моя філософія: знати, як використовувати класи, важливіше, ніж вміти їх визначати.

Існує три варіанти зразків класів: простий, ключовий та позиційний. Ми вивчатимемо їх у такому порядку.

### Прості зразки класів

Ми вже бачили приклад із простими зразками класів, що використовуються як підзразки в «Зіставлення зі зразком із послідовностями» (стор. 38):

```python
case [str(name), _, _, (float(lat), float(lon))]:
```

Синтаксис зразків класів виглядає як виклик конструктора. Наступний зразок класу відповідає значенням `float` без прив'язки змінної:

```python
match x:
    case float():
        do_something_with(x)
```

Але це, ймовірно, є помилкою у вашому коді:

```python
match x:
    case float:  # НЕБЕЗПЕКА!!!
        do_something_with(x)
```

У попередньому прикладі `case float:` відповідає будь-якому суб'єкту, бо Python бачить `float` як змінну, яка потім прив'язується до суб'єкта.

Простий синтаксис `float(x)` є особливим випадком, що застосовується лише до дев'яти «благословенних» вбудованих типів: `bytes`, `dict`, `float`, `frozenset`, `int`, `list`, `set`, `str`, `tuple`.

### Ключові зразки класів

Щоб зрозуміти, як використовувати ключові зразки класів, розглянемо наступний клас `City` і п'ять екземплярів у Прикладі 5-22.

**Приклад 5-22.** Клас `City` і кілька екземплярів

```python
import typing

class City(typing.NamedTuple):
    continent: str
    name: str
    country: str

cities = [
    City('Asia', 'Tokyo', 'JP'),
    City('Asia', 'Delhi', 'IN'),
    City('North America', 'Mexico City', 'MX'),
    City('North America', 'New York', 'US'),
    City('South America', 'São Paulo', 'BR'),
]
```

При таких визначеннях наступна функція поверне список азійських міст:

```python
def match_asian_cities():
    results = []
    for city in cities:
        match city:
            case City(continent='Asia'):
                results.append(city)
    return results
```

Зразок `City(continent='Asia')` відповідає будь-якому екземпляру `City`, де значення атрибута `continent` дорівнює `'Asia'`, незалежно від значень інших атрибутів.

Якщо потрібно зібрати значення атрибута `country`:

```python
def match_asian_countries():
    results = []
    for city in cities:
        match city:
            case City(continent='Asia', country=cc):
                results.append(cc)
    return results
```

Зразок `City(continent='Asia', country=cc)` відповідає тим самим азійським містам, що й раніше, але тепер змінна `cc` прив'язується до атрибута `country` екземпляра.

### Позиційні зразки класів

При визначеннях із Прикладу 5-22 наступна функція поверне список азійських міст за допомогою позиційного зразка класу:

```python
def match_asian_cities_pos():
    results = []
    for city in cities:
        match city:
            case City('Asia'):
                results.append(city)
    return results
```

Зразок `City('Asia')` відповідає будь-якому екземпляру `City`, де перший атрибут має значення `'Asia'`.

Що робить `City` або будь-який клас придатним для роботи з позиційними зразками — це наявність спеціального атрибута класу `__match_args__`, який конструктори класів у цьому розділі створюють автоматично:

```python
>>> City.__match_args__
('continent', 'name', 'country')
```

`__match_args__` оголошує імена атрибутів у порядку, в якому вони використовуватимуться в позиційних зразках.

У «Supporting Positional Pattern Matching» (стор. 377) ми напишемо код для визначення `__match_args__` для класу без допомоги конструктора класів.

## Підсумок розділу

Основною темою цього розділу були конструктори класів даних `collections.namedtuple`, `typing.NamedTuple` і `dataclasses.dataclass`. Ми побачили, що кожний генерує класи даних із описів, наданих як аргументи фабричній функції, або з інструкцій `class` із підказками типів — для двох останніх. Зокрема, обидва варіанти іменованих кортежів виробляють підкласи `tuple`, додаючи лише можливість доступу до полів за іменем і надаючи атрибут класу `_fields`, що перераховує імена полів у вигляді кортежу рядків.

Далі ми вивчили основні можливості трьох конструкторів класів поряд: як витягувати дані екземпляра як `dict`, як отримувати імена і значення за замовчуванням полів і як створювати новий екземпляр із наявного.

Це спонукало нас до першого знайомства з підказками типів, зокрема тими, що використовуються для анотування атрибутів в інструкції `class`. Найбільш несподіваний аспект підказок типів загалом — той факт, що вони взагалі не мають ефекту під час виконання. Python залишається динамічною мовою. Для використання типізованої інформації з метою виявлення помилок за допомогою статичного аналізу вихідного коду потрібні зовнішні інструменти, наприклад Mypy.

Потім ми розглянули найбільш часто використовувані можливості `@dataclass` і параметр `default_factory` функції `dataclasses.field`. Ми також розглянули спеціальні псевдоанотації типів `typing.ClassVar` і `dataclasses.InitVar`. Основна тема завершилась прикладом на основі схеми Dublin Core, що ілюстрував використання `dataclasses.fields` для ітерації по атрибутах екземпляра `Resource` у власному `__repr__`.

Потім ми попередили про можливе зловживання класами даних, що порушує основний принцип об'єктно-орієнтованого програмування: дані та функції, що їх обробляють, мають знаходитися разом в одному класі.

В останньому розділі ми побачили, як зіставлення зі зразком працює з суб'єктами, що є екземплярами будь-якого класу — не лише класів, побудованих за допомогою конструкторів, представлених у цьому розділі.

## Додаткова література

Стандартна документація Python для розглянутих конструкторів класів даних є дуже хорошою і містить чимало невеликих прикладів.

Для `@dataclass` зокрема, більша частина PEP 557 — Data Classes була скопійована в документацію модуля `dataclasses`. Але PEP 557 містить кілька дуже інформативних розділів, які не були скопійовані, зокрема «Why not just use namedtuple?», «Why not just use typing.NamedTuple?» та розділ «Rationale».

На RealPython.com Ґейр Арне Г'єлле написав дуже повний «Ultimate guide to data classes in Python 3.7».

На PyCon US 2018 Реймонд Геттінґер представив «Dataclasses: The code generator to end all code generators» (відео).

Для більш розширених можливостей, зокрема валідації, проєкт `attrs` під керівництвом Хінека Шлаваша з'явився роками раніше за `dataclasses` і пропонує більше можливостей. Вплив `attrs` на `@dataclass` визнає Ерік В. Сміт у PEP 557.

Ґліф — засновник проєкту Twisted — написав чудовий вступ до `attrs` у «The One Python Library Everyone Needs».

Щодо класу даних як запаху коду, найкраще джерело — книга Мартіна Фаулера *Refactoring*, 2nd ed. Найновіша версія пропускає цитату з епіграфа цього розділу, «Класи даних — як діти...», але в іншому відношенні є найкращим виданням найвідомішої книги Фаулера, особливо для Python-розробників, тому що приклади написані сучасним JavaScript, що є ближчим до Python, ніж Java — мова першого видання.

## Трибуна

### «Запис» Ґвідо у «Jargon File»

Запис для «Guido» у «The Jargon File» є про Ґвідо ван Россума. Серед іншого там написано:

> Mythically, Guido's most important attribute besides Python itself is Guido's time machine, a device he is reputed to possess because of the unnerving frequency with which user requests for new features have been met with the response "I just implemented that last night…"

Довгий час одним із відсутніх елементів синтаксису Python був швидкий, стандартний спосіб оголошення атрибутів екземпляра в класі. Тепер у нас є `@dataclass`!

Але вони приносять власні проблеми.

По-перше, при використанні `@dataclass` підказки типів є не необов'язковими. Нам обіцяли протягом останніх семи років, починаючи з PEP 484 — Type Hints, що вони завжди будуть необов'язковими. Тепер у нас є нова важлива можливість мови, що їх вимагає. Якщо ви не прихильники тренду статичної типізації, можливо, захочете замість цього використовувати `attrs`.

По-друге, синтаксис PEP 526 для анотування атрибутів екземпляра та класу перевертає усталену конвенцію інструкцій `class`: все, оголошене на верхньому рівні блоку `class`, було атрибутом класу (методи також є атрибутами класу). З PEP 526 і `@dataclass` будь-який атрибут, оголошений на верхньому рівні з підказкою типу, стає атрибутом екземпляра.

Я хотів би позичити машину часу Ґвідо, щоб повернутися в 2017 рік і продати цю ідею основній команді:

```python
@dataclass
class HackerClubMember:
    .name: str
    .guests: list = field(default_factory=list)
    .handle: str = ''
    all_handles = set()
```

Атрибути екземпляра мали б оголошуватися з префіксом `.`. Будь-яке ім'я атрибута без префіксу `.` є атрибутом класу (як завжди було). Граматику мови довелось би змінити для прийняття цього синтаксису. Я вважаю це досить читабельним і це дозволяє уникнути проблеми «виняток із винятку».
